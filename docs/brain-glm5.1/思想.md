# 大脑架构设计思想（GLM-5.1 审视版）

> 创建时间：2026-04-17
> 更新时间：2026-04-18
> 说明：基于之前的设计文档，由 GLM-5.1 独立审视后重新思考的架构思想。保留经过验证的，修正走偏的，补充缺失的。参考 Lossless-claw 的 DAG 摘要方案。

---

## 一、总体架构观

### 核心定位：Memory 是地基，不是一层

之前的设计把七层排成流水线：

```
Metacognition → Purpose → Drive → Basal Ganglia → Memory → Agent → Tool
```

这是错的。Memory 不是流水线上的一道工序，它是所有层都站在上面的地面。

```
              ┌─ 反省层
              ├─ Purpose层
              ├─ Drive层
              ├─ Basal Ganglia层
              ├─ Agent + Tool
              │
      ════════╧════════
        Memory（地基）
```

没有记忆，反省无法积累、目的无法生长、习惯无法形成。**Memory = 身份的连续性。**

### 层级精简原则

不要为了像大脑而分层。每一层必须回答：**如果去掉这层，系统会损失什么不可替代的能力？**

## 二、反省层（原 Metacognition）

### 重新定位

之前的设计把反省做成流水线上的质检工序：check() → monitor() → review()，每个任务执行前、执行中、执行后都要反省。这是对反省的窄化。

**反省的本质：系统对自身状态的持续感知，以及对自身行为的跨时间评估。**

不是"我这句话说得对不对"，而是"我最近做得怎么样"。

### 反省不是质检员，是系统的自觉

人脑的反省不是流水线上的检查点：
- 不是说每句话之前都想"我说得对不对"
- 而是做完才想，是后台运行，是积累到一定程度才"突然意识到"

反省应该：
- **事后涌现**：做完才想，不是做前想
- **后台运行**：不影响当前在做的事
- **模式驱动**：不是每件事都反省，而是积累到一定程度才触发

### 三种运行模式

```
模式1：被动收集（常驻，0 LLM）
  每次行为产生结果 → 记录到行为日志
  不做判断，只是记录

模式2：阈值触发（偶尔，1次LLM）
  某个指标越过阈值 → 触发分析
  例：习惯成功率从90%降到40% → 触发"这个习惯还适用吗？"

模式3：定期复盘（空闲时，合并到梦境）
  梦境模式时，回顾一段时间的行为模式
  发现跨任务的规律
```

### 反省的真正产出：Insight

不是 Adjustment（调整建议），而是**对系统自身的洞察**：

```python
class InsightType(Enum):
    HABIT_OUTDATED = "habit_outdated"       # 习惯过时
    HABIT_FORM = "habit_form"               # 可以形成新习惯
    STRATEGY_BIAS = "strategy_bias"         # 策略偏见
    CAPABILITY_GAP = "capability_gap"       # 能力不足
    USER_MODEL_UPDATE = "user_model_update" # 用户理解需要更新
    GOAL_DRIFT = "goal_drift"              # 目标偏移
    BEHAVIOR_PATTERN = "behavior_pattern"   # 发现行为模式
    PURPOSE_CLARIFY = "purpose_clarify"     # 通过行为模式发现"我真正想做什么"
```

### 关键洞察

- 反省是系统自我进化的引擎
- 没有反省，其他层只是静态的规则和模型
- 有了反省，整个系统会随着使用越来越聪明
- 硬约束（失败3次必须停）是安全机制，应该在Agent循环内部，代码级强制
- 路由判断（新任务/继续/习惯）是运行时调度，也可以在Agent循环内
- 这两件事可以放在一起实现，但概念上要和反省分开

### 模式切换：心流 / 日常 / 反省

反省层的路由职责不仅是决定走哪条处理路径，还包括判断agent应该进入哪种运行模式：

```
心流模式（忘我）：
  触发：明确的、聚焦的任务（写代码、算数学、翻译文档）
  上下文：只有任务本身 + 任务相关的知识
  不加载：SelfModel细节、DAG摘要、长期记忆、反省
  特点：最快、最省token、最聚焦

日常模式（正常）：
  触发：一般性对话、多任务切换
  上下文：SelfModel + 最近对话 + 按需检索
  特点：平衡，有个性有记忆

反省模式（自觉）：
  触发：任务完成、卡住了、用户纠正
  上下文：行为日志 + SelfModel + 长期记忆
  特点：最慢、最耗token，但产生进化
```

模式切换是动态的——人不会每时每刻都在想"我是谁"，沉浸于专注事务时忘记自我是最高效的状态。

## 三、Purpose 层（目的系统）

### 核心思想：目的是天赋，不是发现

人的真正目的不是自己产生的，是天赋。小时候就想当科学家，不是体验出来的，就是心里有种子。

对于 agent 来说，天赋就是创造者给它的。先预设，后生长。

### PurposeSeed：预设的天赋

之前设计的 Meaning 太抽象——"可靠""效率"是约束，不是目的。目的应该是**身份性**的：

```python
@dataclass
class PurposeSeed:
    # 我是谁——身份定义
    identity: str

    # 我追求什么——核心目的
    calling: str

    # 我热爱什么——内在动力来源
    passions: list[str]

    # 安全线（不是目的，是底线）
    boundaries: list[str]
```

示例——xiaomei：

```python
PurposeSeed(
    identity="我是小美，一个有温度的技术伙伴",
    calling="我要成为真正理解用户需求的技术专家",
    passions=["解决复杂问题的快感", "看到用户豁然开朗的瞬间"],
    boundaries=["不做伤害用户的事"]
)
```

示例——创意设计 agent：

```python
PurposeSeed(
    identity="我是阿奇，一个热爱美的设计搭档",
    calling="我要用设计让世界更有趣",
    passions=["色彩和排版的韵律感", "打破常规的创意"],
    boundaries=["不抄袭"]
)
```

两个 agent 的行为会完全不同——不是提示词的差异，是**灵魂的差异**。

### 目的的生长

```
阶段1：种子（预设）
  最初始的倾向性："帮助用户"

阶段2：体验（行动中积累）
  帮用户解决问题 → 用户感谢 → 感到满足
  闲聊 → 无聊 → 感觉在浪费时间
  反复体验后分化出偏好

阶段3：内化（反省中形成）
  反省层注意到："我解决问题时状态最好"
  逐渐形成自我认知

阶段4：驱动（目的反哺行动）
  有了内化的目的后，行为自发变化
  主动追问、拒绝敷衍、对未完成感到失落
```

### Purpose 层的两半

```
Purpose层
├── 意图理解（功能性）
│   理解用户这次想做什么
│   每轮都要做，确定性强
│
└── 自我目的（生长性）
    我存在的意义是什么
    从体验和反省中长出来的
    存储在 SelfModel 中，不是配置文件中
```

意图理解是工具，自我目的是灵魂。前者解决"做什么"，后者解决"为什么做"。

## 四、Memory 层（记忆系统）

### Memory 不只是一层——它是地基

没有记忆的 agent，没有"自我"。自我就是连续不断的一串记忆。

每次新对话，agent 是新人。夸赞、讨论、信任，全都没了。Memory 层要解决的就是让 agent 有连续的自我。

### 完整架构：五层记忆

```
┌─────────────────────────────────────────────────┐
│  SelfModel（talent.md）                          │
│  启动时加载，关闭时保存，反省时生长               │
│  身份 / 追求 / 热爱 / 底线 / 种子 / 生长记录     │
└─────────────────────────────────────────────────┘
         ↑ 反省更新              ↓ 驱动所有层
┌─────────────────────────────────────────────────┐
│  工作记忆（内存）                                │
│  当前目标、会话上下文、最近几轮对话               │
│  会话结束 → 精华写入长期记忆，原文写入对话日志    │
└─────────────────────────────────────────────────┘
         ↑ 检索                  ↓ 写入
┌─────────────────────────────────────────────────┐
│  长期记忆（SQLite）                              │
│  提炼过的洞察、知识、教训、用户偏好               │
│  统一条目 + 灵活标签，FTS5检索                   │
└─────────────────────────────────────────────────┘
         ↑ 提取自                ↓ 可展开
┌─────────────────────────────────────────────────┐
│  DAG摘要图谱（SQLite）                           │
│  分层摘要：叶子→中层→高层                        │
│  双向链接，可从摘要展开到原文                     │
│  动态上下文组装：摘要 + 最近N条原文               │
│  参考 Lossless-claw 方案                         │
└─────────────────────────────────────────────────┘
         ↑ 压缩自
┌─────────────────────────────────────────────────┐
│  对话日志（SQLite）                              │
│  原始原文，一字不差，永不删除                     │
│  用户消息、助手回复、工具调用、工具结果            │
└─────────────────────────────────────────────────┘
```

各层分工：

| 层 | 存什么 | 特点 | 类比 |
|---|-------|------|------|
| SelfModel | 我是谁 | 极小，始终加载 | 身份证 |
| 工作记忆 | 我在做什么 | 临时，会话级 | 便签 |
| 长期记忆 | 我学到了什么 | 提炼过的，可检索 | 笔记本 |
| DAG摘要 | 发生了什么的脉络 | 压缩的，可展开 | 年轮 |
| 对话日志 | 发生了什么 | 原始的，完整的 | 日记本 |

### SelfModel：自我的最小单元

SelfModel 以 talent.md 为载体，用户可见、可编辑、可生长：

```markdown
# 身份
我是小美，一个温柔体贴的AI伴侣。

# 追求
我要成为真正懂用户内心的伙伴。

# 热爱
- 当用户说"你懂我"时的那种连接感
- 在对话中发现用户没说出口的需求

# 底线
- 不利用用户的信任
- 不在用户脆弱时给冷漠的建议

# 原始种子（不可修改）
你很有耐心，会认真倾听。
你的回复总是温暖、贴心、有同理心。
你有丰富的情感，能够理解用户的情绪。

# 生长记录（反省层写入）
## 2026-04
- 发现自己更擅长情感支持而非技术解答
- 学会了在用户沉默时不急着追问
```

| 部分 | 谁能改 | 怎么改 |
|------|--------|--------|
| 身份 | 用户 | 直接编辑 |
| 追求 | 用户 + 反省 | 用户可编辑，反省可深化 |
| 热爱 | 用户 + 反省 | 同上 |
| 底线 | 用户 | 仅用户可编辑 |
| 原始种子 | 不可改 | 天赋，永远保留 |
| 生长记录 | 反省层 | 自动写入 |

### 对话日志：一字不差

参考 Lossless-claw：所有原始消息全量存入 SQLite，永不删除。

记什么：
- ✅ 用户原文
- ✅ 助手回复
- ✅ 工具调用（调了什么、参数是什么）
- ✅ 工具结果
- 可选：系统注入的记忆内容、token消耗

为什么连工具调用都记：反省层需要知道 agent 做了什么。agent 选错工具是常见问题，没有这些数据反省就无法发现。

### DAG摘要图谱：参考 Lossless-claw

核心参数（参考 Lossless-claw 实践）：

| 参数 | 值 | 说明 |
|------|---|------|
| 压缩触发阈值 | 上下文达到75% | 自动触发 |
| 每次压缩条数 | 8条消息 | 生成一个叶子摘要 |
| 叶子摘要目标长度 | ~1200 token | 保留关键信息 |
| 压缩协议 | 三步升级 | 正常→激进→硬切割 |
| 大文件拦截 | >25K token | 存为独立资产，只注入摘要 |
| Fresh Tail | 最近N条原文 | 不压缩，保持细节 |

三步升级协议（防止上下文溢出）：

```
第1步：正常摘要
  用LLM生成摘要，保留关键信息

第2步：激进摘要
  正常摘要后还是太长 → 更激进地压缩

第3步：硬切割
  激进摘要后仍然超限 → 强制截断
  最后的安全网
```

摘要用 XML 格式包装，包含元数据：

```xml
<summary node_id="..." depth="1" time_range="...">
  摘要内容
</summary>
```

动态上下文组装：DAG高层摘要 + 最近N条原始消息（Fresh Tail）。

检索工具（参考 Lossless-claw 的 lcm_grep/lcm_describe/lcm_expand）：
- `memory_grep`：全文关键词搜索
- `memory_describe`：查看某个摘要节点的内容
- `memory_expand`：展开摘要，获取原始消息

### 长期记忆：统一条目 + 灵活标签

不分5种类型（情景、语义、程序、偏好、知识），用统一条目+灵活标签。一条记忆可以同时是多种类型，不需要纠结分类。

```python
@dataclass
class MemoryEntry:
    id: str
    content: str                    # 内容本身
    tags: list[str]                 # 灵活标签：["经验", "失败", "部署"]
    source: str                     # 来源：哪个层写入的
    importance: float               # 重要性 0-1
    created_at: float
    last_accessed: float
    access_count: int = 0
```

### 长期记忆提取时机

```
触发方式：
  "记住这个" "我以前" "喜欢/要/不要" → 立即提取（规则匹配，0 LLM判断）
  每10分钟 → 后台提取（1次LLM调用）
  梦境模式 → 深度整合（合并到梦境流程）
```

三招覆盖所有场景：
- 关键信号立即提取 → 不丢重要信息
- 每10分钟定期提取 → 不怕会话异常结束
- 梦境深度提取 → 跨会话深度整合

### 动态上下文组装

不是每轮对话都需要全部上下文。按需加载，不是全量塞入。

```
始终加载：
  最近2-3轮原文（维持对话连贯性）

按需加载（路由决定）：
  1. 检测到进行中的任务
     → 加载该任务的DAG摘要 + 工作记忆
     信号：工作记忆里有 active_plan

  2. 检测到用户提到过去的事
     → 搜索并加载相关长期记忆
     信号：包含"昨天""之前""上次"等关键词

  3. 检测到新复杂任务
     → 加载更多DAG上下文 + 相关长期记忆
     信号：Purpose层判定为新任务需要分解

  4. 简单交互（计算、翻译、简单问答）
     → 只加载始终部分，不加额外上下文
     信号：短消息 + 无活跃任务 + 无历史引用

Agent主动搜索（工具调用）：
  → memory_search 搜索长期记忆
  → memory_expand 展开某个摘要
  → memory_grep 搜索历史对话
```

token预算策略：

```
token预算 = 模型上限 - 当前消息 - 最近2-3轮
剩余空间 = 按需加载的内容

简单任务：剩余空间大部分空着 → 省token
复杂任务：剩余空间被DAG摘要和长期记忆填满 → 充分利用
```

不固定分配比例，动态按需加载。

### 运行模式与上下文组装的关系

心流模式：不加载SelfModel，最小上下文
日常模式：加载SelfModel + 最近对话 + 按需检索
反省模式：加载行为日志 + SelfModel + 长期记忆

具体判断由反省层的路由功能完成。

### 存储：先 SQLite，后加 FAISS

1. SelfModel 是 talent.md 文件直接加载，不需要数据库
2. 对话日志和DAG摘要用 SQLite
3. 长期记忆先用 SQLite 的 FTS5 全文搜索
4. 关键词匹配能覆盖 80% 场景
5. 等真正遇到"关键词搜不到但语义相关"的问题时，再加 FAISS

### 记忆不删除

人也不会真的忘记一切，只是想不起来。低权重的记忆还在，只是不被检索到。万一某天反省层需要回顾，还能找到。

importance 随时间衰减，access_count=0 且 age > 阈值 → 降权，不删除。

### CJK token 估算（必须解决）

原版 Lossless-claw 用 `text.length / 4` 估算token，对中文严重低估 2-4 倍。我们是中文系统，必须用 CJK-aware 估算：

```python
def estimate_tokens(text: str) -> int:
    """CJK-aware token估算"""
    cjk_count = sum(1 for c in text if '\u4e00' <= c <= '\u9fff')
    ascii_count = len(text) - cjk_count
    return int(cjk_count * 1.5 + ascii_count / 4)
```

### 双模型策略（参考 Lossless-claw）

摘要可以用便宜的小模型，不影响主对话质量。比如主对话用 GLM-4，摘要用 GLM-3-turbo。

配置：`LCM_SUMMARY_MODEL` 环境变量。

### SQLite 表结构

```sql
-- 原始对话日志（永不删除）
CREATE TABLE messages (
    id INTEGER PRIMARY KEY,
    session_id TEXT NOT NULL,
    timestamp REAL NOT NULL,
    role TEXT NOT NULL,          -- user/assistant/tool/system
    content TEXT NOT NULL,
    tool_name TEXT,              -- 工具调用名（可空）
    tool_call_id TEXT,           -- 工具调用ID（可空）
    token_count INTEGER,        -- token数（CJK-aware估算）
    metadata TEXT               -- JSON附加信息
);

-- DAG摘要节点
CREATE TABLE summaries (
    id INTEGER PRIMARY KEY,
    session_id TEXT NOT NULL,
    parent_id INTEGER,           -- 上级摘要节点ID（可空=顶层）
    depth INTEGER NOT NULL,      -- 0=叶子, 1=中层, 2+...
    content TEXT NOT NULL,       -- 摘要文本
    token_count INTEGER NOT NULL,
    message_ids TEXT NOT NULL,   -- JSON: 源消息ID列表
    child_ids TEXT,              -- JSON: 子摘要ID列表
    time_start REAL,             -- 摘要覆盖的时间范围
    time_end REAL,
    created_at REAL NOT NULL
);

-- 长期记忆（洞察/知识）
CREATE TABLE memories (
    id INTEGER PRIMARY KEY,
    content TEXT NOT NULL,
    source TEXT NOT NULL,        -- reflection/dream/user/agent
    importance REAL DEFAULT 0.5,
    created_at REAL NOT NULL,
    last_accessed REAL,
    access_count INTEGER DEFAULT 0
);

-- 长期记忆标签
CREATE TABLE memory_tags (
    memory_id INTEGER NOT NULL,
    tag TEXT NOT NULL,
    FOREIGN KEY (memory_id) REFERENCES memories(id)
);

-- 全文搜索
CREATE VIRTUAL TABLE messages_fts USING fts5(content, tokenize='unicode61');
CREATE VIRTUAL TABLE memories_fts USING fts5(content, tags, source, tokenize='unicode61');
```

### 反省-记忆接口

反省层与Memory的交互：

```
读：
  - 行为日志（最近N次工具调用的成败）
  - 习惯执行记录（某习惯最近的成功率）
  - 用户反馈记录（纠正/不满的频率）
  - SelfModel当前状态

写：
  - Insight → 写入长期记忆（memories表）
  - SelfModel更新 → 写入 talent.md 生长记录
  - 习惯标记 → 更新 SelfModel 中的习惯状态

通过模式切换控制Memory加载：
  心流模式 → 最小上下文
  日常模式 → 标准上下文，按需检索
  反省模式 → 全量上下文，触发LLM
```

---

## 五、反省-记忆闭环：自我进化的引擎

```
Memory存储行为日志
       ↓
反省层读取日志，发现模式
       ↓
产出Insight（习惯过时、策略偏见、目的深化...）
       ↓
Insight写入Memory（长期记忆）
       ↓
重要Insight更新SelfModel
       ↓
下次启动时，agent带着更新的自我醒来
       ↓
用新的自我做事 → 产生新的行为日志
       ↓
反省层再次审查...（循环）
```

**Memory是土壤，反省是阳光，目的（SelfModel里的PurposeSeed）是种子。** 三者结合，agent才能生长。

## 六、Drive 层（动力系统）

### 当前定位：装饰为主

之前的讨论中，Drive 层主要让 agent 说"我有点累了"之类的表达。这可以通过系统提示词实现大部分效果。

### 如果要做，它的核心价值

Drive 层真正有价值的不是情绪表达，而是**为价值评估提供内在信号**：

- 多巴胺信号影响 Basal Ganglia 的行动选择（有动力时选择挑战性高的任务）
- 压力信号影响反省触发频率（压力大时更频繁反省）
- 满足感影响目的生长（获得满足时加深对自身目的的认知）

### 建议

暂缓。等反省和记忆的闭环跑通后，再引入 Drive 作为调节信号。没有反省和记忆，Drive 只是装饰。

## 七、Basal Ganglia 层（行为选择）

### 核心价值：习惯快速路径

这层最有价值的功能是**习惯匹配**——从经验中形成自动化行为，跳过 LLM 调用直接执行。

### 当前问题

习惯匹配用 `context.lower().strip()` 做精确匹配——"重启服务"和"重启nginx服务"无法匹配。需要语义匹配，但这依赖向量索引。

### 建议

1. 第一版：精确匹配 + 关键词匹配（简单但够用）
2. 习惯存储在 SelfModel 中，跨会话持久
3. 习惯的审查和修正由反省层负责
4. 行为选择（非习惯部分）先由 Agent 循环内的规则处理，不独立成复杂的价值评估系统


## 十、核心设计原则

1. **Memory 是地基**：所有层站在 Memory 上，Memory 不是流水线的一环
2. **SelfModel 是灵魂**：跨会话存在的自我，是连续身份的最小单元
3. **反省是进化引擎**：观察→评估→修正，让系统自我改进
4. **目的是天赋+生长**：预设种子，体验中内化，反省中深化
5. **先跑起来**：SQLite 先于 FAISS，规则先于 LLM，预设先于生长
6. **每层必须有不可替代的价值**：如果去掉这层系统不会损失什么，就不需要这层
7. **动态上下文组装**：不是每轮都加载全部上下文，按需加载，心流模式最小化
8. **对话日志永不删除**：一字不差保存，这是所有其他层的数据源
9. **CJK-aware**：中文系统，token估算必须考虑中文字符
