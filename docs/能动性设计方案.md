# 能动性设计方案

> 2026-05-27 讨论记录与方案框架

---

## 一、核心问题

当前 PACE 的执行模式：预设目标树 → 顺序执行子目标 → 卡住重试/退出。本质是**流水线式执行**，缺乏人类执行复杂任务时的灵活性。

## 二、人类可借鉴的执行特征

### 已识别的特征（按优先级排序）

| # | 特征 | 能动性提升 | 本质 |
|---|------|-----------|------|
| 1 | 上下文僵化感知 + 主动切换 | 高 | 元认知 |
| 2 | 卡住→换策略（不只是换次数） | 高 | 闭环 |
| 3 | 执行中动态调整计划 | 高 | 闭环 |
| 4 | 价值重估（"还值得做吗"） | 高 | 元认知 |
| 5 | 即时学习闭环（非事后复盘） | 中 | 闭环 |
| 6 | 观察线打断做事线 | 中 | 架构 |
| 7 | 满意即止（80分够了） | 中 | 判定逻辑 |
| 8 | 具体化求助 | 中 | prompt |
| 9 | 叙事驱动坚持 | ? | prompt |

### 关键洞察

**PACE 的核心问题不是缺"换策略"选项，而是它的设计假定 context 越多越好。** 当检测到僵化信号（REPEATED_OUTPUT、TOOL_LOOP）时，PACE 的反应是往 context 里**加 nudge**——恰恰是在已经僵化的上下文里再加东西。

## 三、为什么 LLM 有能力但完不成复杂任务

### 现象
LLM 具备远超单个人类的知识广度和编程能力，足以理解 ERP 的所有领域知识、工厂流程、设备细节——理论上能开发出远超人类水平的产品。但实际做不到。

### 根因不是能力不够

1. **没有活的骨架**：SelfModel 是静态文本注入 context，不会在执行中被新信息自动激活相关部分。人类做库存模块时，"订单状态机"会自然浮现检查一致性——agent 的骨架不会自己浮现。

2. **没有层次化的工作记忆**：人类做 ERP——核心数据模型在长期记忆（一直在线），当前模块细节在工作记忆（当前焦点），其他模块"待机"（需要时加载）。Agent 只有 context：要么在，要么不在。没有"待机但随时可唤醒"的中间态。

3. **没有内在的一致性驱动力**：人类发现设计前后矛盾时产生认知失调——不舒服感驱动人去解决矛盾。Agent 能检测到恰好在 context 里的矛盾，但不会主动去找。

4. **上下文僵化**：PACE 每一步都在往 context 里加东西（observations、nudge、lessons），从不清理。LLM 被自己之前的输出锚定——跟人类"钻牛角尖"同一机制。当检测到僵化信号时，PACE 的反应是往 context 里**再加 nudge**——恰好在已经僵化的上下文里再加东西。

### 关键洞察
Agent 需要的不是在 context 里放更多东西，而是：
- 骨架在执行中**自动激活**相关部分
- 僵化时**清空**而非**加料**
- 用**多个视角**审视同一问题，突破单一执行者视角的局限

## 四、视角切换：利用 LLM 的核心优势

### 重新框定
之前讨论的"潜意识调用"——极简上下文、旁观者视角——本质上是在用一个模糊的词描述更具体的能力：**视角切换**。

区别：
- "潜意识"暗示神秘的、不可控的机制
- "视角切换"是**可操作的、结构化的能力**——而且恰好是 LLM 远超人类的优势

### 人类 vs LLM
人类切换视角的瓶颈是**能力**——程序员很难真正用客户的脑子思考，用安全审计员的脑子思考。只能近似。

LLM 没有这个瓶颈。给 LLM 一个"你是 20 年经验的安全审计员"的 prompt，产出远超普通程序员模拟的安全审计。**LLM 拥有极强的视角切换能力，但当前架构从未利用它。**

### 视角切换的执行模式

PACE 卡住时（重试 ≥ 3 次 / InnerVoice "方向不对" / 僵化信号）：

```
并行调用 3-4 个视角（极简上下文，只有子目标描述）：
1. 架构师视角："这个问题的解法在整体架构里合理吗？"
2. 极简主义视角："有没有更简单的做法？甚至可以不做？"
3. 用户视角："从使用者的角度看，真正重要的是什么？"
4. 反面视角："这个方案最可能出问题的地方在哪？"

聚合输出 → 注入执行者 context → 重置重试计数 → 重新执行
```

关键设计决策：
- 视角调用**并行**——它们之间不需要共享上下文，只需要共享核心问题描述
- 每个视角的上下文**极简**——不加载失败历史、observations、nudge，正是"不被锚定"的关键
- 产出是"念头"而非"方案"——不直接执行，交给执行者评估
- 与 Layer 5（多模型审查）的关系：同一机制，不同时机和目的。Layer 5 审方案对不对，这个是"帮我换个脑子"

## 五、"换脑子"的四个维度

### 1. 视角切换（上面已描述）
最核心的机制。利用 LLM 的多角色能力突破单一执行者视角。

### 2. 清除工作记忆
- 保存状态 → 清空 context → 重新加载（从新鲜视角）
- 不是"继续"，是"重新开始但已经知道了一些事"

### 3. 资源恢复（部分已有基础）
- Drive 层有 energy，但"耐心消耗"和"创造力消耗"还没建模
- 需要：**cognitive_stamina** 维度，随重复失败衰减，随视角切换恢复

### 4. 内在一致性驱动
- 执行中定期用"审查视角"检查近期决策的一致性
- 检测到矛盾 → 产生"不舒服"信号 → 触发策略重评估

## 五、当前架构基础

### 组件关系
```
ConsciousLiving
  ├─ purpose: PurposeEngine          ← 唯一目标树
  └─ task_orchestrator: TaskOrchestrator
       ├─ _purpose → 同一个 PurposeEngine
       ├─ _pace_runner: PACERunner（懒加载）
       │    └─ self._purpose → 同一个 PurposeEngine
       ├─ _inner_voice: InnerVoice
       └─ _run_pace() → PACERunner.run()
```

### 关键：PACE 和 Purpose 已深度融合
- PACE 持有 `self._purpose`（同一个 PurposeEngine 实例）
- 已读目标：`get_current()`、`get_sub_goals()`
- 已写目标：`complete_goal()`、`goal.complete()`、`goal.append_log()`、`save()`
- 加新写路径（如动态插入子目标、重新分解）是自然扩展，不需要跨系统通信

### PACE 执行循环关键节点
- `_run_loop()` 主循环：`while True`，每步执行 + 检测 + 决策
- 重试上限：`max_retries_per_goal = 5`，达到后 `EXIT_STUCK`
- 决策分支：ESCALATE / REPORT_PARTIAL / CLARIFY / RETRY_DIFFERENT / SIMPLIFY / CONTINUE
- InnerVoice TASK_STEP：步骤后产生 reflection，但**信号没回路到策略层**
- `_inject_relevant_lessons()`：注入历史教训到 context，但**不改变计划**

### InnerVoice 现有能力
- `pause()` 四种触发：CHAT_TURN / TASK_STEP / TASK_DONE / SILENCE
- `_extract_continue_signal()`：纯正则，"换个方法"→ retry，"放弃"→ escalate
- `_has_experience_signal()`：检测值得记住的经验
- 输出路由：SelfImage / Drive / Purpose / LearningQueue
- **问题**：观测结果只写入存储，不传回 PACE 的策略层

## 六、活的骨架：热层 + 冷层

### 问题
蓝图 Project Mental Model 的问题是：DAG 摘要用模块名、文件路径做关键词匹配决定"什么跟什么相关"——这是代码决定的，不是语义理解的。而且骨架完全依赖被动注入，不注入就不存在。

### 方案：两层骨架

**热层**（始终在 system prompt 里，≤ 200 字）：
```
[当前项目骨架]
项目: xiaomei-brain，多 Agent AI 大脑框架
核心模块: consciousness/ memory/ purpose/ drive/ metacognition/
关键约束: MiniMax 只支持1个system message；同步架构；context context 不是越多越好
当前在做: 能动性改造——让 agent 在 PACE 执行中能自我感知僵化并主动切换视角
最近关键决策: 用视角切换（多视角并行LLM调用）替代单次"潜意识"调用
```

作用：不是提供详细信息，是让 agent **知道自己有骨架、知道骨架的大致形状**——从而在做决策时主动去查冷层。

**冷层**（向量存储，按需语义召回）：
- 存储：Project Mental Model 的完整摘要 + 操作记录，LanceDB 向量索引
- 召回：执行每一步前，用当前子目标描述做语义检索，加载 top-3 相关片段
- 更新：操作记录 → 摘要 → 向量化写入，已有 DAG 摘要机制可复用

### 为什么向量而不是 DAG

| | DAG（蓝图方案） | 向量（本方案） |
|---|---|---|
| 相关性判断 | 关键词匹配（模块名、文件路径） | 语义接近度 |
| "做库存时订单浮现" | 需要预设 inventory→order 关联 | embedding 自动识别语义接近 |
| 跨域关联 | 做不到（树结构限定路径） | embedding 空间自然支持 |
| 已有什么 | 部分实现（DAG 摘要） | **LanceDB 已部署**，长期记忆就在用 |

### 已有的基础
- **LanceDB**：已在 LongTermMemory 中使用，`BAAI/bge-m3` embedding
- **DAG 摘要**：已有 leaf → mid → top 的分层压缩机制，摘要格式可复用
- **ContextAssembler**：已有 daily/flow/reflect 模式的上下文组装，task 模式可加冷层注入

## 七、修改方向

### Phase 1：视角切换突破 + InnerVoice 回路（最小闭环）

**改动文件：** `runner.py`、新增 `metacognition/perspectives.py`、`task_orchestrator.py`

**触发条件**（两条路，任一触发）：
1. InnerVoice TASK_STEP 输出 "方向不对" / "retry" 信号 → 立即触发
2. 同一子目标重试 ≥ 2 次 → 保底触发

> InnerVoice 的上下文比执行者干净——没有累积的 observations、nudge、失败历史，同一个模型 + 不同 prompt + 更少上下文 = 不被锚定的判断。执行者的硬失败（重试 ≥ 2）作为确定性保底。

1. **僵化检测**（在 `_run_loop` 中）
   - InnerVoice reflection 包含 retry/escalate 信号 → 触发 `_try_perspective_breakthrough()`
   - `current_goal_retries >= 2` → 同样触发

2. **多视角并行调用**（新模块 `metacognition/perspectives.py`）
   - 并行调用 3-4 个视角（架构师 / 极简主义 / 用户 / 反面视角）
   - 每个视角：极简上下文，只有原始子目标描述，不加载失败历史
   - 可配置视角列表和数量
   - 产出：每个视角 1-2 句自然语言，聚合为一个"方向感"

3. **产出消费：直接注入，执行者自然过滤**
   - 视角产出是"方向感"（暗示、念头），不是指令
   - 直接注入执行者 context 作为前置提示
   - 执行者在执行中自然判断采不采纳——跟人类一样，潜意识给的想法不一定听

4. **重置执行**
   - 用聚合的方向感作为新的 context 前缀
   - 清空当前子目标的 observations（不带到新尝试里）
   - 重置重试计数，重新执行
   - 如果视角切换后仍失败，再走原 ESCALATE 路径

### Phase 2：动态目标树修改

**改动文件：** `runner.py`、`purpose_engine.py`

1. PACE 执行中发现新必要步骤 → `self._purpose.add_sub_goal()` 动态插入
2. 阻塞信号（等待外部输入）→ 标记子目标 BLOCKED，推进其他
3. 价值重估：LLM 定期评估"当前子目标还值得做吗？"

### 暂缓：资源维度扩展（cognitive_stamina）

**原因**：视角切换本身已经做了"换上下文"——这也是 stamina 的核心目的（换事 = 换上下文）。人类的耐心是自适应上限，功能上已被"视角突破后仍失败 → ESCALATE"覆盖。先不加。

### 暂缓：一致性驱动力（认知失调检测）

**原因**：需要热层骨架先落地（记录关键决策），才能在子目标完成后对比"第 2 步决策 vs 第 5 步决策"。Phase 1 不包含热层，等骨架机制跑通后再做。

## 八、与现有蓝图的关系

现有蓝图（`docs/大脑架构_经验系统蓝图.md`）的目标是"让她像顶级工程师一样成长"——通过积累经验、建立项目地图、多模型审查。核心是**让下一次执行更好**。

本方案的目标是**让这一次执行更灵活**——执行中自我感知 + 视角切换 + 动态调整计划。

两者的关系是互补的，不是替代：
- 蓝图 Phase 2（Goal DAG）是本方案 Phase 3（动态目标树修改）的前置依赖
- 蓝图 Layer 5（多模型审查）的机制与本方案的视角切换共享底层——都是多 LLM 调用、不同角色、聚合输出。区别是审查在"完成时"，视角切换在"卡住时"

**本方案与蓝图不存在架构冲突。**

## 十、设计原则

1. **默认流水线，设置清晰的切换触发条件**——不是抛弃计划，是让计划可被修改
2. **context 不是越多越好**——僵化时需要清空，不是加 nudge
3. **视角产出不直接执行**——作为"念头"交给执行者评估，执行者决定是否采用
4. **修改 goal tree 是自然的**——PACE 已经持有 PurposeEngine 引用，只是目前只读不写（除了 complete）

## 十一、决定总结

| 问题 | 结论 |
|------|------|
| 先做什么 | 视角切换 + InnerVoice 回路（Phase 1），独立于 Goal DAG |
| 触发粒度 | InnerVoice "方向不对" 为主，重试 ≥ 2 保底 |
| 产出消费 | 直接注入 context，执行者自然过滤 |
| InnerVoice 回路 | 跟视角切换同一 Phase |
| cognitive_stamina | 暂缓——视角切换已覆盖"换上下文"功能 |
| 一致性驱动力 | 暂缓——需要热层骨架先落地 |

### 额外洞察
- **人类的耐心是自适应断路器**：同一问题上投入意愿随失败递减，防止"一条道走到黑"。agent 的硬编码重试上限是同一机制，但缺乏"耐心耗尽 → 果断放弃 → 重新分解或延期"的智能决策。未来 stamina 若做，应朝这个方向。
