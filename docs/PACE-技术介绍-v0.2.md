# PACE：LLM Agent 的认知回合制执行架构

> v0.2 — 案例：开发ERP系统

## 一、问题：ReAct 为什么不够用？

ReAct（Reasoning + Acting）是当前 LLM Agent 的主流执行模式：LLM 输出一段推理 + 一个工具调用，系统执行工具，把结果喂回去，循环往复。

它的核心假设是：**LLM 每一步的输出都是合理的**。但实际运行中，这个假设经常被打破：

- **盲冲**：LLM 看到了"开发ERP系统"，一次性把 6 步全做了，中间不检查、不汇报
- **死循环**：同一个工具连调 20 次，每次结果一样，LLM 不觉得有问题
- **静默失败**：LLM 输出为空，系统默默推进到下一步，实际什么都没做
- **目标漂移**：从"调研 asyncio"漂移到"写 asyncio 教程"漂移到"对比所有并发模型"
- **拒绝执行被忽略**：LLM 明确说"这个我不做"，系统当没听见，继续推下一个子目标

这些问题的共同根因：**ReAct 是流水线，没有反馈控制环**。它只管执行，不管检查。

## 二、起源：从 Task Manager 到 Metacognition

PACE 不是凭空设计的。它的起点是一次代码重构中的发现。

### 2.1 Task Manager 长歪了

在 xiaomei-brain 的早期架构中，有一个 `TaskManager` 模块负责管理任务执行。它做三件事：

- 管理 Task 生命周期（创建/暂停/恢复/完成）
- 记录 cognitive_log（认知日志：每步的决策、发现、产出）
- 跟 PurposeEngine（目标树）对接

随着功能迭代，`TaskManager` 变得越来越臃肿。它既管"做什么"（跟 Purpose 重叠），又管"做得怎么样"（认知监督），还管"接下来怎么办"（调度决策）。代码在，但职责边界模糊。

真正的问题不是 `Task` 和 `Purpose` 该不该合并，而是：**Metacognition 层没建起来，Task 临时顶替，结果长歪了。**

### 2.2 回到 OS 的类比

在操作系统中，进程管理是一个独立的层——它不挂在文件系统上，也不跟内存管理混在一起。同样的道理：

```
应该的架构：
  Purpose（Goal 树）       → 管"做什么"（拆解/优先级/进度）
  Metacognition（认知控制） → 管"做得对吗？要不要换个干法？"
  Agent（执行）           → 管"怎么做"

当前的混乱：
  Task 同时干了三件事——它是 Purpose 的 facade，是 Metacognition 的替代品，
  又是 Agent 的执行记录。
```

**Task 不应该是 Purpose 的 facade，也不应该和 Purpose 合并——它应该是 Metacognition 层的执行上下文。** 这个认知直接催生了 PACE 的设计。

### 2.3 AI 为什么做不了复杂任务

顺着这个思路往下想，一个更大的问题浮现了：

> 人类说 AI 没有执行复杂任务的能力。是真的没有，还是**有但没有被发挥出来**？

当前的 Agent 范式本质上是**执行脚本**，不是**执行任务**：

|        | 执行脚本      | 执行任务    |
| ------ | --------- | ------- |
| 前提假设   | 世界是静态的    | 世界会变    |
| 对错误的反应 | 报错退出      | 调整路径    |
| 对中间发现  | 忽略        | 重新评估    |
| 对自己的认知 | 不知道不知道    | 知道自己不知道 |
| 成功标准   | 每一步都按计划做完 | 目标达成    |

人类做复杂任务时，每一步都在重新评估：

```
行动 → 观察 → 比较预期 → 发现偏差 → 更新认知 → 调整行动
```

比如写一个不熟悉的代码库：

```
1. 开始改 → 发现比预期复杂（意外检测）
2. 暂停 → "我可能低估了"（自我评估）
3. 重新读代码 → 发现新依赖（知识更新）
4. "要不换个思路？"（策略切换）
5. 继续 → 又遇到问题 → "这个我真不会"（能力边界）
6. 决定：问同事 / 搜索 / 简化目标（重新规划）
```

关键不是每一步都做对，而是**每一步都在重新评估**。

### 2.4 能力是有的，只是没给它发挥的场合

LLM 其实能做到这些。让你让 LLM 反省，它能说出哪里不对劲。问题是**没人让它做**。

```
当前调用模式：
  LLM 是一个工具 —— 你问它，它回答。

应该的模式：
  LLM 是一个心智 —— 它有一个持续运行的自我监控循环。
```

| 什么 | LLM 能做到吗 |
|------|------------|
| 理解上下文 | ✅ |
| 规划步骤 | ✅ |
| 执行单步 | ✅ |
| 执行中发觉"不对劲" | ✅（如果你问它的话） |
| 主动暂停并重新评估 | ❌（没人问它） |
| 认识到自己走了弯路 | ❌（没人问它） |
| 从失败中提炼教训 | ❌（没人问它） |
| 承认"这个我做不到" | ❌（没人问它） |

后面四个不是 LLM 不行，是**调用架构不允许它这么做**。

ReAct 循环虽然加了 `think → act → observe → think`，但它仍然是一个**单线程执行器**——就像让一个人一边写代码一边做 code review，一边开车一边评估驾驶技术。

区别不在于 LLM 本身，在于**有没有一个独立的、持续的、不参与执行的观察者**。这个观察者也可以用 LLM 实现——但它是一个不同的 instance，带着不同的 context，问不同的问题。

就像你写代码时，过一会站起来走一走，脑子自动开始想"刚才那个方向对吗？"——不是你刻意切换的，是你大脑里一直有一个后台进程在跑。

**Metacognition 就是那个后台进程。当前 Agent 没有后台进程，只有一个前台线程。**

### 2.5 PACE 的诞生

基于这些分析，PACE 的设计目标就清晰了：

```
从 Task Manager 的碎片功能中，提取出认知控制的核心：

  Task 的 cognitive_log   → Metacognition 的回合记录
  暂停/恢复               → Metacognition 的回合间状态保存
  确认流程                → Metacognition 的 pre-execution check
  知识提取                → Metacognition 的 post-execution lesson
```

这些东西不应该挂在 Purpose 上（Purpose 管"做什么"），也不应该挂在数据类上，而应该是一个独立的**认知控制器**。

PACE 的名字本身就是设计：**Pause → Assess → Choose → Execute**。不是 LLM 做不了复杂任务，是我们只给了它一个"做事"的接口，没给它一个"想事"的空间。PACE 补的就是这个空间。

## 三、PACE 的核心思想：认知回合制

PACE 把 Agent 执行从"流水线"改造成"回合制游戏"：

```
ReAct:  LLM → 工具 → LLM → 工具 → ...  （一路跑到黑）

PACE:   停 → 看 → 判 → 行 → 停 → 看 → 判 → 行 → ...  （步步检查）
```

四个阶段组成一个**认知回合**（Cognitive Turn）：

```
用户消息 → ┌─ Pause ─── 停下来，注入上一轮的认知提示（nudge）
           │             "上一步你卡住了，换个方法"
           │
           ├─ Execute ── Agent.stream()，正常执行一步
           │             LLM 输出 + 工具调用，跟普通 chat 一样
           │
           ├─ Assess ─── 两层检查：
           │             1. 规则检测（7条，零 LLM 成本）
           │             2. LLM 判断（3步抽样，控制成本）
           │
           └─ Choose ─── 6 个分支决定下一步：
                         CONTINUE / RETRY_DIFFERENT / SIMPLIFY
                         / REPORT_PARTIAL / CLARIFY / ESCALATE
```

每个回合只推进一步，完成后再进入下一回合。就像下棋：每走一步都要看看棋盘，而不是闭眼走完整个棋局。

## 四、Assess 层：两层检测机制

### 3.1 规则检测（7 条，零 LLM 成本）

每步执行完，7 条纯规则自动跑：

| 规则 | 检测内容 | 触发条件 | 成本 |
|------|----------|----------|------|
| TOOL_LOOP | 连续 ≥3 次调同一工具 | 工具名相同 | 0 |
| TOOL_STORM | 单步工具调用 > 10 次 | 计数器 | 0 |
| EMPTY_RESPONSE | 去掉 PROGRESS 标签后无内容 | 字符串空 | 0 |
| REPEATED_OUTPUT | 连续 2 步输出高度相似 | Jaccard > 0.9 | 0 |
| SLOW_STEP | 单步耗时 > 历史均值 × 3 | 计时器 | 0 |
| NO_PROGRESS | 有工具调用但没 PROGRESS 标签 | 正则匹配 | 0 |
| GAVE_UP | Agent 明确拒绝（关键词） | 关键词匹配 | 0 |

GAVE_UP 的关键词库包括中文 Agent 常见的拒绝表达："不重复""已交付""同一个子目标推了""死循环""拒收"等。

### 3.2 PROGRESS 标签：Agent 的进度汇报协议

Agent 在每步结束时需要输出进度标签。支持两种格式：

```xml
<!-- XML 属性格式 -->
<PROGRESS status="completed" summary="ERP数据模型设计完成，12个核心实体"/>

<!-- JSON 体格式（LLM 更自然） -->
<PROGRESS>{"status": "in_progress"}</PROGRESS>
```

三种状态语义：

| status | 含义 | 系统行为 |
|--------|------|----------|
| `completed` | 当前子目标完成 | 推进到下一个子目标 |
| `in_progress` | 还没做完 | 不推进，等下一回合 |
| 无标签 | 忘了写 | 进入自动判断逻辑 |

### 3.3 LLM 判断（三层抽样，控制成本）

规则检测发现了异常信号后，需要 LLM 来判断严重程度。但每一步都调 LLM 检查会让成本翻倍。

PACE 的策略是**三层抽样**：
1. **硬规则优先**：TOOL_STORM / EMPTY_RESPONSE / GAVE_UP（连续）直接判定，不调 LLM
2. **LLM 抽样**：从最近 5 步中选 3 步，给 LLM 判断当前步是否有问题
3. **规则降级**：LLM 调用失败时，退回规则判断

这样把 LLM 检查成本控制在每次 PACE 运行约 1-3 次额外调用。

## 五、Choose 层：6 个决策分支

```python
# Step 1: 硬规则优先（零 LLM 成本）
if TOOL_STORM → RETRY_DIFFERENT  # "干了太多事，退回去只做这一步"
if EMPTY_RESPONSE → RETRY_DIFFERENT  # "没产出，换个思路"
if GAVE_UP (连续2次) → ESCALATE  # "Agent 明确拒绝了，弹框问用户"

# Step 2: 有 surprises 但没触发硬规则 → LLM check
llm_step_check() → 返回以下建议之一

# Step 3: 根据建议决定下一步
CONTINUE       → 推进到下一个子目标
RETRY_DIFFERENT → 重新执行当前子目标（换方法，注入 nudge）
SIMPLIFY       → 重新执行当前子目标（简化后）
REPORT_PARTIAL → 确认部分完成，尝试推进
CLARIFY        → 需要澄清，尝试推进
ESCALATE       → 保存 checkpoint + 弹确认框 + 等待用户
```

**ESCALATE 确认框**的设计：

```
[元认知] Agent 连续 2 次拒绝执行，可能已实际交付

  1. 跳过这一步
  2. 我自己来操作
  3. 换个思路重试
  4. 标记为已完成
  0. 取消任务
```

用户选 4 → 直接标记子目标完成，不恢复 PACE。用户选 1-3 → 保存 checkpoint，注入用户回答，恢复执行。

## 六、子目标推进：接力机制

PACE 的推进单位是**子目标**。每个回合结束时：

```
1. _update_goal_progress("completed")  → 标记当前子目标完成
2. PurposeEngine 自动激活下一个子目标
3. _build_intent_context_for_goal()    → 重建 system prompt 上下文
4. 构造新消息 "[系统] 子目标 4/6: 开发核心模块"
5. 循环继续，LLM 看到的是新的子目标描述
```

视觉化表示：

```
┌──────────────────────────────────────────────────────────┐
│ 主 Goal: 开发一个企业ERP系统                                │
│                                                          │
│  [✓] 调研ERP业务需求和现有系统  ← 回合1: 访谈+读文档+需求分析 │
│  [✓] 确定技术选型和架构方案     ← 回合2: 分析+决策           │
│  [→] 设计数据模型和接口规范    ← 回合3: 正在执行...          │
│  [ ] 开发核心模块                                         │
│  [ ] 系统集成测试和优化                                    │
│  [ ] 用户验收和部署上线                                    │
└──────────────────────────────────────────────────────────┘
```

### 5.1 自动完成的三层判断

`_maybe_auto_advance()` 是推进决策的核心：

```
第一层：PROGRESS=completed → 正常推进
  LLM 明确标记完成了 → Goal.complete() → 激活下一个

第二层：无 PROGRESS + 无异常 + 无工具调用 → 纯文本确认
  LLM 说"数据模型设计好了"但忘了放 PROGRESS 标签 → 自动完成
  （但先检查是否在拒绝/等待用户，防止误判）

第三层：无 PROGRESS + LLM 判定 CONTINUE → 信任 LLM 判断
  规则检测到小问题，但 LLM step_check 说可以继续

阻断条件：
  PROGRESS=in_progress       → Agent 说还没做完，不推进
  拒绝/等待关键词检测        → "等用户确认""我不执行" → 设 waiting_user
```

### 5.2 非分解型目标

如果一个目标没有子目标（无同级 sibling），执行完当前步后：
- `_maybe_auto_advance` 返回 True（自动完成判断通过）
- 但找不到下一个子目标
- → 直接标记当前 Goal 完成并退出

## 七、异常处理矩阵

| 异常类型 | 检测方式 | 处理策略 |
|----------|----------|----------|
| 单步空响应 | 规则检测 | 换方法重试（最多 5 次） |
| 连续 3 步空响应 | 规则检测 | 标记 ERROR，暂停任务 |
| 单步工具风暴（>10次） | 规则检测 | 换方法重试 |
| GAVE_UP（单次） | 规则检测 | 换方法重试 |
| GAVE_UP（连续 2 次，含历史共 ≥3 次） | 规则检测 | ESCALATE → 弹确认框 |
| TOOL_LOOP（连调 3 次） | 规则检测 | LLM 判断是否严重 |
| REPEATED_OUTPUT | 规则检测 | LLM 判断 |
| SLOW_STEP | 规则检测 | LLM 判断 |
| NO_PROGRESS | 规则检测 | 自动完成（若判断 OK） |
| 子目标重试 5 次上限 | 主循环计数 | 标记 STUCK，暂停 |
| 等待用户反馈 | 关键词检测 | 标记 WAITING_USER，暂停 |
| 模型异常/网络错误 | try/except | 标记 ERROR，记录 cognitive_log |

## 八、Checkpoint 机制：暂停后恢复

PACE 支持在任意点保存状态，后续恢复：

```
中断场景：用户打断 / ESCALATE 弹框 / 取消

1. save_checkpoint(goal_id, step_index)
   → 序列化 observations、step_index、goal_id 到
     ~/.xiaomei-brain/pace_checkpoints/xiaomei_{goal_id}.json

2. 用户的下一轮输入：
   → restore_checkpoint() 恢复 observations
   → _resume_pace() 设置 _resume_step + 用户回答作为 nudge
   → _run_loop() 从断点继续，注入用户回复作为上下文
```

Checkpoint 文件是 JSON 格式，持久化到磁盘，支持跨会话恢复。

## 九、退出原因体系

PACE 不再"静默退出"。每次 `run()` 返回一个退出原因：

| 退出原因 | 含义 | 系统后续行为 |
|----------|------|-------------|
| `completed` | 正常完成，所有子目标 done | 正常返回 |
| `waiting_user` | Agent 在等用户反馈 | 打印等待提示，阻止 PACE 重入 |
| `escalated` | 弹出确认框等待用户 | 等待用户选择 |
| `stuck` | 重试耗尽，无法继续 | 打印"无法继续" |
| `error` | 执行异常 | 记录错误日志 |

`waiting_user` 的处理链路：

```
PACE 检测到 Agent 说"等用户确认"
  → _maybe_auto_advance() 返回 False + _exit_reason = "waiting_user"
  → _run_pace() 打印 "[小美] 正在等待你对「设计数据模型和接口规范」的反馈..."
  → 设 _pace_waiting = True
  → 用户发下一条消息 → 清除标记 → 消息正常进入意图分析
  → 重新进入 PACE，Agent 看到用户回复
```

## 十、完整链路示例

从用户发 `!开发一个企业ERP系统` 到全部完成：

```
1. handle_message()
   ├─ ! 前缀 → 自动 _task_mode = True
   ├─ 意图分析 → LLM 拆成 6 个子目标
   ├─ 创建 Goal → 存入 PurposeEngine
   └─ _run_pace() → PACERunner.run()

2. 回合 1: "调研ERP业务需求和现有系统"
   ├─ Execute → LLM 访谈分析、读文档、需求整理
   ├─ Assess → 规则检测：无异常
   │           PROGRESS=completed → has_progress_tag=True
   ├─ Choose → _maybe_auto_advance() → True
   └─ 推进 → 激活下一个子目标

3. 回合 2-6: 同上，依次完成技术选型→数据模型设计→核心模块开发→集成测试→验收部署

4. 最后一个子目标完成
   → 所有子目标 completed
   → 主 Goal 完成
   → 打印 "全部 6 个子目标已完成"

5. post_review → LLM 复盘总结 → 存 lesson 到持久化
```

## 十一、架构取舍与设计哲学

### 10.1 为什么选择回合制而非流水线？

| 维度 | ReAct 流水线 | PACE 回合制 |
|------|-------------|------------|
| 每步检查 | 无 | 规则 + LLM 双重 |
| 死循环检测 | 无（依赖 LLM 自觉） | 规则实时检测 |
| 用户中断 | 粗暴取消 | Checkpoint 暂停/恢复 |
| 错误恢复 | LLM 自己决定 | 6 种决策分支，系统级控制 |
| LLM 额外成本 | 0 | ~1-3 次额外调用/运行 |
| 可控性 | 低（LLM 自治） | 高（系统级决策） |

### 10.2 为什么规则检测放在 LLM 判断之前？

三层原因：
1. **成本**：规则零成本，LLM 每次调用都有费用和延迟
2. **确定性**：规则判断是确定性的，LLM 判断有随机性
3. **速度**：规则微秒级，LLM 秒级

### 10.3 什么情况走硬规则，什么情况走 LLM？

硬规则的门槛是**信息量足够**：
- TOOL_STORM（>10次）：计数器足够 → 硬规则
- EMPTY_RESPONSE：字符串为空 → 硬规则
- GAVE_UP（连续 2 次）：拒绝信号强 → 硬规则

LLM 判断的门槛是**信息量不够**：
- TOOL_LOOP（3次）：可能是正常的三步操作，也可能是死循环 → LLM
- REPEATED_OUTPUT：可能是内容确实相似（如重复确认），也可能是卡住 → LLM
- SLOW_STEP：可能是网络慢，也可能是模型有问题 → LLM

---

## 十二、总结

PACE 的核心贡献不是又发明了一个 Agent 框架，而是提出了一个**认知控制层**（Metacognition Layer）的概念：

> 在 LLM 的 ReAct 执行循环之上，叠加一个不依赖 LLM 自身自觉性的检查-决策层。这个层不替代 LLM 的推理能力，而是补上 LLM 缺乏的**执行自觉性**——在执行每一步之前和之后，用一个确定性的规则系统 + 轻量 LLM 判断来确保 Agent 走在正确的轨道上。

代价约为 10% 的额外 LLM 调用（3 步抽样），换来的是：
- 死循环自动检测和中断
- Agent 拒绝执行时系统级响应
- 执行过程的完全可观测和可恢复
- 用户对执行过程的实时介入能力
