# Metacognition 层：复杂任务执行的核心缺失

> 时间：2026-05-10
> 缘起：复盘 Task/Goal/Purpose 架构时，发现 TaskManager 实质是在填补
>       Metacognition 未实现的空缺。以下从"为什么 AI 做不了复杂任务"
>       这一问题出发，展开设计讨论。

---

## 一、核心命题

当前 AI Agent 的范式本质上是在 **执行脚本**，不是在 **执行任务**。

```
感知 → 规划 → 执行 → 报告
```

区别：

| | 执行脚本 | 执行任务 |
|---|---|---|
| 前提假设 | 世界是静态的 | 世界会变 |
| 对错误的反应 | 报错退出 | 调整路径 |
| 对中间发现 | 忽略 | 重新评估 |
| 对自己的认知 | 不知道不知道 | 知道自己不知道 |
| 成功标准 | 每一步都按计划做完 | 目标达成 |

Agent 在执行的第一步就已经"死"了——它把自己锁在了一条线性路径上。

---

## 二、人类做复杂任务的区别

写一个不熟悉的代码库时：

```
1. 开始改 → 发现比预期复杂              （意外检测）
2. 暂停 → "我可能低估了"               （自我评估）
3. 重新读代码 → 发现新依赖              （知识更新）
4. "要不换个思路？"                     （策略切换）
5. 继续 → 又遇到问题 → "这个我真不会"  （能力边界）
6. 决定：问同事 / 搜索 / 简化目标      （重新规划）
```

关键不是每一步都做对，而是 **每一步都在重新评估**：

```
行动 → 观察 → 比较预期 → 发现偏差 → 更新认知 → 调整行动
```

这个循环不是 bug，是 feature。

## 三、两个关键能力

### 1. Surprise Detection（意外检测）

不是"出错了"这种明确错误，而是"不对劲"——预测与现实的偏差：

- "这个函数应该很简单，但我已经读了 10 分钟了"
- "这个方案我在别的项目用过，但这里似乎不适用"
- "用户说'随便'但语气不太对"

LLM 有能力做——你让它反省，它能说出哪里不对劲。问题是**没人让它做**。当前流程是：LLM 输出 → 解析 PROGRESS 标签 → 下一个子目标。中间没有"等等，好像哪里不对"的环节。

### 2. Capability Calibration（能力校准）

- Goal 拆解时从不问"我能做这个吗"
- 遇到困难时从不判断"暂时障碍 还是 根本做不到"
- 完成后从不比较"实际执行和当初预计的差距"

---

## 四、这层缺失的深层原因

不是技术上做不到，而是 **当前 Agent 架构没有给这层留位置**。

```
当前架构：LLM 就是一切，决策和执行混在一起
应该架构：执行层 → 干什么
         元认知层 → 干得对吗？要不要换个干法？
```

Metacognition 需要独立——不是因为 LLM 做不到这些判断，而是因为**判断者不能是执行者自己**。

就像写代码时，debug 的那个人不能是刚好写完这段代码的那个状态——你需要"切换一下脑子"。这就是 Metacognition 作为独立层的认知意义：它不代表更强的能力，而是**不同的视角**。

---

## 五、设计思想：认知回合制

把复杂任务执行变成**回合制**，而不是流水线：

```
每个回合：
  1. Metacognition 读取当前状态 → 产出一个问题
  2. Agent 回答这个问题
  3. Metacognition 评估回答 → 决定下一回合做什么
```

回合的具体形式：

| 阶段 | Metacognition 的问题 | 可能的 Agent 回答 |
|------|---------------------|------------------|
| 开始前 | "我理解对了吗？有没有遗漏信息？" | "用户说'快一点'，可能需要简化方案" |
| 每步后 | "结果和预期一致吗？学到了什么？" | "API 文档有误，实际行为与文档不符" |
| 卡住时 | "为什么卡住？暂时障碍还是根本问题？" | "这个需要用户提供权限，我做不到" |
| 完成后 | "实际执行和最初规划的差距？怎么做更好？" | "第二阶段花了预期 3 倍时间，低估了测试工作量" |

## 六、当前系统中 Metacognition 雏形

xiaomei-brain 中已有 Metacognition 的碎片，但散落在各处：

| Metacognition 职责 | 当前位置 | 状态 |
|-------------------|---------|------|
| Pre-execution check | `_build_confirm_info()` | 死代码，从未触发 |
| 执行中进度监控 | `_should_auto_advance()` | 仅靠 LLM 输出的 PROGRESS 标签 |
| 出错恢复 | `_handle_confirmation` cancel 分支 | 有 bug，只放弃了 Goal 没放弃 Task |
| 执行后复盘 | `_extract_task_knowledge()` | 功能正常，但只是事后总结 |
| 知识累积 | `Task.cognitive_log` | 只记录 output/pitfall，不记录 decision/discovery |
| 暂停/恢复 | `TaskManager.pause/resume` | 与 PurposeEngine 双指针不同步 |

**TaskManager 实质上是在填补 Metacognition 缺失的坑**，但它是挂在 Purpose 侧面的数据管理器，不是独立的认知控制器。

## 七、架构定位

```
用户输入  →  IntentUnderstanding（理解意图）
                    ↓
            ┌─ Metacognition ───────────────────────┐
            │  pre:  确认理解，检测信息遗漏           │
            │  step: 每步后比较预期与实际             │
            │  stuck: 判断障碍类型，决定策略          │
            │  post: 复盘，校准能力认知，提取教训     │
            │  meta:  暂停/恢复/切换 执行上下文       │
            └──────────────┬────────────────────────┘
                           │ Goal + context
                           ▼
                    Agent.run(goal)
                           │
                           ▼
                    PurposeEngine（Goal 树：拆解/优先级/进度）
```

三层各司其职：
- **PurposeEngine** → "做什么"（Goal 拆解、优先级、进度追踪）
- **Metacognition** → "做得对吗"（意外检测、能力校准、策略调整、执行管控）
- **Agent** → "怎么做"（LLM 推理 + 工具调用）
