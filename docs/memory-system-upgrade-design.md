# 记忆系统升级：需求与设计文档

## 问题

当前记忆系统的存储和检索方式过于简单，无法支撑更自然的联想和回忆：

1. **存储粒度太粗** — 一条记忆存一整段信息（"用户喜欢川菜，尤其是水煮鱼"），无法独立关联或复用其中的子信息
2. **没有场景标签** — 记忆存了但不知道在什么场景下该被唤起。用户在聊"成都旅游"时不应弹出"买鼠标"的记忆
3. **关系没有权重** — 两条记忆之间的关联只有"有/无"，无法区分"偶尔相关"和"强相关"
4. **检索全靠语义搜索** — 每次都在全部记忆中搜索，没有先按场景缩小范围

---

## 设计方案：三层记忆架构

### 第一层：存储层

**1.1 碎片化存储**

提取记忆时，LLM 要把复合信息拆成原子碎片。一条碎片只包含一个事实/偏好/经验。

```
现在（不要）              →  改成（要）
━━━━━━━━━━━━━━━━━━       ━━━━━━━━━━━━━━━━━━
"用户喜欢川菜，            "用户喜欢川菜"
 尤其是水煮鱼"             "用户喜欢吃水煮鱼"
                          "用户做菜不擅长调味"
```

为什么：原子碎片可以独立关联、自由组合、部分衰减。整句存的话，关联"水煮鱼"就必须带上"川菜"，太僵硬。

**1.2 场景标签**

每条记忆存储时带一个 `scene_tags` 字段，标记这条记忆在什么场景下会被用到。

场景标签举例：`["家里"]`, `["公司", "开会"]`, `["成都旅游"]`, `["运动"]`

规则：
- LLM 提取时判断场景归属
- 适用所有场景的填 `[]`
- 一条记忆可以有多个场景标签
- 标签由 LLM 自由生成（非预设词表），后期可在梦境中聚类标准化

**1.3 关系带权重**

记忆之间的关联关系（因果/时序/对比/包含）现在只有"有/无"，改成带权重（0~1）。

- 默认权重 0.5
- 两条记忆经常一起被召回 → 梦境中加固权重
- 长时间未使用 → 梦境中衰减权重
- 权重 < 0.05 → 标记为休眠（不再参与检索，但不删除）

**1.4 共现追踪**

新增 `co_occur` 表，记录"哪些记忆经常一起被召回"。每次 `recall()` 返回 >=2 条结果时，自动记录它们成对出现的次数。

这个表是中间数据，不直接参与检索，只供梦境阶段使用。

---

### 第二层：检索层

**2.1 场景优先过滤**

`recall()` 新增可选参数 `scene`。调用方可以根据当前对话场景传入场景标签。

检索流程变为：

```
if scene 指定:
    1. 从 SQLite 查出带有该 scene_tag 的记忆 ID 列表
    2. 语义搜索只在这些 ID 的范围内进行
    3. 如果结果不足，用关联记忆扩散补齐
else:
    保持现有全局搜索逻辑
```

**2.2 关系权重参与排序**

`get_related()` 返回关联记忆时附带权重，调用方可以按权重过滤（如只返回 >=0.3 的），按权重降序排列。

**2.3 关联扩散**

如果场景过滤后的语义搜索结果不足 top_k，从这些结果的关联记忆中（按权重排序）补充。相当于"想到A的时候顺便想起B"。

---

### 第三层：梦境加固层（dream模块）

在现有梦境流程中新增一个 Job，在 DREAMING 阶段执行以下操作：

**3.1 共现→关系加固**

- 读 `co_occur` 表，取共现次数最高的 50 对记忆
- 如果它们之间已有关系 → 加固权重（`weight += 0.1 * (1 - weight)`）
- 如果共现 >= 3 次且无关系 → 新建一条 `co_occur` 类型的边，低权重起步（0.2）

**3.2 关系权重衰减**

- 所有超过 7 天未加固的关系 → 权重乘以 0.95
- 权重低于 0.05 的 → 标记休眠

**3.3 场景自动聚类（可选）**

- 取所有 `scene_tags` 为空的记忆（按重要性排序取前 20 条）
- 调 LLM 分析内容，判断应该属于什么场景
- 自动追加场景标签

这个步骤依赖 LLM，如果不想增加 API 调用可以关闭。

**3.4 管道位置**

在 `MemoryOrganizer.organize()` 中的执行顺序：

```
1. ReinforceJob    → 单条记忆强度加固（已有，不改）
2. ExtractJob      → 从对话提取新记忆（已有，不改）
3. RelationReinforceJob → 新增，加固关系权重 + 衰减 + 场景聚类
```

## 关键设计决策

### 做的

- 关系权重从 0 起步自然生长，不依赖人工预设
- 共现追踪是轻量的 SQLite 计数，不加额外依赖
- scene 过滤先于语义搜索，减少 LanceDB 搜索范围
- 所有新增功能都可以单独开关（在配置里控制）

### 不做的

- 不重建 LanceDB 索引（关系数据写 SQLite 即可）
- 提取 prompt 不改核心逻辑，只改格式要求
- 梦境加固不每轮都做，只在 DREAMING 阶段执行
- 不强制 LLM 判断场景（`scene_tags` 可为空，后期聚类补上）

---

## 实施顺序

1. **longterm.py** — 数据结构迁移 + 新方法，这是地基
2. **extractor.py + prompts** — 让新数据带着 scene_tags 进来
3. **recall() + get_related()** — 检索层用上场景过滤和权重排序
4. **dream 模块** — 加固 + 衰减 + 场景聚类，形成闭环

第 1~2 步做完后系统就可以正常工作了（新存的数据带场景标签，旧数据无场景标签不报错）。第 3~4 步是渐进增强。

---

## 边界情况

- 旧数据没有 scene_tags → 默认 `[]`，不参与场景过滤，只参与全局搜索
- 旧关系没有 weight → 默认 0.5，不报错
- co_occur 表数据超过 30 天 → 梦境中自动清理
- 场景过滤后一条结果都没有 → 回退到全局搜索（可选，由调用方决定）
