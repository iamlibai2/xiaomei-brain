# xiaomei-brain 项目理解文档

**日期**: 2026-04-25

# 一、项目理解

## 核心思想

xiaomei-brain 是一个**类脑架构的 AI Agent 框架**。它不是在 LLM 外面套一层薄薄的 tool-calling，而是尝试模拟人脑的分层结构，让 Agent 拥有持续的意识、记忆、情绪和目标系统。

核心隐喻：**意识如火焰**。代码维护火焰骨架（定时器、状态机、规则），LLM 负责加柴（理解、生成、涌现）。代码做代码能做的，LLM 做代码做不到的。

## 分层架构（自底向上）

```
┌─────────────────────────────────────────┐
│  Consciousness  意识系统  │  "火焰"       │  L0-L3 心跳，tick() 总控
├─────────────────────────────────────────┤
│  Purpose        前额叶层  │  "我要做什么" │  目标树，存在意义，意图理解
├─────────────────────────────────────────┤
│  Drive          边缘系统  │  "我感受如何" │  情绪/激素/欲望/激励，状态机+衰减
├─────────────────────────────────────────┤
│  Memory         记忆层    │  "我记得什么" │  对话日志/DAG摘要/长期记忆/向量搜索
├─────────────────────────────────────────┤
│  Agent          执行层    │  "我怎么做"   │  ReAct 循环，工具调用，流式输出
├─────────────────────────────────────────┤
│  Tools / Channels / Base 基础设施       │
└─────────────────────────────────────────┘
```

每层有明确的职责边界：
- **Consciousness**：总控制器，决定"什么时候做什么"。L0（每秒骨架维护）→ L1（每分钟异常检测）→ L2（动态 LLM 加柴，生成意图）→ L3（深度梦境燃烧）
- **Purpose**：目标管理，三层（Meaning → Phase Goals → Executable Goals），LLM 意图分析+自动分解
- **Drive**：边缘系统，纯规则+状态机（不调 LLM）。情绪分钟级衰减，激素小时级衰减，欲望超过阈值触发行为
- **Memory**：对话日志（SQLite+FTS5，一字不差）→ DAG 摘要（8条→叶子→高层，LLM压缩）→ 长期记忆（LanceDB 向量索引，user_id 隔离）

## 关键设计决策

1. **纯同步架构**：LLM 调用和 embedding 加载都是同步阻塞的，asyncio 事件循环不适用。用 `queue.Queue` + `threading.Thread` 代替
2. **MiniMax 单 system message 约束**：所有上下文（DAG 摘要 + 长期记忆 + 意图上下文）必须合并到一个 system prompt 里
3. **所有记忆第一人称**："我"=小美，"用户..."=用户信息。CJK token 估算 `cjk * 1.5 + ascii / 4`
4. **字符级输入清洗**：避免 `surrogatepass` 破坏 CJK 多字节边界，用 `ord(ch)` 逐字符过滤

# 二、问题分析

## 结构问题

### 1. ConsciousLiving 仍然是"上帝类"（1336→1100 行）

虽然刚移除了 8 个已吸收的方法，但它仍然承担太多：创建所有子系统、管理生命周期、消息处理、意图消费、命令路由、确认流程。所有东西都混在一个类里。

### 2. SelfImage 双重实现（最大代码异味）

`self_image.py`（665 行）和 `self_image_proxy.py`（617 行）大量重复。Proxy 号称"代理模式"，实际上把 diff/detect_anomaly/interpret_changes 逻辑几乎照抄了一遍。两者同时存在，`Consciousness` 用 Proxy，但 Proxy 内部的 `_modules` 又引用 `self_image.py` 的 `SelfImage` 类。

### 3. 两套 Agent 生命周期

`agent/living.py` 的 `AgentLiving` 和 `consciousness/conscious_living.py` 的 `ConsciousLiving` 各自实现了完整的状态机（DORMANT→WAKING→AWAKE→SLEEPING→DREAMING），没有共享基类。`AgentLiving` 基本被废弃但代码还在。

### 4. core.py 职责过重（938 行）

`Consciousness` 类包含了 LLM prompt 模板（100+ 行）、意图解析、L0-L3 所有逻辑、状态管理。prompt 模板应该抽到独立文件。

### 5. examples/ 目录混乱

30+ 个文件，测试、demo、实验脚本、应用示例全部混在一起。`tests/` 只有 3 个文件（test_agent.py, test_llm.py, test_tools.py），真正的集成测试都在 examples/ 下。

### 6. bak/ 目录已统一（2026-04-25 完成）

原本分散在 4 处（`/bak/`, `src/bak/`, `src/xiaomei_brain/bak/`, `src/xiaomei_brain/memory/bak/`），现已合并到单一 `/bak/` 目录：

```
bak/
├── agent.py, agent_manager.py, context.py, context_extractor.py
├── proactive.py, reminder.py, scheduler.py, session.py
├── config.example.yaml.bak, config.yaml.bak, gateway_rpc_server.py.bak
├── from_memory/
│   ├── dream.py
│   └── scheduler.py
└── from_src/xiaomei_brain/
    ├── agent_context.py, agent_context_extractor.py
    ├── proactive.py, tools_builtin_memory.py
```

所有源目录 `src/xiaomei_brain/bak/` 和 `src/xiaomei_brain/memory/bak/` 已删除。

## 代码质量问题

### 7. 硬编码 "xiaomei"

`agent_id = "xiaomei"` 出现在至少 20 处。框架声称支持多 Agent，但实际只有 xiaomei 能用。ConsciousLiving 构造函数里的 DriveEngine、PurposeEngine、Consciousness 全部硬编码。

### 8. 直接访问私有字段

`conscious_living.py` 直接访问 `self.consciousness._last_l2_time`、`self.consciousness._l0_count`。刚加了 `l0_count` property，但 `_last_l2_time` 仍然裸访问。

### 9. 魔术字符串上下文传递

`_get_l2_context()` 返回 `"user_idle_long"`、`"accumulated_changes"`、`"periodic"` 等字符串，然后在 `_fallback_intent()` 里用 `if context == "user_idle_long"` 做分支判断。应该用 Enum。

### 10. 没有依赖注入

所有子系统在构造函数里直接 `DriveEngine()`、`PurposeEngine()` 实例化，路径硬编码。测试时无法 mock。

### 11. 同步阻塞无法中断

LLM 调用阻塞 15-25 秒期间，整个 Agent 冻结。新的消息无法被处理。`_cancel_requested` 标志位只在 `_run_chat` 线程结束后才被检查，等于无效。

## 架构问题

### 12. tick() 统一不彻底

tick() 刚实现了统一入口，但 `_cmd_manual_fuel()` 仍然直接调用 `self.consciousness.tick_L2("manual")`，绕过了 tick() 的判断逻辑。旧的 L2/L3 API 和新 API 共存。

### 13. Drive 事件提取被移除但没重新接入

`_extract_drive_events()` 随 `_check_dynamic_fuel()` 清理一起被删了。这是 Drive 从对话中学习的唯一机制。现在 Drive 只会衰减，不会从对话中获得反馈。

### 14. 反省层缺失

设计文档里反复提到，架构讨论中也确认了位置（`consciousness/reflection/`），但代码一行没写。

### 15. 无结构化事件系统

状态变化、意图生成、目标完成——全靠直接方法调用和返回值传递。没有事件总线，模块间耦合紧密。

### 16. 配置分散

identity.md、perception.md、talent.md、drive_config.yaml、config.json——各自有独立的解析器，没有统一的配置 schema。
