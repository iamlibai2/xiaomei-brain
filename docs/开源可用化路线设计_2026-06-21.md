# xiaomei-brain 开源可用化路线设计

> 设计日期：2026-06-21

## 背景

xiaomei-brain 在架构深度和原创性上具有显著优势，但距离"能被别人轻松使用"的开源产品还有相当距离。本设计通过三个 Phase 逐步缩短这个距离，每个 Phase 交付一个用户可感知的能力跃迁。

**目标用户画像**：普通用户（想有一个有记忆、有性格的 AI 朋友） + 开发者（想理解/复用/扩展 Agent 架构）

**策略**：打磨现有，快速发布。不做架构大改，在现有代码基础上做体验和文档层的提升。

## Phase 1：「5 分钟跑起来」

**一句话**：从克隆仓库到第一次对话，不超过 5 分钟，不需要读源码。

### 1.1 `pip install` 一条命令安装

**当前状态**：安装依赖需要手动激活虚拟环境、指定 PYTHONPATH、逐个安装可选依赖。命令分散在 CLAUDE.md 中。

**目标**：
```bash
pip install xiaomei-brain
```
或开发模式：
```bash
git clone https://github.com/xxx/xiaomei-brain.git
cd xiaomei-brain && pip install -e .
```

**工作项**：
- 确认 `pyproject.toml` 依赖版本约束正确
- 可选依赖分组合理：`pip install xiaomei-brain[feishu,server,tui]`
- 修复可能的安装路径问题（`src/` 布局下的包发现）
- 安装后 `xiaomei-brain` 命令可直接使用

### 1.2 `xiaomei-brain setup` 交互式初始化

**当前状态**：配置需要手动编辑 `config.json`，格式是嵌套的自定义结构，没有文档引导。

**目标**：一个三步 wizard，用户不需要知道 `config.json` 和 `identity.md` 文件的存在。

```

    1. 温暖陪伴型 — "我在这里陪着你"
    2. 理性助手型 — "你想怎么做？我帮你分析"
    3. 活泼朋友型 — "哇！这个好有趣！"
```

**工作项**：
- 实现 `xiaomei-brain setup` 命令（或使用已有的 `agent create` 命令增强交互性）

### 1.4 Quick Start 文档

**目标**：README.md 中的 Quick Start 部分，用 5 分钟让用户完成从安装到对话的完整流程。

## Phase 3：「开发者能改」

**一句话**：有经验的 Python 开发者，半小时内理解架构、找到想改的代码、写出第一个自定义插件。

### 3.1 架构文档 — ARCHITECTURE.md

一份独立的架构总览文档，包含：

- **ASCII 架构图**：分层 + 数据流方向
- **启动流程**：`__main__` → `ConsciousLiving` → 各子系统初始化顺序
- **消息处理流程**：Gateway → Living → ConversationDriver → Agent.stream() → Router → Channel 输出
- **关键设计决策 Why**：
  - 为什么用同步而非 async？
  - 为什么 Memory 共享一个 SQLite？
  - 为什么 Drive 不做 LLM 调用？
  - 为什么 DAG 摘要用 8 条消息为叶子？
  - 为什么记忆是第一人称视角？
- **目录地图**：每个目录一句话职责说明 + 最重要的 1-2 个文件
- **新人阅读路径**：
  - "想了解聊天怎么发生 → 从 `agent/core.py:stream()` 看起"
  - "想了解记忆怎么存储 → 从 `memory/longterm.py` 看起"
  - "想了解意识骨架 → 从 `consciousness/self_image_proxy.py` 看起"
  - "想了解状态管理 → 从 `drive/engine.py` 看起"

### 3.2 核心接口 Protocol 化

不重构全部 `Any`，只给最关键的 3-5 个跨层依赖定义 Protocol：

```python
# drive/protocol.py

from typing import Protocol

class DriveProtocol(Protocol):
    """Agent 需要的 Drive 接口"""

    def on_praise(self, delta: float = 0.3) -> None: ...
    def on_criticism(self, delta: float = 0.3) -> None: ...
    def on_goal_completed(self, progress: float) -> None: ...
    def on_goal_failed(self, reason: str = "") -> None: ...
    def on_user_active(self) -> None: ...
    def on_user_idle(self, duration: float) -> None: ...
    def consume_energy(self, amount: float) -> None: ...
    def get_signals(self) -> DriveSignals: ...
    def apply_social_signal(self, signal_type: str, intensity: float) -> None: ...
```

```python
# purpose/protocol.py

class PurposeProtocol(Protocol):
    """Agent 需要的 Purpose 接口"""

    def get_current(self) -> Goal | None: ...
    def get_state_summary(self) -> str: ...
    def complete_goal(self, goal_id: str) -> None: ...
    def get_next(self) -> Goal | None: ...
```

```python
# memory/protocol.py

class LongTermMemoryProtocol(Protocol):
    """长期记忆接口"""

    def recall(self, query: str, user_id: str, top_k: int = 5, ...) -> list[dict]: ...
    def store(self, content: str, source: str, user_id: str, ...) -> int: ...
    def search_by_tags(self, tags: list[str], user_id: str, ...) -> list[dict]: ...
```

**价值**：不改变任何现有代码结构，只在类型注解上声明"这个参数需要什么方法"。IDE 立刻有自动补全，新人读代码立刻知道各层依赖关系。

## 三阶段总览

| Phase | 用户之前的状态 | 用户之后的状态 | 核心交付 |
|-------|-------------|-------------|---------|
| **1** | 不知道从哪开始 | "装上了，跑起来了，能聊了" | `pip install` + `setup` 向导 + Quick Start + Docker |
| **2** | Agent 是 demo | "它是我的小美了" | 身份定制 + 记忆可见 + 渠道连接 + 状态可视化 |
| **3** | 黑盒，不敢改 | "我理解它了，能改了" | ARCHITECTURE.md + Protocol 接口 + 插件指南 + 测试 + CI |

| Phase | 预估工作量 | 涉及文件数 | 风险 |
|-------|-----------|-----------|------|
| **1** | 2-3 周 | ~15 文件（新增 CLI 命令 + 文档） | 低 — 纯增量，不改核心逻辑 |
| **2** | 2-3 周 | ~20 文件（CLI 命令 + 文档 + 模板） | 低 — 主要是暴露已有能力 |
| **3** | 3-4 周 | ~30 文件（Protocol + 测试 + 重构 Bootstrapper） | 中 — Bootstrapper 提取可能影响初始化行为 |
