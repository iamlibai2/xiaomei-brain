## WS CLI 重构设计（参照 Hermes）

### 架构

**Hermes 的核心模式**：`prompt_toolkit` full_screen=False + Rich 渲染内容

```
┌─ 终端窗口 ───────────────────────────┐
│                                       │
│  [banner / 历史消息]                   │  ← Rich ANSI 输出
│  [agent] 回复内容...                   │     (scrolls naturally)
│                                       │
│  ─────────────────────────────────── │
│  [bosshi]> _                          │  ← prompt_toolkit 输入行
└───────────────────────────────────────┘
```

本质上就是把 Rich 渲染的内容打印到终端，让它在 prompt_toolkit 输入行上方自然滚动。**不需要 Rich Live/Layout**，结构简单得多。

### 关键设计点

**1. 输出渲染：Rich → ANSI → prompt_toolkit**

Hermes 的做法：
- 用 `rich.console.Console` 把 Markdown 渲染成 ANSI 字符串
- 通过 `prompt_toolkit` 的 `print_formatted_text(ANSI(...))` 输出
- 这样输出内容不会被 `patch_stdout` 吞掉

**2. 流式输出：SimpleMarkdownRenderer（精简版）**

Hermes 有 470 行的 `StreamingMarkdownRenderer`。我们不需要那么复杂，核心逻辑：
- 接收 chunk → 累积到 buffer
- 按行或按段落边界渲染成 ANSI
- 用 `\r` + ANSI 清行 + 重写来更新当前行（非阻塞刷新）
- 完整消息到达后做最终渲染

**3. 输入：TextArea + 多行**

Hermes 用 `TextArea` widget 配合 `patch_stdout`。我们保持现有的 `PromptSession` + thread executor 结构，加 Alt+Enter 多行支持（prompt_toolkit 的 `PromptSession` 原生支持）。

**4. 状态栏**

用 Rich 渲染状态行，固定在输入行下方或嵌入 prompt。简洁方案是每次输出后打印一个状态行到输入区上方。

### 文件变更

- `examples/ws_cli.py` ~ 从 230 行重写到 ~400 行
- 新增依赖：无（`rich` 已安装）

### 核心结构

"Shim" 就是一个薄薄的代理层，自己不干活，只负责把旧路径转发到新路径：

```python
# channels/ws/__init__.py — 不干实活，只是转发
from ...gateway.ws_adapter import WSAdapter    # ← 真正代码在这

def register(ctx):
    ctx.register_channel("ws", WSAdapter(cm))  # ← 插件系统调用它
```

它存在的唯一原因：插件系统启动时会自动扫描 `channels/` 目录，找到 `ws/` 就 import 并调 `register()`。如果删了这个 shim，插件系统找不到 `"ws"` 频道，Router 就没有 WSAdapter，所有 WS 回复都推不出去。

这不是永久方案。以后插件系统如果能配置从 `gateway/` 加载 WS，这个目录就可以彻底删掉。但现在删了会断。
