搜索到的关键信息总结：

**prompt_toolkit 官方的标准方案就是当前实现的方向**：

1. **[prompt-toolkit Issue #1286](https://github.com/prompt-toolkit/python-prompt-toolkit/issues/1286)** — 维护者 @jonathanslenders 确认的模式：
   - 用 **callable 作为 prompt message**（每次重绘时重新求值）
   - 后台线程更新共享状态，调 **`app.invalidate()`** 触发重绘
   - 这就是我实现的方案

2. **`run_in_terminal` 的固有限制**：每次调用都要 擦除 prompt → 写文本 → CPR 定位 → 重绘 prompt，这是一个串行化的异步操作链。在流式场景下（10Hz+），这些 future chain 互相排队，字面量的丢失和闪烁是这种架构的必然结果。`patch_stdout` 和 `print_formatted_text` 底层都走 `run_in_terminal`，所以无论怎么调参都解决不了。

3. **另一个可选方案**是设置 `session.message = FormattedText(...)` 从后台线程直接改，然后 `session.app.invalidate()`。这和 callable 方案本质上一样，只是设置属性 vs 函数返回值。

**当前实现的架构是这样的**：

```
后台线程 _flush_chunks()
  → self._streaming_text += text        (写共享状态)
  → self._app.invalidate()              (触发重绘)

主线程 prompt_message() 被渲染器调用
  → 读取 self._streaming_text           (读共享状态)
  → to_formatted_text(ANSI(streaming))  (转换ANSI→样式元组)
  → 返回 FormattedText                  (直接渲染在 prompt 上方)
```

这个路径完全在 prompt 自身的渲染循环内完成，不经过 `run_in_terminal`。你先测试看看实际效果。
