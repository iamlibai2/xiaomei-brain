## OpenClaw 的设计分析

OpenClaw 关键发现：**聊天和配置走同一个 WebSocket 连接**

### OpenClaw Gateway 架构

```
openclaw gateway (同一个进程)
│
├── WebSocket (同一个端口 18789)
│   ├── 聊天协议: session_start, chat, text_chunk...
│   └── 配置 RPC: config.get, config.patch, logs.tail...
│
├── Channel 管理 (feishu, discord, slack...)
└── Agent/会话管理
```

**所有操作都通过 WebSocket**，而不是分开的管理端口。

### OpenClaw CLI 两种模式

| 命令 | 做什么 |
|------|--------|
| `openclaw config get <path>` | **离线** 直接读文件 |
| `openclaw gateway call <method>` | **在线** 通过 WebSocket RPC 调用 Gateway |

### OpenClaw 配置管理流程

```
1. openclaw config set agents.defaults.heartbeat.every "2h"
   ↓ (离线模式，直接修改 openclaw.json)

2. openclaw gateway call config.patch --params '{"raw": "{...}", "baseHash": "..."}'
   ↓ (在线模式，通过 WebSocket RPC)
```

### 对比我们的设计

| | OpenClaw | 我们（当前） |
|---|---|---|
| 聊天 | `/ws` 端点 | `/ws` 端点 |
| 配置 | `gateway call config.*` 通过同一 WebSocket | `/rpc` 端点（独立） |
| CLI 离线 | `config get/set` 直接文件 | `cli.py config get/set` 直接文件 |

### 关键区别

OpenClaw 的 **config RPC 也是通过 Gateway WebSocket** 调用，不是独立端点：

```bash
# OpenClaw：配置 RPC 和聊天共用 gateway WebSocket
openclaw gateway call config.get --params '{}'
openclaw gateway call config.patch --params '{"raw": "...", "baseHash": "..."}'
```

这意味着 OpenClaw 的配置管理和聊天都在同一个连接里，只是不同类型的消息。

## 设计考量：统一 vs 分离

### OpenClaw 选择统一的原因

```
客户端                  Gateway
   │                       │
   ├─ session_start ───────┼─> 建立会话
   ├─ chat ────────────────┼─> 对话
   │                       │
   ├─ config.get ──────────┼─> 配置查询
   ├─ config.patch ───────┼─> 配置修改（其他人改，全员收到）
   │                       │
   └─ ping ────────────────┼─> 心跳
```

**核心优势：**
1. **实时同步** - 任何人通过 RPC 修改配置，所有在线客户端立即收到更新
2. **连接复用** - 不用为管理操作再开一个连接
3. **统一认证** - 一套 token/password
4. **代码复用** - 消息路由、认证、错误处理共用一套代码

### 分离的考量

| | 统一 | 分离 |
|---|---|---|
| 实时配置推送 | ✅ 自动推送 | ❌ 需要额外机制 |
| 复杂度 | 消息路由、分类型处理 | 两套独立逻辑 |
| 隔离性 | chat 和管理带宽共享 | 完全隔离 |
| 扩展性 | 管理操作可能影响 chat | 互不影响 |

### 我们的情况

我们当前分离式设计（`/ws` 聊天、`/rpc` 配置）适合：

- **管理操作少** - 配置不频繁修改
- **独立部署** - RPC 服务可以单独重启
- **简化初期** - 两套逻辑清晰分开

但**长期来看**，如果需要：
- 配置修改后聊天客户端立即生效
- 简化客户端连接

**统一是更好的选择**。

## 助手回答（2026-04-15 00:18:26）



好问题。核心原因是：**这是两种不同的通信模式，本质上不兼容。**

## 两种通信范式

### Chat - 流式/事件驱动

```
客户端 ──> chat {content: "你好"}
客户端 <── text_chunk {content: "你"}
客户端 <── text_chunk {content: "好"}
客户端 <── text_done {}
```

- 服务端**主动多次推送**
- 消息带有 `type` 类型标记
- 是**事件流**，不是请求-响应

### Config RPC - 请求-响应

```
客户端 ──> {method: "config.get", params: {...}, id: 1}
客户端 <── {result: {...}, id: 1}
```

- 一问一答，**严格一一对应**
- 每次请求必须返回（JSON-RPC 规范）

## 为什么不能统一

| 特性 | Chat 流式协议 | JSON-RPC |
|------|-------------|----------|
| 响应次数 | 无限次（流） | 一次 |
| 消息匹配 | `type` 字段区分 | `id` 字段匹配 |
| 典型场景 | 实时对话 | 配置查询 |
| 标准 | 自定义 | JSON-RPC 2.0 标准 |

**流式和请求-响应是矛盾的需求**：
- 如果把 `text_chunk` 强行套成 JSON-RPC，你需要不断发通知（notification），但每次流式推送的 chunk 并不是某个调用的"返回值"
- JSON-RPC 的 `id` 是为了匹配请求和响应，但 chat 服务端主动发的消息不服务于任何客户端请求

## 一句话

> Chat 是**实时事件流**，Config RPC 是**请求-响应**。这是两种不同的通信模式，强行统一反而更复杂。

| **sessions** | 两个进程同时读写同一会话文件 |
| **embedding 模型** | 加载两次到内存（内存浪费） |
| **dream_scheduler** | 重复后台处理，可能互相干扰 |
| **context_extractor** | 重复处理相同数据 |

## 严重问题：文件锁

```
basic_agent.py  ──→  写入 memory/conversations/xxx.jsonl
ws_server.py    ──→  同时写入同一文件  ──→  数据损坏
```

## 正确的多客户端方式

不是启动两个进程，而是：

```
ws_server.py (单进程)
    │
    ├── WebSocket 连接 1 (客户端 A)  ──→ default agent
    ├── WebSocket 连接 2 (客户端 B)  ──→ default agent
    └── WebSocket 连接 3 (客户端 C)  ──→ xiaomei agent
```

**多个客户端连接同一个 ws_server**，共享 agent 的 LLM、memory 等资源。

## 如果需要远程访问

应该用 OpenClaw 那样的分布式架构：
- gateway 一个进程
- agent 另一个进程
- 通过 WebSocket 通信

**当前我们的架构不支持这样**。

## FastAPI 的作用

FastAPI 在 `ws_server.py` 中负责**HTTP/WebSocket 服务器框架**：

```python
# ws/server.py
from fastapi import FastAPI, WebSocket, WebSocketDisconnect

app = FastAPI(title="xiaomei-brain WebSocket Server")

@app.get("/health")
async def health() -> dict:
    return {"status": "ok"}

@app.websocket("/ws")
async def ws_endpoint(ws: WebSocket) -> None:
    await ws.accept()
    ...
```

## 核心功能

| 功能 | 实现 |
|------|------|
| **路由** | `/health`, `/ws` 等端点 |
| **WebSocket** | `@app.websocket("/ws")` 处理连接 |
| **并发** | asyncio 异步事件循环 |
| **生命周期** | startup/shutdown 钩子 |
| **错误处理** | 自动捕获并返回错误响应 |

## 为什么用 FastAPI

1. **自带 WebSocket 支持** - 比 uvicorn 原生 API 更简洁
2. **路由方便** - 同一个 app 可以同时有 HTTP 和 WebSocket 端点
3. **异步原生** - 非阻塞 I/O
4. **健康检查** - `/health` 端点

## 底层

```
FastAPI (高层 API)
    ↓
uvicorn (ASGI 服务器)
    ↓
asyncio (事件循环)
```

### 五、关键设计原理

#### 1. 线程/协程隔离

```
主事件循环 (asyncio.run(main()))
│
├── _process_messages 协程 (async)
│   └── 从 queue.Queue() 读取 → 处理消息
│
└── WebSocket 线程 (threading.Thread, daemon=True)
    └── lark-oapi SDK 专用事件循环
```

**原理**: 
- `lark-oapi` SDK 在独立线程中运行，避免与主事件循环冲突
- `queue.Queue()` 实现线程间消息传递（线程安全）
- 协程处理业务逻辑，线程处理 IO

#### 2. 路由机制

```python
# 路由查找顺序：
def _get_agent_for_msg(msg):
    # 1. 显式指定 (WebSocket)
    if msg.extra.get("agent_id"):
        return get_agent(msg.extra["agent_id"])
    
    # 2. Bindings 路由 (Channel)
    account_id = msg.extra.get("account_id", "default")
    for binding in self._bindings:
        if binding["match"]["channel"] == msg.platform and \
           binding["match"]["accountId"] == account_id:
            return get_agent(binding["agentId"])
    
    # 3. 默认
    return get_agent("xiaomei")
```

**配置示例**:
```json
{
  "channels": {
    "feishu": {
      "accounts": {
        "xiaomei": {"appId": "cli_xxx"},  → 绑定到 xiaomei
        "default": {"appId": "cli_yyy"}     → 绑定到 xiaoming
      }
    }
  },
  "bindings": [
    {"agentId": "xiaomei", "match": {"channel": "feishu", "accountId": "xiaomei"}},
    {"agentId": "xiaoming", "match": {"channel": "feishu", "accountId": "default"}}
  ]
}
```

#### 3. Lazy Loading

```python
class AgentManager:
    def build_agent(self, agent_id, config):
        instance = self._agents.get(agent_id)  # 仅元数据
        if instance.llm is None:              # 首次访问才加载
            instance.llm = LLMClient(...)
            instance.memory = MemoryStore(...)
            instance.tools = ToolRegistry(...)
        return instance
```

**优点**:
- 启动快：不预先加载所有 LLM/Memory
- 按需：只有用到的 agent 才初始化
- 节省：不用的 agent 不占资源

#### 4. Agent 构建流程

```
AgentInstance (元数据)
    │
    ├── llm: None → LLMClient(model, api_key)
    ├── tools: None → ToolRegistry() + 注册工具
    ├── memory: None → MemoryStore(embedder, index)
    ├── session_manager: None → SessionManager(session_dir)
    └── working_memory: None → WorkingMemory()
    │
    ↓
Agent (完整执行引擎)
    ├── llm: LLMClient
    ├── tools: ToolRegistry (包含 shell, file_ops, memory...)
    ├── memory: MemoryStore (向量检索)
    ├── dream_scheduler: DreamScheduler (定期处理记忆)
    ├── proactive_engine: ProactiveEngine (主动消息)
    ├── reminder_manager: ReminderManager (提醒管理)
    └── context_extractor: ContextExtractor (上下文提取)
```

### 八、关键代码解释

```python
# AgentGateway: 核心路由逻辑
class AgentGateway(Gateway):
    def __init__(self, agent_manager, config, bindings):
        self.agent_manager = agent_manager  # Agent 工厂
        self.config = config              # 全局配置
        self._bindings = bindings         # 路由规则
        self._agents = {}               # Agent 缓存

    def _get_agent_for_msg(self, msg):
        # 懒加载 + 缓存
        agent_id = self._resolve_agent_id(msg)
        if agent_id not in self._agents:
            instance = self.agent_manager.build_agent(agent_id, self.config)
            self._agents[agent_id] = self._build_full_agent(instance)
        return self._agents[agent_id]
```

## 5. Problem Solving

**Identified Core Issue**: The streaming response mechanism in `ws/server.py` is broken due to nested `asyncio.run()` inside `loop.run_in_executor()` creating a separate event loop that doesn't communicate with the main event loop where WebSocket messages need to be sent.

**Why This Fails**: When `asyncio.run()` creates a new event loop inside the executor thread:
1. The `run_stream()` function is decorated as `async def run_stream():`
2. Inside it, `for chunk in client_session.agent.stream(content):` iterates synchronously
3. `asyncio.create_task(send(...))` is called inside this loop
4. These tasks are scheduled in the NEW nested event loop
5. But `run_stream()` completes synchronously and returns to executor
6. The new nested event loop doesn't get a chance to run tasks because it closes when `run_stream()` returns
7. When back in the main event loop, `await send(build_msg(...))` is called, but these sends are in the wrong event loop

**Solution Path**: The streaming should be done in the main event loop, not inside a nested event loop. Either:
- Remove `asyncio.run()` wrapper and run `run_stream()` directly in the main event loop
- OR collect all chunks synchronously in executor, then send them from main event loop
