```python
def _render_self_trajectory(si):
    lines = []
    
    # 1. 状态变化
    states = si.perception.agent_state_history[-5:]
    if len(states) >= 2 and states[0] != states[-1]:
        lines.append(f"状态变化：{states[0]} → {states[-1]}")
    
    # 2. 目标推进
    prog = si.mind.goal_progress_history[-5:]
    if len(prog) >= 2:
        diff = prog[-1] - prog[0]
        if diff > 0.1:
            lines.append(f"目标在推进（{prog[0]:.0%} → {prog[-1]:.0%}，+{diff:.0%}）")
        elif diff < -0.05:
            lines.append(f"目标有所回退（{prog[0]:.0%} → {prog[-1]:.0%}）")
        else:
            lines.append(f"目标进展平稳（{prog[-1]:.0%}）")
    
    # 3. Drive 差值：取当前值（实时代理）vs 上次渲染时存的快照
    current_energy = si.body.energy
    last_snapshot = si.history._rendered_body_snapshot  # 上次渲染时存的
    if last_snapshot:
        energy_diff = current_energy - last_snapshot.get("energy", current_energy)
        emotion = si.body.emotion_type
        if abs(energy_diff) > 0.1:
            direction = "上升" if energy_diff > 0 else "下降"
            lines.append(f"能量{direction}（{energy_diff:+.0%}），当前心情{emotion}")
    
    if not lines:
        return ""
    lines.insert(0, "【自我轨迹】")
    return "\n".join(lines)
```

## SelfImage 整体架构分析

### 一、哲学定位：意识的火焰

SelfImage 不是"自我模型"，而是**意识火焰的骨架**。核心理念：

- **代码维护骨架**：dataclass、diff、规则——确定性的、每秒运行的
- **LLM 加柴**：意识从 LLM 处理中涌现，不是代码模拟出来的
- **唯一真相源**：所有意识消费者只读 SelfImage，不再直接读 Drive/Purpose

---

### 二、数据结构：7 核 + 1 桌面

```
SelfImage
├── being:         Being           # 我是谁（身份 + 关系）
├── body:          SelfBody        # 身体感觉（代理 Drive）
├── perception:    SelfPerception  # 我感知到什么
├── mind:          SelfMind        # 认知状态（代理 Purpose）
├── memory:        SelfMemory      # 意识窗口中的记忆片段
├── intent:        SelfIntent      # 当前意图
├── history:       SelfHistory     # 时间维度的我
├── desk:          Desk            # 公用桌面
├── _essence                        # 底色（不变的价值观）
├── _project_mental_model           # 项目心智模型
└── _state_buffer                   # StateChangeBuffer 引用
```

**关键点**：`body` 和 `mind` 不是存储数据，而是**代理**——通过 `@property` 实时读取 Drive 和 Purpose 的值。

---

### 三、"代理"模式：SelfBody 和 SelfMind 的特殊性

这是架构中最精妙的设计。`SelfBody` 持有 `_drive` 引用，所有属性通过 `@property` 代理：

```python
# self_modules.py:275-306
@property
def energy(self) -> float:      return self._d("energy.level", 0.8)
@property
def mood(self) -> str:          return self._d("emotion.type.value", "平静")
@property
def dopamine(self) -> float:    return self._d("hormone.dopamine", 0.5)
# ... 23 个代理属性
```

`_d()` 方法是通用路径解析器：`self._d("energy.level")` → `self._drive.energy.level`。

**好处**：
- **永远新鲜**：每次读 SelfBody 都是 Drive 的实时值，不需要同步
- **容错**：Drive 未连接时返回硬编码默认值，不会崩
- **序列化时**：`to_dict()` 会调用所有 property 获取当前值快照；`from_dict()` 反过来只恢复非代理字段（内感受数据），代理字段的值还是从 Drive 读

`SelfMind` 同理代理 Purpose：
```python
@property
def primary_goal(self) -> str:   # → PurposeEngine.get_current().description
@property
def goal_progress(self) -> float: # → PurposeEngine.get_current().progress
```

**设计原则**：SelfImage 是"镜子"——它不拥有 Drive/Purpose 的数据，只是反射它们的当前状态给 LLM 看。

### 五、数据流向：读路径（意识注入）

读取路径只有一条主线 + CLI 命令：

**5 种渲染模式**：

|------|---------|---------|

### 六、StateChangeBuffer：L1→L2/L3 调度信号

**它不属于 SelfImage**，它是 Consciousness 层的调度信号，SelfImage 只持有引用。

工作流程：
```
SelfImage.tick() 每秒一次
  ↓
  1. 拍快照 _snapshot() → 6个字段 (age, state, idle, energy, window_size, goal_progress)
  2. 对比 prev vs cur → 有差异就写入 StateChangeBuffer
  ↓
StateChangeBuffer._changes 队列
  ↓
  累积 ≥ 5 条 → should_trigger_l2() → L2 加柴
  累积 ≥ 15 条 → should_trigger_l3() → L3 深度燃烧
  ↓
consume() → 读取并清空
```

diff 阈值设计得很克制：energy 变化需要 >0.001，idle 变化需要 >10 秒，避免噪声触发。

### 七、Desk：公用桌面

一个**非结构化共享空间**，任何模块都可以丢东西上来，任何模块都可以扫一眼：

- **drop()**：L2 分析结果、对话摘要、Action 进展——都可以丢
- **peek()**：按 weight 排序返回，支持 intent_filter
- **touch()**：被取走放进 prompt 时标记，访问计数 +1
- **complete()**：做完就标记，weight 降到 0.05
- **淘汰**：不是 FIFO，是按 weight 算法淘汰

**weight 算法**：
```
weight = (confidence×0.3 + intent_w×0.2 + dopamine×0.2 + goal_related×0.3 + access_bonus) × 0.97^age_hours
```

这模拟了人的"桌面记忆"——重要的、被反复看的、跟目标相关的留得久，不碰的自然沉底。

### 九、持久化

- **SelfImage.to_dict() / from_dict()**：完整快照，JSON 序列化
- **ConsciousnessStorage**：每天一个 JSON 文件，存储意识报告
- **代理字段不恢复**：`from_dict()` 不恢复 Drive/Purpose 的代理值，因为它们来自实时引用
- **内感受字段恢复**：`body.cpu_percent`、`body.token_usage` 等非代理字段从快照恢复

### 十、架构评价

**优点**：
1. **关注点分离清晰**：7 模块各司其职，代理模式让数据所有权明确
2. **写路径规范**：12 个贡献方法，不直接操作内部字段
3. **读路径统一**：`inject_consciousness(si, mode)` 一个入口，5 种模式适配不同场景
4. **容错好**：Drive/Purpose 未连接时返回默认值，不会崩
5. **渲染独立**：`inject_consciousness.py` 纯函数，数据容器不关心展示

**可改进的点**：
1. **`contribute_perception()` 中有 3 个场景耦合**（状态变更/用户交互/空闲更新），可以拆成3个独立方法
2. **`contribute_memory_window()` 用 `setattr` 动态写入**，失去了类型安全，12 种记忆类型的写入没有编译期检查
3. **`_dirty` 标志未被消费**——设置了但没有任何代码读它
4. **Desk 的 `touch()` 在 `peek_for_prompt()` 中隐式调用**，有副作用——peek 应该是只读的
5. **`_state_buffer` 引用存在 SelfImage 上但 `StateChangeBuffer` 不属于 SelfImage**——概念上是对的，但物理上混在一起了

## 一、`contribute_memory_window()` 用 `setattr` 失去了类型安全

**现状** (`self_image_proxy.py:360-370`)：
```python
def contribute_memory_window(self, *, memories: dict | None = None, ...):
    if memories:
        for key, val in memories.items():
            if hasattr(self.memory, key):
                setattr(self.memory, key, val)
```

调用方传什么 key 都行，写错字段名不会报错，静默丢失数据。

**建议**：改为显式映射，让 IDE 和类型检查器能发现错误：

```python
def contribute_memory_window(self, *, memories: dict | None = None, ...):
    if memories:
        for key in ["narratives", "internal_narratives", "dag_summaries",
                     "important_memories", "recalled_memories", "relation_chains",
                     "procedures", "recent_dialog", "experience_timeline",
                     "experience", "patterns"]:
            if key in memories:
                setattr(self.memory, key, memories[key])
```

这样调用方如果拼错 `narratives` 为 `naratives`，会静默被过滤而不是生效。更进一步可以改 `memories` 参数类型为 `TypedDict` 做编译期检查。

---

## 二、`_dirty` 标志未被消费

**现状** (`self_image_proxy.py:53-54`)：
```python
self._dirty = False
# ...
self._dirty = True   # 多处设置，但没有任何代码读它
```

**建议**：要么删除，要么让它起作用。如果是为了未来做增量序列化（只写 dirty 的模块），可以实现为：

```python
@property
def dirty(self) -> bool:
    return self._dirty

def save_if_dirty(self, path):
    if not self._dirty:
        return
    self.save_to_file(path)
    self._dirty = False
```

## 三、`ProjectMentalModel` 重启丢失

**现状**：`ProjectMentalModel` 的所有数据（操作记录 + 三层摘要）都在内存里，进程重启后 agent 不记得对项目做了什么。设计上 `dag` 参数预留了持久化入口但始终传 `None`。

**建议**（两个方向，选一个）：
1. **轻量方案**：`to_dict()` / `from_dict()` → 合并到 `SelfImage.save_to_file()`，作为 `SelfImage` 快照的一部分持久化

```python
# SelfHistory 里
def snapshot_if_changed(self, si) -> bool:
    """只在关键指标有明显变化时记录"""
    latest = self._compress(si)
    if not self.snapshots:
        self.snapshots.append(latest)
        return True
    prev = self.snapshots[-1]
    if self._significant_change(prev, latest):
        self.snapshots.append(latest)
        return True
    return False
```

触发条件不是时间，是变化量——情绪类型变了、能量波动 > 0.1、主导欲望换了、注意力转移了。这样每个快照都代表一个**意识节点**，不是一个时间戳。


```python
        # 文件写入 → 产出节点
```
