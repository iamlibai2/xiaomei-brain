## 项目概述

**xiaomei-brain** 是一个通用 AI Agent 框架，基于 Python + 火山引擎（Volcengine）API。核心是 **ReAct 循环**：LLM 推理 → 决定是否调用工具 → 执行工具 → 观察结果 → 继续推理或返回答案。

## 文件说明

```
src/xiaomei_brain/
├── __init__.py          # 包入口，导出所有公开类
├── agent.py             # 核心：Agent 的 ReAct 主循环
├── llm.py               # LLM 客户端，调用火山引擎 API
├── config.py            # 配置管理（provider、model、api_key 等）
└── tools/
    ├── __init__.py       # 导出 Tool、tool、ToolRegistry
    ├── base.py           # Tool 基类 + @tool 装饰器
    ├── registry.py       # 工具注册表（注册、查找、执行、转 OpenAI 格式）
    └── builtin/
        ├── __init__.py   # 导出内置工具
        ├── shell.py      # 内置工具：执行 shell 命令
        └── file_ops.py   # 内置工具：读写文件
```

### 各文件详解

**`agent.py`** - 核心 Agent 类
- `Agent.run(user_input)` 执行主循环
- 每步：把消息发给 LLM → 有 tool_calls 就执行工具 → 把结果喂回 LLM → 重复
- 无 tool_calls 则返回文本响应
- `max_steps` 防止无限循环
- `reset()` 清空对话历史

**`llm.py`** - LLM 接入层
- 用 `requests` 直接调用火山引擎 API（OpenAI 兼容格式）
- `LLMClient.chat()` 发消息，返回 `ChatResponse`
- `ChatResponse` 包含 `content`（文本）和 `tool_calls`（工具调用列表）
- `ToolCall` 解析工具名和参数

**`config.py`** - 配置
- 默认 `provider=volcengine`, `model=doubao-pro-4k`
- 自动从 `VOLCENGINE_API_KEY` 或 `OPENAI_API_KEY` 环境变量读取密钥

**`tools/base.py`** - 工具定义
- `Tool` 类：name、description、parameters（JSON Schema）、func
- `@tool` 装饰器：从函数签名自动生成 JSON Schema

**`tools/registry.py`** - 工具注册表
- `register(tool)` 注册工具
- `execute(name, **kwargs)` 按名称执行工具
- `to_openai_tools()` 转为 OpenAI function calling 格式（给 LLM 用）

**`builtin/shell.py`** - 内置 shell 工具
- 执行 shell 命令，30 秒超时，返回 stdout/stderr

**`builtin/file_ops.py`** - 内置文件工具
- `read_file`: 读取文件内容
- `write_file`: 写入文件

## 记忆系统架构与技术原理

### 整体架构

```
用户输入 "我去过日本旅游"
        │
        ▼
┌─────────────────────────────────────────────┐
│  Agent.run()                                │
│                                             │
│  1. memory.search(query)  ←── 自动触发      │
│        │                                    │
│        ▼                                    │
│  ┌─────────────────────────────┐            │
│  │  MemoryStore                │            │
│  │                             │            │
│  │  Embedder.embed(query)      │            │
│  │      → 查询向量 [0.12, ...] │            │
│  │                             │            │
│  │  VectorIndex.search()       │            │
│  │      → cosine 相似度排序    │            │
│  │      → 返回 top_k 主题      │            │
│  └─────────────────────────────┘            │
│        │                                    │
│        ▼                                    │
│  2. 注入 system prompt                      │
│     "你是一个少妇...                         │
│      ## Relevant Memory                     │
│      ### travel                             │
│      去过日本东京、京都..."                    │
│        │                                    │
│        ▼                                    │
│  3. LLM 基于增强上下文回答                    │
└─────────────────────────────────────────────┘
```

### 三层结构

| 层 | 文件 | 职责 |
|---|---|---|
| **索引层** | `indexer.py` | 管理 MEMORY.md 索引文件（主题名+摘要） |
| **存储层** | `store.py` | 统一入口：读/写/搜索记忆 |
| **搜索层** | `search.py` | 向量化 + 相似度计算 |

### 存储结构（磁盘）

```
~/.xiaomei-brain/memory/
├── MEMORY.md              # 索引：列出所有主题及摘要
└── topics/                # 每个主题一个 .md 文件
    ├── travel.md          # 旅行偏好
    ├── cooking.md         # 调试技巧
    └── hobby.md           # 爱好
```

纯文本 markdown 存储，可直接人工阅读和编辑。

### 语义搜索技术原理

**1. Embedding（向量化）**

用 `sentence-transformers` 的 `all-MiniLM-L6-v2` 模型，将任意文本转为 384 维浮点向量：

```
"我去过日本旅游" → [0.023, -0.041, 0.087, ..., 0.012]  # 384个浮点数
```

语义相近的文本 → 向量在空间中距离近。这个模型 80MB，本地运行无需 GPU。

**2. VectorIndex（向量索引）**

初始化时，对所有主题文件内容做 embedding，构建索引：

```python
# 索引构建
texts = ["travel: 旅行偏好...内容前200字", "cooking: 调试技巧..."]
vectors = embedder.embed_batch(texts)  # 一次性向量化所有主题
# 存为两个列表: texts[] 和 vectors[][]
```

**3. Cosine 相似度搜索**

查询时：

```python
query_vector = embedder.embed("我去过哪些地方")  # 查询向量化

# 对每个主题向量计算余弦相似度
score = dot(query_vector, topic_vector) / (norm(q) * norm(t))

# 按分数降序排列，取 top_k
```

余弦相似度范围 [-1, 1]，越大越相关。直观理解：两个向量方向越一致，语义越相关。

**4. 懒加载**

模型和索引都是懒加载的：
- `Embedder`: 首次调用 `embed()` 时才加载 sentence-transformers 模型
- `VectorIndex`: 首次调用 `search()` 时才扫描磁盘构建索引

避免启动时阻塞。

### 两条数据流

**读取流（自动）：**
```
用户消息 → embed → cosine search → top_k 主题 → 注入 system prompt → LLM 回答
```

**写入流（需 LLM 主动调用工具）：**
```
LLM 调用 memory_save("hobby", "喜欢看电影") → 写 topics/hobby.md → 更新 MEMORY.md 索引 → 更新内存向量索引
```

### 当前局限

1. **写入靠 LLM 自觉** — 没有自动提取和保存机制
2. **向量索引全在内存** — 重启后需重建，主题多了会慢
3. **粗粒度** — 整个主题文件作为一个检索单元，无法定位到具体段落

### 当前问题 vs 优化方向

| # | 问题 | 优化方案 | 复杂度 |
|---|---|---|---|
| 1 | 写入靠 LLM 自觉 | 自动提取 + prompt 指令双保险 | 中 |
| 2 | 每次重启重建向量索引 | 持久化索引到磁盘 | 低 |
| 3 | 整篇主题作为检索单元 | 按段落 chunk 分片检索 | 中 |
| 4 | 无相似度阈值过滤 | 低于阈值的记忆不注入 | 低 |
| 5 | 记忆无更新/去重 | 语义去重 + 合并更新 | 中 |
| 6 | 无记忆衰减机制 | 时间衰减权重 | 低 |
| 7 | save 时全量重建索引 | 增量更新向量索引 | 低 |
| 8 | 无记忆容量控制 | LRU 淘汰 + 容量上限 | 低 |

### 逐项详解

**1. 自动提取记忆**

每轮对话结束后，自动让 LLM 判断是否有值得记住的信息：

```python

只提取客观事实、用户偏好、重要决定。
如果没有值得记住的，回复 EMPTY。

用户: {user_input}
助手: {response}"""

extracted = self.llm.chat(messages=[{"role": "user", "content": extraction_prompt}])
if extracted.content != "EMPTY":
    # 自动保存
    self.memory.save(topic=auto_topic, content=extracted.content)
```

同时在 system prompt 中加指令，让 LLM 主动调用 `memory_save` 作为双保险。

**2. 持久化向量索引**

```python
# 保存
index_path = os.path.join(self.memory_dir, "index.faiss")
# 用 pickle 或 numpy 保存 vectors + texts
np.savez(index_path, vectors=self.vectors, texts=self.texts)

# 加载时直接读磁盘，跳过重建
```

启动从秒级降到毫秒级。

**3. 段落级 chunk 分片**

```python
# 当前: 一整篇作为一个检索单元
"travel.md" → 1个向量

# 优化: 按段落/标题分片
"travel.md" → 3个向量
  chunk_1: "# 旅行偏好\n我最喜欢自由行..."
  chunk_2: "## 去过的地方\n日本东京..."
  chunk_3: "## 喜欢的食物\n日本拉面..."
```

检索精度从"整篇相关"提升到"具体段落相关"。

**4. 相似度阈值过滤**

```python
# 当前: top_k 无论多低分都返回
# 优化: 低于阈值的不注入，避免噪音
MIN_SCORE = 0.3  # cosine 相似度阈值

results = [r for r in results if r.score >= MIN_SCORE]
```

**5. 语义去重 + 合并更新**

```python
# 保存前先搜索是否已有相似主题
existing = memory.search(topic_content, top_k=1)
if existing[0].score > 0.85:
    # 合并: 读取旧内容 + 追加新信息
    old = memory.read(existing[0].topic)
    merged = merge_content(old, new_content)
    memory.save(existing[0].topic, merged)
else:
    memory.save(new_topic, new_content)
```

**6. 时间衰减**

```python
# 越久未访问的记忆，权重越低
import time
decay = math.exp(-0.01 * days_since_access)
final_score = cosine_score * decay
```

**7. 增量索引更新**

当前 `save()` 已经是增量的（只更新/新增单个向量），但 `_ensure_index()` 初次加载时是全量扫描。优化为：

```python
# 记录已索引的文件列表
if cached_index_exists:
    load_from_cache()
    # 只索引新增/修改的文件
    new_files = [f for f in all_files if f not in indexed_files]
```

**8. 容量控制**

```python
MAX_TOPICS = 100
MAX_TOPIC_SIZE = 5000  # chars

if len(memory.list_topics()) >= MAX_TOPICS:
    # 淘汰最久未访问的
    oldest = min(topics, key=lambda t: t.last_accessed)
    memory.delete(oldest.topic)
```

| 3 | 段落级 chunk 分片 | `ChunkedVectorIndex` + `chunk_markdown()`，按 ## 标题+段落分片 |
| 6 | 时间衰减 | `_apply_decay()`: `score * exp(-0.005 * days)`，6个月未访问分数减半 |
| 8 | 容量控制 | `max_topics=100` + `max_topic_chars=5000`，LRU 淘汰最久未访问 |

## 助手回答（2026-04-13 03:51:23）

## 缓存持久化技术原理

### 核心问题

每次启动时，MemoryStore 需要把所有主题文件变成向量才能搜索。这个过程很慢：

```
启动 → 读5个.md文件 → 调用 embedding 模型计算9个chunk向量 → 可搜索
                          ↑
                     这步最耗时（模型推理）
```

### 解决思路

向量算好后存磁盘，下次直接读，跳过模型推理。

### 双文件缓存设计

```
~/.xiaomei-brain/memory/
├── index_cache.npz      # 向量数据（二进制）
└── index_meta.json       # 文本数据（JSON）
```

**为什么两个文件？** 因为存储的数据类型不同：

| 文件 | 存什么 | 格式 | 为什么用这个格式 |
|---|---|---|---|
| `index_cache.npz` | 9个384维浮点向量 | NumPy 二进制 | 向量是纯数值矩阵，npz 压缩存储快、加载快 |
| `index_meta.json` | 主题内容、元数据、文件列表 | JSON 文本 | 有字符串和嵌套结构，JSON 更灵活 |

### npz 内部结构

```python
np.savez(
    "index_cache.npz",
    vectors=np.array([[0.023, -0.041, ...], ...], dtype=float32),  # shape: (9, 384)
    chunk_topics=np.array(["travel", "travel", "travel", "cooking", ...]),
    chunk_texts=np.array(["# 旅行偏好...", "## 去过的地方...", ...]),
)
```

一个 `.npz` 文件本质是一个 zip 包，里面装了多个 `.npy` 数组文件。加载时：

```python
data = np.load("index_cache.npz", allow_pickle=True)
vectors = data["vectors"].tolist()       # 9x384 的 list
chunk_topics = data["chunk_topics"].tolist()  # ["travel", "travel", ...]
chunk_texts = data["chunk_texts"].tolist()    # ["# 旅行偏好...", ...]
```

### 启动时的决策流程

```
_ensure_index()
    │
    ├─ index_cache.npz 存在？
    │   ├─ 是 → _load_index_cache()
    │   │        ├─ 加载 npz → 还原向量
    │   │        ├─ 加载 json → 还原内容和元数据
    │   │        ├─ _incremental_update() → 检查是否有新/修改文件
    │   │        └─ 跳过全量 embedding ✓
    │   │
    │   └─ 否 → _build_index_from_topics()
    │            ├─ 扫描 topics/ 目录所有 .md
    │            ├─ chunk_markdown() 分片
    │            ├─ embed_batch() → 模型推理（慢）
    │            └─ _save_index_cache() → 写缓存供下次用
    │
    └─ 后续 search() 调用时不再重复构建
```

### 性能对比

| 操作 | 无缓存 | 有缓存 |
|---|---|---|
| 加载9个chunk | 模型推理 ~2-5秒 | 读磁盘 ~10ms |
| 100个主题 | 可能 30秒+ | 仍 ~10ms |

### 增量更新的配合

缓存还记录了 `_indexed_files`（已索引的文件路径集合）和 `_topic_meta`（含 `last_modified` 时间戳），这样加载缓存后：

```python
# 检查哪些文件是新来的
new_files = current_files - indexed_files

# 检查哪些文件被修改过（磁盘mtime > 记录的last_modified）
if os.path.getmtime(filepath) > meta.last_modified:
    modified_files.add(filepath)
```

只对变化的文件做 embedding，其余直接用缓存。

新: 只返回 ## 喜欢的食物 那个 chunk
    → "日本拉面、寿司\n泰式冬阴功汤\n韩国烤肉"

## 助手回答（2026-04-13 03:54:12）

从当前系统出发，我认为有三个层次的方向：

## 一、必须解决的短板

**1. 上下文窗口管理**

当前 `self.messages` 无限增长，迟早超出模型的 token 上限，API 会报错。

```
现在:  messages = [sys, user1, asst1, user2, asst2, ..., user50, asst50]
                                                    ↑ 越来越长
方案:  滑动窗口 + 摘要压缩
      - 最近 N 轮保留原文
      - 更早的对话压缩成一段摘要插入 system prompt
      - 类似人脑：近期清晰，远期只剩印象
```

**2. 流式输出**

现在 `agent.run()` 是同步阻塞的，用户要等 LLM 完整回复才看到文字。体验差。

```
现在:  print(agent.run("你好"))  → 等3秒 → 一次性输出全部
目标:  for chunk in agent.stream("你好"):
           print(chunk, end="")   → 逐字输出
```

## 二、让记忆系统更聪明

**3. 记忆层次化**

现在只有长期记忆（topics 文件），缺短期记忆：

```
短期记忆 (working memory)
  - 本轮对话关键信息
  - 存在内存，会话结束即消失
  - 例："用户刚才说心情不好"

长期记忆 (topics)
  - 持久化的事实和偏好
  - 通过梦境从对话中提取

场景记忆 (episodic)
  - 特定事件的完整叙事
  - 例："2024年1月15日聊了关于离婚的话题"
  - 现在对话日志有这个潜力，但还没用起来
```

**4. 记忆关联图谱**

现在是扁平的 topic 列表，记忆之间没有关联：

```
现在:  travel ── cooking ── hobby  (互相独立)

目标:  travel ──喜欢──→ 日本料理 ──属于──→ cooking
       travel ──去过──→ 东京 ──有──→ 拉面店
       
       检索"日本"时，不仅匹配 travel，还能沿关联链找到 cooking
```

技术上可以用轻量图数据库（NetworkX 就够），不需要 Neo4j 这么重。

**5. 记忆置信度**

现在保存的记忆不加区分——用户随口说的和认真告诉你的权重一样：

```
"我好像去过日本"    → 置信度低，可能记错
"我是山东人"        → 置信度高，确定性事实
"我不太喜欢吃辣"    → 偏好，可能改变

梦境提取时让 LLM 标注类型和置信度，影响检索权重
```

## 三、走向更远的方向

**6. 多模态感知**

如果作为伴侣型 agent，文字不够：

```
语音 → TTS/STT → 能听能说
图片 → 视觉理解 → "看看我今天做的菜"
```

技术上：智谱有语音 API，图片理解 GLM 也支持，接口层面改动不大。

**7. 主动行为**

现在 agent 纯被动——你不说话它就不动。但一个有"生活"的 agent 应该：

```
- 你3天没来聊天了 → 主动发消息："最近忙吗？"
- 记得你说周五要考试 → 周五问："考得怎么样？"
- 梦境后有了新记忆 → 主动提及："我昨晚想了想，你说你喜欢..."
```

技术上：定时任务 + 消息推送。DreamScheduler 已经有基础设施了。

**8. 多 Agent 协作**

单一 agent 角色固定。可以拆分：

```
小美 (主角色) ── 对话、陪伴
记忆管家 ── 专门管理记忆的读写、去重、整理
梦境分析师 ── 深度分析对话模式、情感变化
```

## 我的建议：优先级排序

```
P0 (不做出问题):  上下文窗口管理
P1 (体验提升):    流式输出
P2 (核心差异化):  记忆层次化 + 场景记忆
P3 (长期价值):    记忆关联图谱
P4 (锦上添花):    主动行为、多模态
```

你想从哪个方向开始？

当前规模不需要换，但到一定量级后部分组件值得换。

### 现有存储拆解

| 数据 | 当前格式 | 规模瓶颈 | 要不要换 |
|---|---|---|---|
| 主题内容 | `.md` 文件 | 万级文件时 `os.listdir()` 变慢 | **不换** |
| 向量索引 | `.npz` 缓存 | 千级 chunk 内纯 Python 够用 | **暂不换** |
| 元数据 | `.json` | 几百条没问题 | **可以换** |
| 对话日志 | `.jsonl` | 每天一个文件，量大了管理麻烦 | **建议换** |

### 为什么主题内容不换数据库

**人类可读性是无价的优势**。你可以在任何编辑器里直接打开、修改记忆：

```bash
$ cat ~/.xiaomei-brain/memory/topics/travel.md
# 旅行偏好
我最喜欢自由行...
```

换成一堆 BLOB 就没这个能力了。而且 Claude Code 自己也是用 markdown 文件做记忆，证明这个方案在 agent 场景下是合理的。

### 什么时候需要数据库

```
主题 < 1000    → 纯文件，完全够
主题 1000~1万  → 元数据和日志换 SQLite，内容仍用文件
主题 > 1万     → 向量搜索换专用向量库，内容考虑对象存储
```

### 建议的混合方案

```
topics/              → markdown（人可读、可编辑）
xiaomei.db (SQLite)  → 元数据 + 对话日志 + 场景记忆
index_cache.npz      → 向量缓存（暂时保留）
```

SQLite 的好处：

| 场景 | JSON 文件 | SQLite |
|---|---|---|
| 查最近7天的对话 | 扫7个文件，逐行解析 | `SELECT * FROM conversations WHERE date > ?` |
| 找最久未访问的主题 | 加载全部 meta.json | `SELECT topic FROM meta ORDER BY last_accessed LIMIT 1` |
| 并发写入对话日志 | 可能写坏文件 | ACID 事务保护 |
| 梦境标记已处理 | 维护 `.processed` 文件 | `UPDATE logs SET processed = 1` |
| 统计记忆使用频率 | 自己遍历计数 | `SELECT topic, access_count FROM meta` |

Python 内置 `sqlite3`，零依赖，单文件部署。

### 要现在做这个切换吗

**我建议先不做**。原因：

1. 当前 5 个主题，SQLite 带来的收益接近零
2. 刚完成一大轮优化，该先跑起来验证整体效果
3. 等对话日志和场景记忆真正积累起来再迁移，需求会更清晰

**优先级排在**：上下文管理 > 流式输出 > SQLite 迁移
