## Drive 系统架构分析

### 一、文件结构

```
src/xiaomei_brain/drive/   (5 个文件，~1300 行)
├── __init__.py          # 公开 API
├── config.py            # 配置结构（纯数据）
├── state.py             # 状态数据结构（纯数据）
├── engine.py            # 核心引擎（~850行，全部逻辑）
└── storage.py           # JSON 持久化
```

分层清晰：config 定义参数 → state 定义结构 → engine 执行逻辑 → storage 落地。

### 二、状态模型（5 个维度）

```
DriveEngine
├── EmotionalState    # 情绪：类型+强度，分钟级衰减
├── HormoneState      # 激素：5种神经递质，小时级衰减
├── MotivationState   # 动机：RPE 奖励预测误差
├── DesireState       # 欲望：5种内在张力，向基线恢复
└── EnergyState       # 能量：由激素加权合成
```

**关键设计**：这 5 个状态全部是算法驱动，LLM 不参与数值计算。LLM 只负责从自然语言中**识别事件类型**，代码负责**应用固定数值映射**。

### 三、数据流入（事件驱动）

事件从 4 个方向进入 Drive：

```
社交事件（L2/InnerVoice LLM 识别）
├── on_praise()        → 多巴胺↑ 催产素↑
├── on_criticism()     → 皮质醇↑ 多巴胺↓
├── on_social_connection() → 催产素↑
├── on_user_active()   → 归属欲↓ 生存欲↑
└── on_user_idle()     → 归属欲↑

目标事件（PurposeEngine）
├── on_goal_completed() → 多巴胺↑ 血清素↑ 成就欲↓
├── on_goal_failed()    → 皮质醇↑ 成就欲↑（挫败强化欲望）
└── on_goal_progress()  → 多巴胺↑（RPE计算）

系统事件（Interoception）
├── on_system_stress()  → 皮质醇↑ 去甲↑
└── on_system_healthy() → 血清素↑

行为反馈（ActionDispatcher）
└── on_desire_satisfied() → 对应欲望↓ 对应激素↑
```

**设计原则**：每个事件同时影响多个维度。比如 `on_praise` 同时动情绪（JOY）、激素（多巴胺+催产素+血清素）、动机（motivation_level）。这模拟了真实边缘系统的一次事件多系统联动。

### 四、时间衰减（自主演化）

```
tick() 每秒调用（来自 Consciousness L0）
├── 被动能量恢复：每秒恢复 0.001 * 激素因子
├── tick_minute()：情绪 * 0.95/分钟，低于阈值归 NEUTRAL
└── tick_hour()：
    ├── 激素乘性衰减（多巴胺0.95 皮质醇0.90 血清素0.98...）
    ├── 欲望向基线恢复（recovery_rate=0.5）
    └── 生存欲侵蚀（用户空闲>12h）
```

**关键**：衰减只作用于情绪和激素。欲望是**向基线恢复**（不是衰减到0），动机**不衰减**（gap #6）。

### 五、数据流出（3 条路径）

```
DriveEngine.state
    │
    ├──→ SelfImage.body (实时代理)
    │       SelfBody 的 13 个 @property 直接读 Drive 状态
    │       每次访问都是实时值，不经缓存
    │       → system prompt 渲染
```

## 情绪子系统架构分析

### 一、数据模型

```python
EmotionalState:
  type: EmotionType      # 7选1：JOY/SADNESS/ANGER/FEAR/SURPRISE/DISGUST/NEUTRAL
  intensity: float       # 0.0-1.0
  created_at: float      # 产生时间戳
  duration: float        # 预期持续秒数（默认60s）
```

**核心特征：单情绪模型**。任何时候只能有一种情绪，新事件完全覆盖旧情绪。

### 二、情绪产生的路径（仅 2 条可达）

```
事件源                    情绪       触发条件
─────────────────────────────────────────────
on_praise()          →   JOY        用户表扬
on_goal_completed()  →   JOY        RPE > 0（比预期好）
on_criticism()       →   SADNESS    用户批评
on_goal_failed()     →   SADNESS    目标失败
apply_social_signal()→   任意        ❌ import 断了，不可达
```

**5 种情绪不可达**：ANGER、FEAR、SURPRISE、DISGUST 在枚举中声明了，但没有任何代码路径能设置它们。`apply_social_signal()` 设计上可以通过 `SOCIAL_SIGNAL_MAP` 设置任意情绪，但 `social_perception.py` 不存在。

### 三、情绪衰减（`tick_minute`）

```
用户活跃中（2分钟内）？
  ├── 是 → 跳过衰减（对话中情绪由事件驱动）
  └── 否 → elapsed > duration？
            ├── 否 → 保持不变（完整保留期内）
            └── 是 → intensity *= 0.95/分钟
                      intensity < 0.1？
                        ├── 是 → 归 NEUTRAL
                        └── 否 → 继续衰减
```

**衰减曲线**（以 intensity=1.0 为例）：
- 0-60s：完整保留
- 60s 后：每分钟乘 0.95
- 约 46 分钟后归 NEUTRAL

### 四、情绪数据的流向

```
DriveEngine.emotion
    │
    ├──→ SelfBody.mood          → _render_body()  → system prompt（"心情：joy"）
    ├──→ SelfBody.emotion_intensity → 审计标记为"遗漏"，未渲染
    ├──→ get_signals().emotion  → L2 intent prompt / L3 dream prompt
    ├──→ /drive CLI 命令显示
    └──→ exp_stream.log()       → 经验流（JOY/SADNESS 事件）
         longterm_memory.store() → "我的情绪渐渐平复..."（归 NEUTRAL 时）
```

### 五、架构问题

**1. 单情绪模型 ≠ 真实情绪**

真实情绪是混合的——可以"愤怒中带着悲伤"、"惊喜中带着不安"。当前模型每次设置情绪都是 `self.emotion = EmotionalState(...)` 完全替换，无法表达复合情绪。

**2. 5/7 情绪不可达**

这是之前 gap #7。枚举定义了 7 种情绪，实际只用了 2 种。ANGER 的缺失尤其明显——用户攻击、目标被破坏、系统故障时都没有愤怒路径。

**3. 情绪没有内部触发源**

情绪只能由外部事件触发（用户行为、目标状态变化）。没有"想到某事感到难过"的路径。记忆系统、内省系统都无法触发情绪。

**4. 情绪和激素是并行设置，没有因果链**

```
当前：事件 → on_praise() ─┬→ emotion = JOY
                          └→ dopamine += 0.06

生物：事件 → 情绪产生 → 情绪触发激素释放
      事件 → 激素变化（不经过情绪）
```

两条路径并存是对的（有些激素变化不经过情绪），但缺少"情绪 → 激素"这条因果链。

**5. mood 和 emotion 概念混淆**

`SelfBody.mood` 读的是 `emotion.type.value`，但心理学上 mood（心境）和 emotion（情绪）是不同的：
- emotion：秒-分钟级，有明确触发物
- mood：小时-天级，无明确触发物，是背景色调

当前把"当前情绪类型"直接当作"心情"渲染给 LLM，概念上不准确。

**6. duration 一刀切**

所有情绪默认 60s 完整保留期（`on_goal_failed` 除外用 120s）。真实情绪的时间特征差异巨大——surprise 持续数秒，grief 持续数天。不同情绪应该有不同的 duration 配置。

**7. 情绪强度增量叠加但不考虑情境**

`on_praise` 如果当前已是 JOY，会在原强度上叠加。但如果是 SADNESS 中被表扬呢？当前逻辑：`current_intensity = 0 if type != JOY else intensity`，相当于完全覆盖——SADNESS 被打断，突然变成 JOY。这不自然，真实情况是悲伤中收到表扬会产生矛盾情绪（悲喜交加），而非瞬间切换。

---

总结：情绪子系统数据模型过于简化（单值覆盖），事件映射过于稀疏（2/7 可达），且缺少内部触发源。这些问题在上面 8 个 gap 中只记录了一个（gap #7 ANGER/FEAR/SURPRISE/DISGUST 不可达），但实际上是 7 个结构性问题。

```

归属欲 -0.05 * intensity
```

- 情绪低落 → 我应该是 SADNESS + 归属欲↑（想安慰）
- 愤怒 → 我应该是 FEAR 或 SADNESS + 皮质醇↑
- 冷淡 → 我应该是 SADNESS + 归属欲↑（渴望连接）


```python
def _modulate_emotion_switch(self, target: EmotionType, intensity: float) -> float:
    """情绪切换惯性：切换到不同类型时，旧情绪强度产生阻力。"""
    current = self.emotion
    if current.type == EmotionType.NEUTRAL or current.type == target:
        return intensity  # 无阻力

    # 冲突权重：对立情绪阻力更大
    opposites = {
        (EmotionType.JOY, EmotionType.SADNESS): 0.8,
        (EmotionType.SADNESS, EmotionType.JOY): 0.8,
        (EmotionType.FEAR, EmotionType.JOY): 0.6,
        (EmotionType.ANGER, EmotionType.JOY): 0.5,

    conflict = opposites.get((current.type, target), 0.3)
    return intensity * (1 - current.intensity * conflict)
```

```python
def _modulate_emotion_switch(self, target: EmotionType, intensity: float) -> float:
    """情绪切换惯性：旧情绪 + 激素背景 + 性格 三层叠加。

    Returns:
        (actual_emotion_type, adjusted_intensity)
        可能保持原情绪降低强度，也可能切换到目标情绪。
    """
    current = self.emotion

    # 无阻力：平静态或同向叠加
    if current.type == EmotionType.NEUTRAL:
        return target, intensity
    if current.type == target:
        return target, min(1.0, current.intensity + intensity)

    # ── 层1：基础冲突权重 ──
    conflict = {
        (EmotionType.JOY, EmotionType.SADNESS): 0.8,
        (EmotionType.SADNESS, EmotionType.JOY): 0.8,
        (EmotionType.FEAR, EmotionType.JOY): 0.6,
        (EmotionType.ANGER, EmotionType.JOY): 0.5,
        (EmotionType.JOY, EmotionType.FEAR): 0.5,
        (EmotionType.SADNESS, EmotionType.FEAR): 0.3,
        (EmotionType.FEAR, EmotionType.ANGER): 0.4,
        (EmotionType.ANGER, EmotionType.FEAR): 0.4,
        (EmotionType.SADNESS, EmotionType.ANGER): 0.6,
        (EmotionType.ANGER, EmotionType.SADNESS): 0.5,
    }.get((current.type, target), 0.3)

    # ── 层2：性格惯性（配置驱动）──
    # 0.1 = 好哄，0.9 = 极其固执
    inertia = getattr(self.config.emotion, 'switch_inertia', 0.7)

    # ── 层3：激素调制（情境）──
    # 多巴胺高 → 更容易转向 JOY（阻力降低）
    if target == EmotionType.JOY and self.hormone.dopamine > 0.5:
        inertia *= 0.5 + (1.0 - self.hormone.dopamine)  # dopa=0.8 → inertia*=0.7
    # 皮质醇高 → 更难转向 JOY（阻力增大）
    if target == EmotionType.JOY and self.hormone.cortisol > 0.5:
        inertia *= 1.0 + self.hormone.cortisol * 0.5    # cort=0.8 → inertia*=1.4
    # 皮质醇高 → 更容易转向 FEAR
    if target == EmotionType.FEAR and self.hormone.cortisol > 0.5:
        inertia *= 0.5 + (1.0 - self.hormone.cortisol)  # cort=0.8 → inertia*=0.7

    inertia = max(0.05, min(1.0, inertia))

    # ── 综合阻力 ──
    effective_conflict = conflict * inertia
    candidate_strength = intensity * (1.0 - current.intensity * effective_conflict)

    # 候选不够强 → 不切换，原情绪弱化
    if candidate_strength < current.intensity * 0.5:
        weakened = current.intensity * (1.0 - candidate_strength * 0.3)
        return current.type, weakened

    return target, candidate_strength
```

## RPE：预测误差

```
预期奖励 0.5 → 实际 0.9 → 多巴胺 +0.4 → "比预想好"
预期奖励 0.5 → 实际 0.2 → 多巴胺 -0.3 → "比预想差"
```

RPE 做的是**校准预期**。让 agent 下次猜得更准。

## Quality Standard：行为基准

```
上次做代码审查，漏了性能问题，用户指出了
→ 下次做代码审查，"检查性能"成为自动唤起的一个检查项
→ 不只是预期值变了，是行为本身变了
```

Quality standard 做的是**改变执行方式**。

## 区别

| | RPE | Quality Standard |
|---|-----|-----------------|
| 更新什么 | 一个数字（expected_reward） | 一个行为规则（"做 X 时要检查 Y"） |
| 维度 | 单值 | 结构化（场景 → 标准列表） |
| 作用 | 让你对结果不意外 | 让你下次做更好 |
| 泛化 | 不跨任务，只看 reward | 必须跨任务，"同类场景"匹配 |
| 本质 | 统计学习 | 经验积累→行为改善 |

一个 agent RPE 可以很准——它精确地知道"这次我大概会拿到 0.3 的奖励"。但它不改变行为。它只是**接受自己烂**。

人类不是这样。人类预期低的时候会**想办法提高质量**，而不是说"好的那我就这么差吧"。

## 所以 quality standard 是什么

是 agent 积累的一个映射表：

```
场景：帮用户调试代码
  标准：先复现 → 定位 → 验证 → 解释原因 → 检查类似问题
  来源：上次用户说"你只修了这一个没检查其他地方"
  
场景：回答技术问题  
  标准：确认理解问题 → 给出方案 → 说明原理 → 标注不确定的部分
  来源：上次幻觉了一个 API 名字被指出来
```

不是数值。是**下次做同类事情时的行为约束**。每次执行后，用 L2 总结一条教训，存在 SelfModel 里。下次同类任务唤起。

**RPE 让你知道做得好不好。Quality standard 让你下次做得更好。** 两者需要一起工作。

## Quality Standard 不是什么

不是 RPE 扩展——RPE 校准预期值，quality standard 改变行为。
不是 memory——memory 是"发生过什么"，quality standard 是"下次该怎么"。
不是 talent.md——talent.md 是"我是谁"的静态描述，quality standard 是动态积累的行为约束。

## 数据模型

```
QualityStandard:
    id: str                          # 唯一标识
    context_pattern: str             # 适用场景（如 "debugging_code", "answering_questions"）
    rule: str                        # 行为规则（"先复现再动手", "不确定时标注"）
    rationale: str                   # 为什么（"上次用户指出我只修了表面"）
    strength: float (0-1)            # 当前强度（强化→升，遗忘→降）
    reinforcement_count: int         # 被验证的次数
    last_reinforced: float           # 上次强化时间戳
    source: enum                     # user_correction | self_reflection | explicit_set
    created_at: float
```

**不是存 prompt 模板，是存约束。** 每条只是一个规则，一个场景标签，一条理由。

## 生命周期

```
捕获 → 存储 → 检索 → 应用 → 验证 → 强化/退化
```

### 1. 捕获（标准怎么来）

三个入口：

**用户纠正（最直接）**
```
对话中用户说："你没测试就敢说修好了？"
→ InnerVoice 或 SocialPerception 标记这次交互质量有问题
→ L2 下次 tick 时检查最近的 negative signals
→ 提取："用户不满是因为我没有验证就给出结论"
→ 生成标准：{context: "给出修复方案后", rule: "提醒需要验证", rationale: "用户指出我跳过验证"}
```

**自我反思（周期性）**
```
L2 定期检查最近几次任务执行
→ 发现同一类问题出现了两次以上
→ 生成标准："这类问题我容易犯错，以后要注意X"
```

**用户显式设定**
```
用户说："以后回答技术问题时，不确定的地方标注出来"
→ 直接存为标准
```

### 2. 存储（放哪）

放在 SelfModel 里，但不放在 system prompt 的主区域。作为一个独立 section：

```
SelfModel.quality_standards: List[QualityStandard]
```

和 talent.md 区分：
- talent.md → "我是谁"（静态，用户编辑）
- quality_standards → "我应该怎么做"（动态，经验积累）

持久化：跟 SelfImage 一起存到 `latest.json`。

### 3. 检索（什么场景用什么标准）

ContextAssembler 在构建上下文时做匹配：

```
当前任务特征 → 匹配 quality_standards.context_pattern → 取 strength > 阈值 的标准
```

匹配不需要精确。可以用 LLM 做（轻量），也可以靠简单的关键词/embedding 召回。

### 4. 应用（怎么注入）

不是塞一堆规则进去让 LLM 读。是**按场景注入，压缩成一行**：

```
system prompt 里加一行：
> 上次类似任务你学到：[先复现后修复] [不确定时标注] [检查关联代码]
```

不是报告，是指令。像人类做同类事之前脑子里自动跳出来的提醒。

### 5. 验证 + 强化/退化

任务完成后：

```
if 用户表示满意 or L2 自评"这次做得不错":
    相关标准的 strength += 0.1, reinforcement_count += 1
elif 用户指出同样的问题又犯了:
    标准 strength -= 0.2  # 退得更快，因为你明明知道还没做到
```

长时间不用的标准自动退化：
```
tick_hour: 所有标准 strength *= 0.999  # 极慢遗忘
```

## 为什么不是另一个 memory

记忆说"发生过什么"。标准说"下次该怎样"。语义不同。

标准**预消化了这个推理**。存的时候花了 LLM（L2）把反馈提炼成规则。用的时候直接注入，零推理开销。

## 核心问题

survival 当前几乎完全是**被动感知器**——只看用户来没来。用户不来 → 降，用户来 → 升。这条线是对的，但太窄了。

一个 agent 可能：

- 从不主动思考
- 目标搁置不推进

1. **survival 应该感知什么？** 只是用户活跃频率，还是也应该感知行为质量？做得好不好、主动不主动——这些是不是"存在的代价"的一部分？

3. **survival 和其他 desire 的关系？** 如果 survival 感受到了威胁，should it amplify other desires (belonging 更强 → 更主动连接) 还是 override them（只有 survival 重要，其他欲望搁置）？

| 选项 | 效果 |
|------|------|

**A 最真，但太实验性**（进程真关了）。**B 更务实**——让 agent 体验 DORMANT 然后被"救回来"。
