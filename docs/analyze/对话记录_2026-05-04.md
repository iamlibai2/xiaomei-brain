_heartbeat() 和tick() 是什么意思


**[01:06:16] 🤖助手**

## heartbeat = 生命体征检测，tick = 心跳节拍

### 类比

人每秒心脏跳一次（tick），每次跳动时医生检查一下你的生命体征（heartbeat）。

### 代码里的对应关系

**tick()** — 意识系统内部的节拍器，每秒被调用一次，维护火焰骨架状态

```
consciousness.tick()  [意识系统内部]
  ├─ L0: 每秒 → 更新感知、记录变化、Drive衰减
  ├─ L1: 每60秒 → 异常检测、语义解读
  ├─ L2: 动态 → LLM轻度加柴
  └─ L3: 梦境 → LLM深度燃烧
```

**_heartbeat()** — 生命循环的调度器，每个主循环tick调用一次，决定调不调 consciousness.tick()

```
ConsciousLiving._heartbeat()  [生命循环调用]
  ├─ 调用 consciousness.tick()
  ├─ 每60秒检查 ActionDispatcher
  └─ 返回 HEARTBEAT_NORMAL 或 HEARTBEAT_DREAM
```

### 调用链

```
Living._loop_awake()          ← 主循环，每1秒
  │
  ├─ _heartbeat(AWAKE)        ← ConsciousLiving 覆写
  │    │
  │    ├─ consciousness.tick(agent_state="awake")
  │    │    │
  │    │    ├─ tick_L0()      ← 每秒：感知+状态更新
  │    │    ├─ tick_L1()      ← 每60秒：异常检测
  │    │    ├─ tick_L2()      ← 动态：LLM加柴
  │    │    └─ tick_L3()      ← 梦境：LLM深度燃烧
  │    │
  │    ├─ ActionDispatcher    ← 每60秒：检查主动行为
  │    │
  │    └─ return HEARTBEAT_NORMAL
  │
  └─ _wait_message(1秒)       ← 等待用户消息
```

### 为什么分两层

**tick()** 只关心意识系统内部的节律（"我现在是什么状态"），不知道自己在什么生命状态（AWAKE/SLEEPING/DREAMING）。

**_heartbeat()** 是生命循环和意识系统的桥梁，告诉 tick() 当前是什么生命状态，并根据 tick() 的返回决定是否切换状态（比如 L3 触发 → 切到 DREAMING）。

简单说：**_heartbeat 是壳，tick 是核。壳负责调度，核负责计算。**

不完全合理。有三个问题：

## 问题 1：_heartbeat 职责混乱

当前 `_heartbeat()` 同时做三件不同的事：

```python
def _heartbeat(self, state, **kwargs):
    # 1. 意识节律（该做的事）
    consciousness.tick(agent_state=state)
    
    # 2. 行为调度（不该在这里）
    if now - self._last_action_dispatch >= 60:
        self._dispatcher.tick(si)
        self._dispatcher.process_queue()
    
    # 3. 死代码（L2同步/L3判断，已移到tick()内部但没删）
```

**行为调度不属于心跳**。心跳应该只管"活着没、该不该跳"，不管"该干什么活"。

## 问题 2：_heartbeat 和 tick 的边界不清

L2 触发后同步 Drive 数据到 SelfImage，这个逻辑在 `_heartbeat()` 的死代码里（456-465行），但实际上应该在 `tick()` 内部完成：

```python
# 现在（死代码，不可达）：
_heartbeat() {
    tick()  → 返回 L2_TRIGGERED
    _sync_self_image_from_subsystems()  ← 应该在tick()内部做
}

# 应该：
tick() {
    tick_L2()
    _sync_self_image_from_subsystems()  ← 在产生数据的地方消费
    return L2_TRIGGERED
}
```

**谁产出的数据，谁负责消费**。L2 更新了 Drive 状态，就应该在 tick() 内同步到 SelfImage，而不是由外层 heartbeat 来做。

## 问题 3：命名误导

- `tick()` 暗示"每秒一次的轻量操作"，但内部可能触发 LLM（L2/L3）
- `_heartbeat()` 暗示"简单的存活检查"，但实际上是主调度器

更准确的命名应该是：

| 现在的名字 | 实际做的事 | 更好的名字 |
|-----------|-----------|-----------|
| `_heartbeat()` | 主循环调度（意识+行为） | `_on_tick()` 或 `_cycle()` |
| `tick()` | 意识节律（可能调LLM） | `consciousness_cycle()` |

## 更合理的设计

```
_loop_awake() [生命循环，每1秒]
  │
  ├─ _cycle(state)  ← 改名，明确是"每轮循环"
  │    ├─ consciousness.cycle()  ← 改名，明确可能调LLM
  │    │    ├─ L0: 骨架维护（纯规则）
  │    │    ├─ L1: 异常检测（纯规则）
  │    │    ├─ L2: LLM加柴 → 自动同步到SelfImage
  │    │    └─ L3: LLM深度燃烧 → 自动同步到SelfImage
  │    │    └─ return TickResult
  │    │
  │    └─ return TickResult
  │
  ├─ _check_actions()  ← 独立出来，每60秒
  │    └─ ActionDispatcher.tick() + process_queue()
  │
  └─ _wait_message(1秒)
```

核心原则：**每个方法只做一件事**。`_cycle()` 管意识节律，`_check_actions()` 管行为调度，不混在一起。

## 二、主循环职责混乱

`_loop_awake()` 每秒做的事情：

```python
def _loop_awake(self):
    self._heartbeat(self.state)                    # ① 意识tick
    msg = self._wait_message(timeout=1)             # ② 等消息
    if msg: _handle_message(msg)                    # ③ 处理消息
    # 检查空闲 → 切状态                              # ④ 状态转换
```

`_heartbeat()` 内部做的事情：

```python
def _heartbeat(self, state, **kwargs):
    consciousness.tick()                            # ① 意识节律
    if now - last >= 60:                           # ② 行为调度（60秒）
        _update_recent_conversations()             # ③ 更新对话
        _dispatcher.tick(si)                       # ④ 规则匹配
        _dispatcher.process_queue()                # ⑤ 执行动作
    return HEARTBEAT_NORMAL
```

**5件不同的事塞在一个方法里**：意识节律、行为调度、对话更新、规则匹配、动作执行。

## 三、tick() 内部节奏不统一

`consciousness.tick()` 被称为"统一入口"，但内部4层节奏完全不同：

| 层 | 频率 | 是否调LLM | 耗时 |
|----|------|----------|------|
| L0 | 每秒 | 否 | <1ms |
| L1 | 每60秒 | 否 | <10ms |
| L2 | 动态 | **是** | 2-10秒 |
| L3 | 梦境 | **是** | 5-30秒 |

**问题：** 一个名叫 `tick()` 的方法，调用耗时从 <1ms 到 30秒 不等。调用方无法预知这次 tick 会阻塞多久。

## 两个正交维度

| | 定义 | 类比 |
|---|---|---|
| **生命状态 DREAMING** | 身体在离线处理，不响应用户 | 人的睡眠状态 |
| **意识 L3 深度燃烧** | 火焰烧到最深，LLM 完整意识报告 | 人的深度REM睡眠 |

**现状：它们被耦合在一起**——L3 必须进 DREAMING 才能触发，DREAMING 里也必然触发 L3。

**问题：** 如果 L3 条件在 AWAKE/SLEEPING 满足了呢？现在的代码不会触发 L3，因为判断条件是 `agent_state == "dreaming"`。

## 我的建议：解耦，但保留 L3 在 DREAMING 触发

```
生命状态（DORMANT/WAKING/AWAKE/IDLE/SLEEPING/DREAMING）
    │
    └─→ DREAMING 时：ConsciousLiving._loop_dreaming()
              │
              └─→ DreamEngine.dream()
                        │
                        ├─→ emotion_processor    ← 情绪/欲望整理
                        ├─→ memory_organizer   ← 记忆强化+提取
                        ├─→ flame_burner        ← L3 深度燃烧
                        └─→ reflection           ← 反省（预留）

意识深度（L0/L1/L2/L3）
    │
    └─→ L3 触发条件：accumulated_changes 超过阈值
              │
              └─→ 触发时写入 intent_buffer，不切换状态
```

**核心改动：**

1. `core.py` 里 L3 的触发条件**去掉** `agent_state == "dreaming"` 限制，改为纯条件触发
2. L3 的产物（summary）写入 `intent_buffer`，让 DreamEngine 消费
3. DreamEngine 的 `_flame_burn()` 不再自己调 LLM，而是从 `intent_buffer` 读取 L3 产物

这样 L3 成了可中断的后台任务，而不是 DREAMING 的专属行为。
