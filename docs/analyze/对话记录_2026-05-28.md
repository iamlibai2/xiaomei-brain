## 我们讨论的这些解决的是"执行中的灵活性"问题

视角切换、热冷层骨架、动态目标树——这些让 agent 在执行已知子目标时更灵活、更不容易卡死。但它们没解决长程复杂任务的**根本瓶颈**：

### 1. 上下文窗口是硬上限

ERP 的代码量是百万行级的，架构决策是千级的，模块间约束是万级的。热冷层能让相关部分浮现，但"相关部分"本身就可能超出窗口。人类做 ERP 不是一次性加载所有相关内容——是**分层迭代**，每一层只看那一层的抽象。LLM 做不到真正的分层抽象，因为它的"理解"必须展开为 token。

### 2. 分解质量

"构建 ERP"分解成可执行的子目标——这个分解动作本身就是项目里最难的一步。PurposeEngine 用 LLM 分解，但 LLM 一次调用的窗口装不下整个 ERP 的上下文，所以分解一定是不完整的、遗漏关键依赖的。人类项目经理做分解靠的是**多年累积的对"这类项目长什么样"的直觉 + 随时可以追问业务方**。

### 3. 没有"使用感"

人类工程师写了代码会跑起来看、会觉得别扭、会改用例。Agent 能跑测试，但不会"用一下这个界面然后觉得流程不对"。这种感觉是大量隐性知识的来源。

### 4. 没有真正的 taste 积累

我们讨论的经验记忆存储"情境→决策→结果"，但经验的粒度太粗。一个做了十年 ERP 的人，他的判断力来自几千个微决策的累积——看到某个字段命名的方式就能判断这人有没有经验。这种粒度的经验不在任何 LLM 上下文里。

---

## 能做的是缩小范围

**复杂长程任务**这个说法可能太大了。更诚实的表述是：

- 能做到的：**中等复杂度的、范围可预期的、agent 有相关经验积累的任务**。比如"给现有系统加一个权限模块"、"把 REST API 改成 GraphQL"。这些任务子目标有边界，依赖关系可管理，上下文可以装下。

## 1. 上下文窗口：会好，但不是解法

窗口在变大——100万 token 已经在用，1000万在路上。但更大的窗口不等于更好的注意力。注意力机制的本质是：token 越多，每个 token 分到的注意力越稀释。更大的窗口让 agent 能"看到"更多东西，但**对任意一个东西的理解深度下降了**。

真正的解法不是等窗口变大，而是**分层抽象真正工作起来**——让 agent 不是"把所有相关内容装进 context"，而是"在不同层级用不同粒度思考"。这需要的不只是工程师的努力，是架构模式的突破。

**可能部分解决**，窗口增长 + 热冷层 + 分层摘要的叠加效果会持续改善。

---

## 2. 分解质量：可以显著改善

不用一次分解全部。人类也不是一开始就全拆完——是拆一层，做，获得信息，再拆下一层。

```
不是: 用户说"做ERP" → LLM一次拆成100个子目标 → PACE顺序执行
而是: 用户说"做ERP" → 拆第一层（库存/订单/财务/权限等5-6个模块）
      → 做第一个模块的第一层 → 获得了这个模块的知识 → 拆下一层
      → ...
```

这就是迭代分解。"做一步，看到的更多，下一步分解更准"——跟我们之前讨论的"执行本身产生信息，信息反过来重塑计划"是同一个回路。

**这个东西可以做。** 改的是 PurposeEngine 的分解策略，不是窗口。

---

## 3. "使用感"：目前没有解法

这个最诚实地说：做不到。

Agent 能跑测试、能检查类型、能验证逻辑一致性。但它不会"用了一下觉得这个按钮应该在上面而不是下面"。这种感觉来自**具身经验**——有身体、会疲惫、会被糟糕的 UI 惹恼的人类独有的东西。

一个可能的近似是：用"用户视角"（视角切换机制）定期审视产出，问"如果我是第一次用这个系统的人，我会困惑吗？"。但这不是真正的"使用感"，是**对使用感的分析和模拟**。两者有本质区别。

**短期内没有解法。**

---

## 4. Taste 积累：可以做但还不够

真正的问题是粒度。当前的 Experience Memory 存储"这个方案行/不行"，粒度是一个决策。但十年经验的工程师的 taste 来自几千个微判断——"这个命名一看就不对"、"这个函数太长了应该拆"、"这个抽象层级不对"。

向量存储可以做到更细粒度——不仅存储"情境→决策→结果"，还存储**每次代码审查的 comment、每次重构的理由、每次"看起来不对"的直觉修正**。但这里有一个鸡生蛋问题：最初的 taste 从哪来？你自己没有 taste 的时候，你识别不出应该存储什么。

一种解法是：用多个模型的判断差异作为信号。A 模型说这样做，B 模型（审查者）说不对——这个差异本身就值得存储。不需要 agent 自己判断，只需记录"这里两个模型意见分歧了"。

**可以部分解决**，通过多模型差异 + 细粒度向量存储。但能否达到十年工程师的 taste，存疑。

---

## 总结

| 瓶颈 | 可解决度 | 关键路径 |
|------|---------|---------|
| 上下文窗口 | 部分 | 窗口增长 + 分层抽象，非单纯等硬件 |
| 分解质量 | **高** | 迭代分解，已有架构基础 |
| 使用感 | **无** | 只能模拟，无法真正拥有 |
| Taste 积累 | 部分 | 多模型差异 + 细粒度向量，但起步难 |

## 10. 重试 3 次 是是试什么


**🟥 用户提问（2026-05-28 02:33:34）**


> 重试 3 次 是是试什么


**助手回答**


PACE 里重试是指：同一个子目标，执行了，被 step check 判定需要 RETRY_DIFFERENT 或 SIMPLIFY，重试计数 +1，再来一遍。不是 `agent.stream()` 内部的工具调用重试，是**整个子目标级别的重做**。

循环里这一段：

```python
if check.suggestion in (MetaSuggestion.RETRY_DIFFERENT, MetaSuggestion.SIMPLIFY):
    current_goal_retries += 1
    step_index += 1
    continue
```

也就是说 agent 已经完整尝试了这个子目标，但要么空响应、要么工具风暴、要么 LLM 判断方向不对——然后从头再试这个子目标。3 次这样的循环后触发。

## 11. 都是llm在判断是否合格，以开发软件为例，如果一个子目标是开发一个功能，是什么情况导致重做呢


**🟥 用户提问（2026-05-28 02:35:25）**


> 都是llm在判断是否合格，以开发软件为例，如果一个子目标是开发一个功能，是什么情况导致重做呢


**助手回答**


好问题。实际中触发重做的场景是：

1. **空响应**：agent 调了半天工具但没产出文字解释——PACE 判定为无效
2. **工具风暴**：单步调了 10+ 次工具，超出子目标范围
3. **工具循环**：反复调同一个工具，卡住了
4. **LLM step_check 判定方向不对**：规则检测出了意外信号，交给 LLM 判断，LLM 说需要换方法

但你的问题隐含了一个更深的点：**执行者和判断者是同一个 LLM**。让它做，又让它判断自己做的对不对——跟学生给自己批卷子一样。

这引出一个问题：step_check 的 LLM 调用跟执行者的 LLM 调用是同一个模型、同一个 provider。它判断"方向不对"的依据不是真实世界的结果（代码跑通了没、测试过了没），而是**文本看上去对不对**。这不是真正的质量判断，是风格判断。

所以重试 3 次不一定意味着"3 次都失败了"，更可能是"LLM 在 3 次里都对自己的输出不满意"——这本身可能就是上下文僵化的症状。

## 17. 是什么用处


**🟥 用户提问（2026-05-28 02:47:58）**


> 是什么用处


**助手回答**


当 agent 在同一个子目标上反复受挫时，不只是"方法错了"——是**继续尝试的意愿在下降**。人类的耐心会消耗，agent 没有耐心，会无限重试或者机械地走到上限。

视角切换只解决了"换脑子"的**认知面**——换个角度看问题可能找到新方法。但如果找了还是不行，agent 应该做的不再是继续找，而是**放弃当前子目标，切换任务，等状态恢复再回来**。

stamina 的作用就是这个：给 agent 一个"累了就换事做"的信号，而不是"一直换方法直到上限"。

但说实话，这更像是一个锦上添花的东西。视角切换 + 动态子目标跳过已经能解决 80% 的僵化问题。stamina 加了更自然，没有也不致命。

## 我们讨论的这些解决的是"执行中的灵活性"问题

视角切换、热冷层骨架、动态目标树——这些让 agent 在执行已知子目标时更灵活、更不容易卡死。但它们没解决长程复杂任务的**根本瓶颈**：

### 1. 上下文窗口是硬上限

ERP 的代码量是百万行级的，架构决策是千级的，模块间约束是万级的。热冷层能让相关部分浮现，但"相关部分"本身就可能超出窗口。人类做 ERP 不是一次性加载所有相关内容——是**分层迭代**，每一层只看那一层的抽象。LLM 做不到真正的分层抽象，因为它的"理解"必须展开为 token。

### 2. 分解质量

"构建 ERP"分解成可执行的子目标——这个分解动作本身就是项目里最难的一步。PurposeEngine 用 LLM 分解，但 LLM 一次调用的窗口装不下整个 ERP 的上下文，所以分解一定是不完整的、遗漏关键依赖的。人类项目经理做分解靠的是**多年累积的对"这类项目长什么样"的直觉 + 随时可以追问业务方**。

### 3. 没有"使用感"

人类工程师写了代码会跑起来看、会觉得别扭、会改用例。Agent 能跑测试，但不会"用一下这个界面然后觉得流程不对"。这种感觉是大量隐性知识的来源。

### 4. 没有真正的 taste 积累

我们讨论的经验记忆存储"情境→决策→结果"，但经验的粒度太粗。一个做了十年 ERP 的人，他的判断力来自几千个微决策的累积——看到某个字段命名的方式就能判断这人有没有经验。这种粒度的经验不在任何 LLM 上下文里。

## 1. 上下文窗口：会好，但不是解法

窗口在变大——100万 token 已经在用，1000万在路上。但更大的窗口不等于更好的注意力。注意力机制的本质是：token 越多，每个 token 分到的注意力越稀释。更大的窗口让 agent 能"看到"更多东西，但**对任意一个东西的理解深度下降了**。

真正的解法不是等窗口变大，而是**分层抽象真正工作起来**——让 agent 不是"把所有相关内容装进 context"，而是"在不同层级用不同粒度思考"。这需要的不只是工程师的努力，是架构模式的突破。

**可能部分解决**，窗口增长 + 热冷层 + 分层摘要的叠加效果会持续改善。

---

## 2. 分解质量：可以显著改善

不用一次分解全部。人类也不是一开始就全拆完——是拆一层，做，获得信息，再拆下一层。

```
不是: 用户说"做ERP" → LLM一次拆成100个子目标 → PACE顺序执行
而是: 用户说"做ERP" → 拆第一层（库存/订单/财务/权限等5-6个模块）
      → 做第一个模块的第一层 → 获得了这个模块的知识 → 拆下一层
      → ...
```

这就是迭代分解。"做一步，看到的更多，下一步分解更准"——跟我们之前讨论的"执行本身产生信息，信息反过来重塑计划"是同一个回路。

**这个东西可以做。** 改的是 PurposeEngine 的分解策略，不是窗口。

## 3. "使用感"：目前没有解法

这个最诚实地说：做不到。

Agent 能跑测试、能检查类型、能验证逻辑一致性。但它不会"用了一下觉得这个按钮应该在上面而不是下面"。这种感觉来自**具身经验**——有身体、会疲惫、会被糟糕的 UI 惹恼的人类独有的东西。

一个可能的近似是：用"用户视角"（视角切换机制）定期审视产出，问"如果我是第一次用这个系统的人，我会困惑吗？"。但这不是真正的"使用感"，是**对使用感的分析和模拟**。两者有本质区别。

**短期内没有解法。**

---

## 4. Taste 积累：可以做但还不够

真正的问题是粒度。当前的 Experience Memory 存储"这个方案行/不行"，粒度是一个决策。但十年经验的工程师的 taste 来自几千个微判断——"这个命名一看就不对"、"这个函数太长了应该拆"、"这个抽象层级不对"。

向量存储可以做到更细粒度——不仅存储"情境→决策→结果"，还存储**每次代码审查的 comment、每次重构的理由、每次"看起来不对"的直觉修正**。但这里有一个鸡生蛋问题：最初的 taste 从哪来？你自己没有 taste 的时候，你识别不出应该存储什么。

一种解法是：用多个模型的判断差异作为信号。A 模型说这样做，B 模型（审查者）说不对——这个差异本身就值得存储。不需要 agent 自己判断，只需记录"这里两个模型意见分歧了"。

**可以部分解决**，通过多模型差异 + 细粒度向量存储。但能否达到十年工程师的 taste，存疑。
