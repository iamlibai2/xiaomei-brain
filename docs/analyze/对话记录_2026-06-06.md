## 它要解决的问题

## 它做什么

> 比较 PROGRESS 摘要和（当前目标 + 所有兄弟目标）的 bigram 重叠度。如果摘要和某个兄弟目标的匹配度显著高于当前目标，判为"串台"。

## 它的问题

但这个逻辑的核心假设是错的：

> **假设：PROGRESS 摘要的内容 = 目标描述中的词汇。**

实际上 LLM 的 PROGRESS 摘要描述的是**产出**，不是**目标本身**。目标描述是"执行网站前端界面设计与开发"，而产出是"已完成4个页面（首页/产品中心/工程案例/联系我们）"。产出自然提到具体的页面名称，这些名称可能出现在其他兄弟目标（后端开发）的示例中。

换句话说，一个正确的 PROGRESS 不一定和当前目标描述长得像。用 bigram 来判断"是不是在说当前子目标"，本质上把"产出报告"和"任务标题"混为一谈了。

## 真正的问题本质

不是 bigram 阈值设得对不对的问题。而是：**LLM 不应该被一个自己看不到的校验机制拒绝。**

LLM 说 "我完成了前端开发"，系统说 "你这个摘要和我们后端目标描述更像，我不信"。LLM 不知道发生了什么事，只能再试一次，输出一样的 PROGRESS，系统再拒绝——循环就来了。

**正确做法：** 如果怀疑 PROGRESS 是上一个子目标串台了，就把怀疑告诉 LLM，让 LLM 自己确认，而不是直接否决。或者干脆信任 LLM——LLM 在 `completed` 标签之外主动输出 `waiting_user`，说明它对状态的判断能力远比 bigram 靠谱。

目的是**防止 LLM 在子目标切换后产生困惑**——自动推进到下一个子目标时，如果 LLM 还在输出上一个子目标的 PROGRESS 标签，系统会把新子目标错误地标记为完成。

比如 LLM 刚做完"制定开发计划"，PACE 自动切到"开发核心功能"，但 LLM 这轮还是输出了 `PROGRESS: completed, summary: "已完成开发计划制定"`，如果不校验，就会把"开发核心功能"也标记完成。

但问题是 bigram 匹配太粗糙了——用 0.08 的阈值比较两个中文短文本的 2-gram 重叠率，对长摘要 vs 短描述几乎随机。而且现实中 LLM 很少出现这种"跨子目标混淆"的情况，反而是正常完成被频繁否决，导致子目标无限重试。代价远大于收益。
