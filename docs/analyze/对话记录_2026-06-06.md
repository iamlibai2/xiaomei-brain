## 它做什么

> 比较 PROGRESS 摘要和（当前目标 + 所有兄弟目标）的 bigram 重叠度。如果摘要和某个兄弟目标的匹配度显著高于当前目标，判为"串台"。

## 它的问题

但这个逻辑的核心假设是错的：

> **假设：PROGRESS 摘要的内容 = 目标描述中的词汇。**

实际上 LLM 的 PROGRESS 摘要描述的是**产出**，不是**目标本身**。目标描述是"执行网站前端界面设计与开发"，而产出是"已完成4个页面（首页/产品中心/工程案例/联系我们）"。产出自然提到具体的页面名称，这些名称可能出现在其他兄弟目标（后端开发）的示例中。

换句话说，一个正确的 PROGRESS 不一定和当前目标描述长得像。用 bigram 来判断"是不是在说当前子目标"，本质上把"产出报告"和"任务标题"混为一谈了。

## 真正的问题本质

不是 bigram 阈值设得对不对的问题。而是：**LLM 不应该被一个自己看不到的校验机制拒绝。**

LLM 说 "我完成了前端开发"，系统说 "你这个摘要和我们后端目标描述更像，我不信"。LLM 不知道发生了什么事，只能再试一次，输出一样的 PROGRESS，系统再拒绝——循环就来了。

**正确做法：** 如果怀疑 PROGRESS 是上一个子目标串台了，就把怀疑告诉 LLM，让 LLM 自己确认，而不是直接否决。或者干脆信任 LLM——LLM 在 `completed` 标签之外主动输出 `waiting_user`，说明它对状态的判断能力远比 bigram 靠谱。

## Agent 生命周期流程分析

整个生命周期由三层架构组成，自底向上是：

```
Living（纯状态机） → ConsciousLiving（带意识） → 各子系统线程
```

---

### 一、状态机流转

```
DORMANT → WAKING → AWAKE ⇄ IDLE → SLEEPING ⇄ DREAMING
  ↑                                      ↓
  └──────────── 死亡后复活 ──────────────┘
```

**6 个状态**（`LivingState`）：

| 状态 | 含义 | 触发条件 |
|------|------|----------|
| `DORMANT` | 死亡休眠 | 生存欲归零（`_check_death`） |
| `WAKING` | 启动过渡 | `run()` 入口，瞬间完成 |
| `AWAKE` | 清醒，处理消息 | WAKING 完成后 / 收到消息唤醒 |
| `IDLE` | 浅空闲，自主行为 | 空闲 > `idle_short`（默认 300s） |
| `SLEEPING` | 深度睡眠 | 空闲 > `idle_threshold`（默认 1800s） |
| `DREAMING` | 梦境周期 | Layer2 发出入梦信号 |

---

### 二、主循环流程（`Living.run()`）

```
run()
  ├─ transition(WAKING)
  ├─ _on_wake()          ← 启动初始化（一次性）
  ├─ transition(AWAKE)
  │
  └─ while _running:
       ├─ AWAKE    → _loop_awake()     # 等待消息 / 空闲计时
       ├─ IDLE     → _loop_idle()      # 自主行为 / 意图消费
       ├─ SLEEPING → _loop_sleeping()  # 等待消息 / 等待入梦
       ├─ DREAMING → _loop_dreaming()  # 运行 DreamEngine
       └─ DORMANT  → _loop_dormant()   # 等待复活消息
```

每个循环内部每 tick（1s）执行 `_tick_periodic()` 触发注册的周期任务。

## 梦境执行

`ConsciousLiving._loop_dreaming()` 完全覆盖了基类的循环等待逻辑，改为**同步一次性执行**：

```python
# conscious_living.py:1042-1065
def _loop_dreaming(self):
    if self._should_skip_dreaming():   # 无意识模式 → 直接回 SLEEPING
        self._transition(LivingState.SLEEPING)
        return

    # 串行执行 DreamEngine
    report = self._dream_engine.run()
    # DreamEngine: 情绪整理 → 记忆强化 → L3深度燃烧 → 反省

    self._transition(LivingState.SLEEPING)  # 完成后直接回 SLEEPING
```

与基类的区别：基类 `_loop_dreaming()` 是一个 while 循环，会等待消息、支持超时。`ConsciousLiving` 直接执行完就切回 SLEEPING，不等待、不可中断。

## 退出梦境流程

**正常退出（唯一路径）：**

```
_loop_dreaming()
  └─ DreamEngine.run() 完成
       └─ transition(SLEEPING)   ← 直接切回
```

**异常退出：DreamEngine 崩溃**

```python
except Exception as e:
    logger.error("DreamEngine 运行失败: %s", e)
# 异常后仍然 transition(SLEEPING)
```

**注意**：`ConsciousLiving` 的 `_loop_dreaming()` **没有消息等待**，因此 DREAMING 期间收到消息**无法中断梦境**。必须等 DreamEngine 跑完。

基类 `Living._loop_dreaming()` 虽然有消息等待逻辑（收到消息 → 切 AWAKE），但因为被子类完全覆盖，不会执行。

## 为什么是 Layer2 发出梦境信号

**因为入梦是一种"后台自我反思"行为，不属于消息驱动的前台对话。** Layer2 是默认网络线程，专管 L2 加柴、L3 沉思、入梦这类"面向自己"的后台行为。

---

## 入梦信号的触发条件

在 `layer2.py:151-159`，条件非常简单：

```python
# DREAM: 入梦信号（仅 SLEEPING）
if agent_state == "sleeping":
    if self._c._sleep_start_time == 0:
        self._c._sleep_start_time = time.time()     # 记录入睡时间
    elapsed = time.time() - self._c._sleep_start_time
    if elapsed >= self._c._cc.l3_dream_interval:    # 睡眠时长 >= 阈值
        self._c._sleep_start_time = 0               # 重置计时器
        self._c._dream_signal = True                # 发出信号
else:
    self._c._sleep_start_time = 0                   # 非 SLEEPING → 清零
```

**两个条件（AND）：**

1. **agent 当前处于 SLEEPING 状态** — 主循环每 tick 写 `consciousness._agent_state`
2. **持续睡眠时长 >= `l3_dream_interval`** — 配置项，一旦离开 SLEEPING 计时器就清零

`_sleep_start_time` 是一个"睡眠连续计时器"。如果 agent 从 SLEEPING 被消息唤醒切回 AWAKE，下次再进 SLEEPING 会重新从 0 开始计时。

---

## 时间差：Layer2 发送 vs 主循环消费

Layer2 线程每 `check_interval`（默认 10s）检查一次，而主循环的 `_heartbeat()` 每 `tick_interval`（1s）检查一次。所以：

```
Layer2 设 _dream_signal = True
         ↓  (最多 1s 后)
_heartbeat() 读到 → HEARTBEAT_DREAM
         ↓
_loop_sleeping() → transition(DREAMING)
```

消费发生在 `_heartbeat()` 中，且额外约束 `state == SLEEPING` 才生效——这是双重校验，防止在 Layer2 发信号和主循环消费之间状态发生了变化。

---

## 为什么不由主循环自己决定

如果主循环自己计时决定入梦，那每次 tick 都要嵌一层判断。Layer2 独立线程发信号的好处：

- **关注点分离**：主循环只关心状态机流转，Layer2 决定"什么时候该入梦"
- **共享 Consciousness 状态**：Layer2 能读到 L2 加柴结果、L3 是否刚执行过、梦境间隔配置等，能做更丰富的决策（当前版本还是仅计时，但预留了扩展点）

## 可优化点

### 一、梦境入口过窄

**现状**：`idle_threshold = 10800s`（3小时），且 IDLE 期间每次 ActionDispatcher 消费意图都会重置 `_last_active`，导致 SLEEPING 几乎进不去。

**优化方向**：
- 降低 `idle_threshold` 默认值（如 1800s）
- 区分"用户交互"和"自主行为"，意图消费不应重置空闲计时。`ActionDispatcher.process_queue()` 返回 `True` 时不应该 `_last_active = time.time()`，而是只重置一个独立的 `_last_autonomous` 计时器

---

### 二、DREAMING 期间无法被消息中断

**现状**：`ConsciousLiving._loop_dreaming()` 完全覆盖了基类的 while 循环，DreamEngine 串行跑完才切回 SLEEPING。用户发消息只能排队等待。

**优化方向**：
- 复用基类 `Living._loop_dreaming()` 的消息等待机制：将 DreamEngine.run() 放到独立线程，主循环继续等待消息
- 或者在 DreamEngine 各阶段之间插入 `_wait_message(timeout=0)` 检查点，收到消息即可提前退出

---

### 三、Layer2 轮询浪费

**现状**：Layer2 每 10s `time.sleep()` 后检查一遍所有条件。大部分时候条件都不满足，纯空转。

**优化方向**：
- 改为事件驱动：L1 检测到异常 → 直接唤醒 Layer2；冷却时间由 `threading.Event.wait(timeout)` 替代轮询

## 梦境系统分析

### 入梦流程（SLEEPING → DREAMING）

三段式信号链：

```
[Layer2 线程]  每10s检查 → SLEEPING 满 sleep_to_dream_threshold(300s) → _dream_signal = True
[心跳]        每1s检查 → 消费 _dream_signal → HEARTBEAT_DREAM
[_loop_sleeping]  收到 HEARTBEAT_DREAM → transition(DREAMING)
```

- **触发条件**：用户无交互 → idle_short(300s) → IDLE → idle_threshold(10800s/3h) → SLEEPING → 再等 300s → DREAMING
- **从用户离开到入梦**：约 3小时 + 5分钟
- **中断**：SLEEPING 中收到用户消息立即切回 AWAKE

### 梦境执行（DREAMING → DreamEngine.run()）

单次执行（非循环），5个阶段**顺序**执行：

| 阶段 | 做什么 | 依赖 |
|------|--------|------|
| 1. 情绪处理 | 根据上次梦境摘要，正则匹配调整 Drive 值 | 无需 LLM |
| 2. 记忆整理 | 强化过期记忆 + LLM 提取新记忆 + 关系维护 | LLM |
| 2.5 流程巩固 | 衰减/归档过期流程权重 | 无需 LLM |
| 2.x 叙事巩固 | 归档低权重叙事 + 按场景标签合并 + 推送成长日志 | 无需 LLM |
| 3. 模式提取 | 从经验流提取行为模式 | 无需 LLM |
| 4. 梦境燃烧 | LLM 生成梦境意识报告，存储摘要 + 生成后续意图 | LLM |
| 5. 反省 | 预留空壳，未实现 | - |

```
DreamReport:
├── summary → SelfImage.history.last_dream_summary（下次醒来 greeting 用）
```

- 继续睡够 300s → 再做梦（可循环）

```
goals                        ← purpose/persistence.py
```

### 任务执行（metacognition/goal_run_storage.py）
```
goal_runs                    ← 每次执行目标的完整记录
  ├── 开始/结束时间、总步数
  ├── LLM调用数、工具调用数
  ├── 意外检测、升级次数、自动推进
  ├── 子目标完成/失败、退出原因
  └── goal_steps (FK) ────── 每步的详细执行信息
        ├── LLM输出、工具调用、耗时
        ├── PROGRESS标签、进度状态
        ├── 意外、重试、升级、视角切换
        └── goal_log ──────── 执行过程中的日志流
              └── entry_type: output / thinking / error / decision

pace_checkpoints             ← PACE自检快照
  ├── 观察记录、预算消耗
  ├── 连续空步计数
  └── 最后推进提示
```

### 项目认知（metacognition/project_mental_model.py）
```
project_map                  ← 项目认知地图（按 agent_id + project_id 隔离）
  ├── structure              模块/层次/依赖
  ├── conventions            命名/代码风格/测试习惯
  ├── history                设计决策/踩坑记录
  ├── current_state          进度/阻塞/下一步
  └── quality_standards      质量标准
```

**总结**：`goals` 存"要做什么"，`goal_runs/steps/log` 存"怎么做的和做得怎么样"，`pace_checkpoints` 存"自检快照"，`project_map` 存"对项目环境的认知"。
