## 1. ActionExecutor 执行


**🟥 用户提问（2026-05-02 00:02:15）**


>      ActionExecutor 执行

             ├── TRIGGER_L3 → 触发 L3 深度燃烧
             ├── TOOL      → 执行工具（learn_topic/progress_goal）
             └── NOTIFY    → 打印通知
             │
             ▼
     intent 消费（从 intent_buffer 清除） 为什只有意图被消费，其他的呢？


**助手回答**


好问题。原因是**Desire/Goal/System 的"条件"不是存在 buffer 里，而是 SelfImage 的实时状态值**：

- `desire_belonging > 0.7`：这个值一直在 Drive 层，不消耗
- `goal_progress > 0.6`：目标进度也是持续存在的状态
- `energy < 0.3`：能量值实时存在

这些没有"消费"机制，因为条件本身是**持续状态**，不是**一次性事件**。冷却时间（cooldown）来解决重复触发问题。

Intent 不一样——`GREET` 意图是一次性事件，发完就没了，所以要从 buffer 里清掉。如果不清理，下一秒 `intent_buffer` 里还有 `"GREET"`，又匹配规则，又发消息，形成死循环。

**但 Desire/Goal 确实有潜在问题**：如果 Drive 欲望值一直 > 阈值，每次 `tick()` 都会匹配到（冷却期内除外），这其实是**预期行为**——欲望强就想做这件事，直到欲望衰减下去或者用户回应了。

你问的意思是不是：Desire/Goal 类型的 ActionItem，执行完了要不要也加个"消费"机制，防止重复执行？

## 2. 对，你最好查查看看他们的运行机制


**🟥 用户提问（2026-05-02 00:04:24）**


> 对，你最好查查看看他们的运行机制


**助手回答**


查完了，说清楚它们的区别：

## Intent（一次性事件）

- `GREET/CARE/REFLECT/ACT` 是离散事件，写入 `intent_buffer`
- 执行后**不会自动消失**，必须手动清掉，否则死循环
- 没有衰减机制，没有冷却机制（冷却在 ActionDispatcher 层面）

## Desire（持续状态 + 多重调节）

欲望值是连续值，有三层自我调节：

1. **执行后自然消耗**：`greet_user` 成功后调用 `on_desire_satisfied("belonging", 0.2)` → belonging 下降 0.2，可能跌回阈值以下
2. **小时级衰减**：`tick_hour()` → 所有欲望值向基础值恢复
3. **ActionDispatcher 冷却**：冷却期内不重复匹配

所以 desire 不需要"消费"机制——价值自己会衰减，执行后会自动降，冷却防止重复。

## Goal（目标驱动）

- 行为执行后要么**完成目标**（条件变假不再匹配），要么**部分推进**（成就欲部分满足）
- `on_desire_satisfied("achievement", 0.3)` 降低成就欲

## System（能量状态）

- 能量低于阈值 → 通知 → cooldown 30 分钟
- 能量恢复靠睡眠，不靠"消费"

---

**总结**：Intent 是离散事件必须手动清，Desire/Goal/System 都是连续状态，有衰减+冷却+执行后满足三重机制，不需要消费。

## 3. ：greet_user 成功后调用 on_desire_satisfied("belonging", 0.2)，这个值同...


**🟥 用户提问（2026-05-02 00:11:51）**


> ：greet_user 成功后调用 on_desire_satisfied("belonging", 0.2)，这个值同步到selfimage了吗


**助手回答**


同步链路过一遍：

```
_on_wake() / L2 触发后
    ↓
_sync_self_image_from_subsystems()   ← Drive.desire → SelfImage.desire_*
    ↓
_dispatcher.tick(si)
    ↓
匹配 desire.greeting → 产出 ActionItem
    ↓
process_queue() → ActionExecutor._do_tool()
    ↓
DesireActionExecutor.execute(action)
    ↓
_do_greet_user() → on_desire_satisfied("belonging", 0.2)  ← Drive.desire.belonging 下降
    ↓
同步只在 HEARTBEAT 下次发生，Sync 不在 process_queue() 之后立刻调
```

所以答案是：**同步了，但有时机问题**。

- `on_desire_satisfied()` 改了 Drive 层
- 但 SelfImage 的 `desire_belonging` **要等到下次 `_heartbeat()` 调用 `_sync_self_image_from_subsystems()` 时才更新**

这意味着：**同一 tick 内，greet 成功后欲望值虽然降了，但 SelfImage 还不知道，下一个 tick 如果 `_heartbeat()` 还没到来，规则还会再匹配**。

冷却机制兜底了——`cooldown("desire_greet", 3600)` 让同一 key 的动作不会重复触发。但**欲望值本身在 SelfImage 里还没更新**，这确实是个同步时机的问题。

要改的话，`process_queue()` 执行完 desire action 后立刻调 `_sync_self_image_from_subsystems()`，或者 `_do_tool` 里的 action 执行完直接同步对应字段。

## 6. 现在不是用build_context 吗？to_prompt_text()又是个啥


**🟥 用户提问（2026-05-02 00:51:53）**


> 现在不是用build_context 吗？to_prompt_text()又是个啥


**助手回答**


你说得对，我之前搞混了。`to_prompt_text()` 是 `SelfImage` 的一个方法，但**它不是给 Agent 用的**。

## 实际架构

**Agent system prompt 的真正来源**：

```
用户消息
    ↓
build_context(agent, user_input, consciousness_state, intent_context)
    ↓
    ┌─────────────────────────────────┐
    │ 1. determine_mode()             │  ← consciousness_state 只影响模式选择
    │    (flow / daily / reflect)     │    不进 system prompt
    │
    │ 2. context_assembler.assemble()  │  ← SelfModel.to_prompt_text() 生成 system prompt
    │    → 读取 SelfModel (talent.md)  │    不读 SelfImage Drive 字段
    │
    │ 3. intent_context 注入           │  ← 来自 Purpose 层（_build_intent_context）
    │    (目标/子目标信息)              │    不含 desire/emotion/energy
    │
    │ 4. intent_context 追加到         │
    │    system prompt 尾部            │
    └─────────────────────────────────┘
    ↓
[system prompt + intent_context, DAG摘要, agent.messages]
```

**所以 P1 问题的根因**：

- `SelfImage.to_prompt_text()` → 只给 L2 加柴用（`get_state_summary()`），**不进 Agent**
- `SelfImage` 的 Drive 字段（desire_* / emotion_* / energy_level）→ 只进了 `_get_consciousness_state()` 的字典 → 只影响 `determine_mode()` → **不进 system prompt**
- Agent system prompt 来自 `SelfModel.to_prompt_text()`（talent.md），完全没有 consciousness 状态

**结论**：P1 问题确实存在且严重——意识状态只影响了"用什么模式"，完全没有进 system prompt。需要从 `SelfModel.to_prompt_text()` 或 `context_assembler.assemble()` 的某个环节注入 Drive 状态。
