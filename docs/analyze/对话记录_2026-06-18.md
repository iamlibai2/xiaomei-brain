## OpenClaw 怎么做

**一层抽象就够了：8 种 API 接口**

```
所有 provider → 映射到 8 种 API 之一：
  ├── openai-completions    ← 绝大多数（DeepSeek/MiniMax/火山...）

  ├── google-generative-ai  ← Gemini
  ├── ollama / bedrock-converse / github-copilot / ...
```

**新增一个 provider 只需要填配置**，不需要写代码：

```typescript
// 以 DeepSeek 为例
{
  baseUrl: "https://api.deepseek.com/v1",
  api: "openai-completions",        // 选一个 API 接口
```

**Provider 差异用 `compat` 字段表达**，不改核心逻辑：

```typescript
compat: {

  maxTokensField: "max_tokens",      // 字段名差异
}
```

**运行时适配走 wrapper 链**，往 `streamSimple()` 外面包一层层函数，每层处理一个 provider 的特定转换（tool schema 互转、beta headers、thinking 格式）。

## 差距

| | OpenClaw | xiaomei-brain |
|---|---|---|
| API 接口抽象 | 8 种，provider 选一种 | 1 种（openai-completions） |
| 新增 provider | 填配置，不写代码 | 改 `llm.py` + `config.py` |
| Provider 差异 | `compat` 字段统一管理 | `if model/ if provider` 散落 |
| 运行时适配 | wrapper 链，可组合 | 内联到 `_build_messages()` |

## LLM 适配层重构设计

### 模块结构

```
src/xiaomei_brain/llm/
├── types.py              # 类型定义（ModelApi / ProviderConfig / ModelCompat）
├── providers.py          # Provider 解析与注册（从 config.json 发现）
├── compat.py             # 模型能力归一化（compat 字段默认值/校验）
├── client.py             # 运行时客户端（chat / chat_stream / 重试 / 退避）
├── stream_adapters.py    # 8 种 API 的流式适配器
└── __init__.py           # 对外暴露 LLMClient + 构建函数
```

### 各模块职责

**`types.py`** — 数据结构

```python
class ModelApi(enum.Enum):
    OPENAI_COMPLETIONS = "openai-completions"
    OPENAI_RESPONSES = "openai-responses"
    ANTHROPIC_MESSAGES = "anthropic-messages"
    GOOGLE_GENERATIVE_AI = "google-generative-ai"
    BEDROCK_CONVERSE_STREAM = "bedrock-converse-stream"
    OLLAMA = "ollama"
    GITHUB_COPILOT = "github-copilot"
    OPENAI_CODEX_RESPONSES = "openai-codex-responses"

@dataclass
class ModelCompatConfig:
    supports_developer_role: bool = True
    supports_reasoning_effort: bool = False
    supports_usage_in_streaming: bool = True
    supports_strict_mode: bool = True
    supports_tools: bool = True
    max_tokens_field: str = "max_tokens"
    thinking_format: str | None = None  # "openrouter" / "qwen-chat-template"
    requires_tool_result_name: bool = False
    ...

@dataclass
class ModelDefinition:
    id: str
    name: str
    api: ModelApi
    context_window: int
    max_tokens: int
    reasoning: bool = False
    input_modes: list[str] = field(default_factory=lambda: ["text"])
    cost: dict = field(default_factory=dict)
    compat: ModelCompatConfig = field(default_factory=ModelCompatConfig)

@dataclass
class ProviderConfig:
    name: str
    base_url: str
    api: ModelApi
    api_key: str = ""
    headers: dict = field(default_factory=dict)
    models: list[ModelDefinition] = field(default_factory=list)
```

**`providers.py`** — Provider 注册表

- `ProviderRegistry.from_config(config_json_path)` — 从 `~/.xiaomei-brain/config.json` 的 `models.providers` 解析所有 provider
- `ProviderRegistry.resolve(provider, model_id)` — 根据 `provider/model_id` 查找 `ProviderConfig + ModelDefinition`
- 处理 API key 解析：config 值 > 环境变量（如 `$DEEPSEEK_API_KEY`）
- 处理 `baseUrl` 归一化（去掉尾部 `/v1` 等）

**`compat.py`** — 兼容性归一化

- `normalize_compat(api, model_def)` — 根据 `api` 类型和 `base_url` 自动设置 compat 默认值
- 核心规则：非 `api.openai.com` 的 `openai-completions` 端点，`supports_developer_role`、`supports_usage_in_streaming`、`supports_strict_mode` 默认设为 `False`
- 这些规则是 OpenClaw 从大量 provider 中总结出来的，直接复用

**`client.py`** — 运行时客户端

- `LLMClient(model_id, /, provider)` — 构造时查 ProviderRegistry 获取 endpoint + compat
- `chat(messages, tools)` — 非流式，复用现有重试/退避逻辑
- `chat_stream(messages, tools)` — 流式，按 `model_def.api` 分发到对应 stream adapter
- 协议错误（401/402/403）→ `FatalLLMError`，可恢复错误（429/5xx）→ 重试

**`stream_adapters.py`** — API 适配器

- 每个 `ModelApi` 对应一个流式 handler：
  - `_stream_openai_completions()` — 现有逻辑，SSE `data: {...}` 解析
  - `_stream_anthropic_messages()` — SSE `event: content_block_delta` 格式，tool schema 互转
  - `_stream_google_generative_ai()` — JSONL 流式格式
  - 其余 5 种先 stubbed，使用时再实现
- 统一的输出：`Generator[str, None, None]` + 结束后的 `ChatResponse`

### 和现有代码的关系

- `src/xiaomei_brain/base/llm.py` → 移入 `src/xiaomei_brain/llm/` 并重构
- `src/xiaomei_brain/base/config.py` → `Config.from_json()` 解析 `models.providers` 的逻辑保留，移交给 ProviderRegistry
- `src/xiaomei_brain/agent/agent_manager.py` → `_DEFAULT_CONFIG_TEMPLATE` 的 `models.providers` 格式不变
- 其他调用方（`agent/core.py` 等）→ 只改 import 路径，`LLMClient` API 保持兼容

每个文件就是一个 provider，格式：

```python
# deepseek.py
class DeepSeekProvider(ProviderPlugin):
    name = "deepseek"
    base_url = "https://api.deepseek.com/v1"
    api = ModelApi.OPENAI_COMPLETIONS
    
    @classmethod
    def models(cls) -> list[ModelDefinition]:
        return [
            ModelDefinition(
                id="deepseek-v4-flash",
                name="DeepSeek V4 Flash",
                context_window=128000,
                max_tokens=8192,
                ...
            ),
        ]
    
    # 只在非标准时才需要 override
    @classmethod
    def compat_overrides(cls, model_id: str) -> dict:
        return {"supports_developer_role": False}
```

- **标准 provider**：实现 `name` / `base_url` / `api` / `models()` 即可，不需要写 stream 处理
- **非标准 provider**：可 override `stream_adapter()`、`build_payload()` 等方法
- **第三方扩展**：用户在自己的项目里放一个 `.py`，通过环境变量 `XIAOMEI_LLM_PROVIDERS_DIR` 指定路径即可

## 三架构对比

| | OpenClaw (TS) | Hermes (Python) | 我们当前设计 |
|---|---|---|---|
| Provider 抽象 | plugin + buildProvider() | **一个 `ProviderProfile` 数据类** | ProviderPlugin ABC |
| Wire 协议数 | 8 种 | **4 种** | 8 种 |
| 差异处理 | `compat` 字段 | **hook 方法 override** | `compat.py` 归一化 |
| 响应类型 | pi-ai 内部 | **`NormalizedResponse` + `provider_data`** | `ChatResponse` |
| 复杂度控制 | stream wrapper 链 | **"Profile-first, flag-fallback"** | 全走 adapter 分发 |

## 该调整的点

1. **ProviderProfile 替代 ABC** — Hermes 证明了一个数据类就够了。90% 的 provider 直接 `ProviderProfile(name=..., base_url=..., models=...)`，不需要写子类。只有 DeepSeek/Anthropic 这种需要自定义 hook 的才子类化。比 ABC 轻很多。

2. **Transport 缩减为 4-5 种** — Hermes 4 种传输层（chat_completions / anthropic_messages / codex_responses / bedrock_converse）覆盖了 32 个 provider。`google-generative-ai` 和 `ollama` 都可以走 `chat_completions` 传输层，不需要单独实现。

3. **NormalizedResponse + provider_data** — 所有传输层输出统一的 `NormalizedResponse`，协议特定数据放进 `provider_data` 字典，不污染共享字段。

## 3 个需要修的差距

**Gap 1 — Per-model compat 丢了**

OpenClaw 的 `compat` 是挂在每个模型上的，因为同一个 provider 下不同模型能力不同（GPT-4o 支持视觉，GPT-3.5 不支持）。我们的 hook 方法是 provider 级别的，改了就影响所有模型。

应该把 `compat` 加回 `ModelDefinition`，作为 per-model 的可选覆盖：

```python
@dataclass
class ModelDefinition:
    id: str
    name: str
    context_window: int
    max_tokens: int
    reasoning: bool = False
    input_modes: list[str] = field(default_factory=lambda: ["text"])
    cost: dict = field(default_factory=dict)
    # Per-model 覆盖 provider 默认行为
    supports_vision: bool | None = None
    supports_tools: bool | None = None
    supports_developer_role: bool | None = None
    max_tokens_field: str | None = None      # "max_tokens" / "max_completion_tokens"
```

`None` = 沿用 provider/transport 默认值。

**Gap 2 — 用户插件不能覆盖内置插件**

跟 Hermes 一样的 `_fetch()` 三级缓存：

我们项目有两种扩展机制：

### Transport（对话模型专用）

纯配置驱动。`api: "openai-completions"` 一个 Transport 搞定 13 家。因为协议统一，加 provider 零代码。

### Plugin（工具专用）

代码驱动。图像、音乐、TTS 各家 API 不同，每个新 provider 要写 adapter。Plugin 框架就是管理这些 adapter 的——注册、发现、生命周期。

两者**不冲突**：Transport 处理 `models.providers.xxx.chat`，Plugin 处理工具能力（图像生成 / 音乐 / TTS）。

现在我们有两个选择：

1. **先不做插件**，直接把 minimax → doubao 的工具模型支持加了
2. **先抽象插件接口**，把现有 TTS/音乐/图像改造成插件，再加 doubao
