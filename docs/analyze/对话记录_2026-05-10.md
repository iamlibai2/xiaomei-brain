## 为什么复杂任务做不了

你说的"这层缺失"触及了核心。当前 AI Agent 的范式本质上是：

```
感知 → 规划 → 执行 → 报告
```

这不是在执行任务，这是在**执行脚本**。区别在于：

### 执行脚本 vs 执行任务

| | 执行脚本 | 执行任务 |
|---|---|---|
| 前提假设 | 世界是静态的 | 世界会变 |
| 对错误的反应 | 报错退出 | 调整路径 |
| 对中间发现 | 忽略 | 重新评估 |
| 对自己的认知 | 不知道不知道 | 知道自己不知道 |
| 成功标准 | 每一步都按计划做完 | 目标达成 |

Metacognition 缺失的直接后果是：Agent 在执行的第一步就已经"死"了——它已经把自己锁在了一条线性路径上。

### 人类怎么做复杂任务

写一个不熟悉的代码库时：

```
1. 开始改 → 发现比预期复杂（surprise detection）
2. 暂停 → "我可能低估了"（self-assessment）
3. 重新读代码 → 发现新依赖（knowledge update）
4. "要不换个思路？"（strategy shift）
5. 继续 → 又遇到问题 → "这个我真不会"（capability boundary）
6. 决定：问同事 / 搜索 / 简化目标（replan）
```

关键不是每一步都做对，而是**每一步都在重新评估**。这个循环不是 bug，是 feature：

```
行动 → 观察 → 比较预期 → 发现偏差 → 更新认知 → 调整行动
```

### 两个关键能力

这层要做的，拆开来看是两个能力：

**1. Surprise detection（意外检测）**

不是"出错了"这种明确的错误，而是"不对劲"——一种预测和现实的偏差。比如：
- "这个函数应该很简单，但我已经读了 10 分钟了"
- "这个方案我在别的项目用过，但这里似乎不适用"
- "用户说'随便'但语气不太对"

LLM 其实有能力做这个——你让它反省，它能说出哪里不对劲。问题是没人让它做。当前流程是：LLM 输出 → 解析 PROGRESS 标签 → 下一个子目标。中间没有"等等，好像哪里不对"的环节。

**2. Capability calibration（能力校准）**

知道自己什么会、什么不会、什么不确定。这也是当前完全缺失的：
- Goal 拆解时从不问"我能做这个吗"
- 遇到困难时从不判断"这是暂时的障碍还是根本做不到"
- 完成后从不比较"实际做的和当初预计的差距"

---

## 这层缺失的深层原因

不是技术上做不到，而是**当前的 Agent 架构没有给这层留位置**。

```
当前架构：
  Agent ← LLM 就是一切，决策和执行混在一起

应该的架构：
  执行层（Agent）→ 干什么
  元认知层（Metacognition）→ 干得对吗？要不要换个干法？
```

LLM 本身能做的事和 LLM 被调用的方式是两回事。Metacognition 之所以需要独立——不是因为 LLM 做不到这些判断，而是因为**判断者不能是执行者自己**。

就像你写代码时，debug 的那个人不能是刚好写完这段代码的那个状态——你需要"切换一下脑子"。这就是 Metacognition 作为独立层的认知意义：它不是能力更强，是视角不同。

---

## 一个设计思想：认知回合制

抛开代码，如果重新设计，我倾向于把复杂任务执行变成**回合制**，而不是流水线：

```
每个回合：
  1. Metacognition 读取当前状态 → 产出一个问题
  2. Agent 回答这个问题
  3. Metacognition 评估回答 → 决定下一回合做什么
```

回合的具体形式：

| 阶段 | Metacognition 的问题 | 可能的 Agent 回答 |
|------|---------------------|------------------|
| 开始前 | "我理解对了吗？有没有遗漏的信息？" | "我注意到用户说'快一点'，可能需要简化方案" |
| 每步后 | "这一步的结果和预期一致吗？学到了什么？" | "发现 API 文档有误，实际行为与文档不符" |
| 卡住时 | "为什么卡住？是暂时障碍还是根本问题？" | "这个需要用户提供权限，我做不到" |
| 完成后 | "实际执行和最初规划的差距在哪？下次怎么做更好？" | "第二阶段花了预期 3 倍的时间，低估了测试工作量" |

---

## 回到 Task/Goal 的问题

现在再看 Task 的定位就清楚了：

- Task 的 `cognitive_log` 其实就是 Metacognition 的**回合记录**
- 暂停/恢复是 Metacognition 的**回合间状态保存**
- 确认流程是 Metacognition 的**pre-execution check**（只是没实现）
- 知识提取是 Metacognition 的**post-execution lesson**

这些东西不应该挂在 Purpose 上（Purpose 管"做什么"），也不应该挂在 Task 数据类上（Task 只是数据），而应该是一个独立的**认知控制器**。

TaskManager 应该是这个控制器的雏形——只是它被实现成了数据管理而不是认知管理。

让 Agent 做一个"调研某个开源项目的架构"的任务。

当前流程：
```
1. 搜索项目 → 得到一堆链接
2. 读 README → 总结
3. 读源码 → 总结
4. 组装报告 → 完成
```

问题在哪？第 2 步结束时，Agent 已经知道这个项目用了微内核架构，但它不会停下来想：**"微内核架构的话，我应该去看看它的 IPC 机制，而不是按原计划逐个读目录"**。它不会，因为没人问它。

如果你在这一刻介入，单独问 LLM："根据你目前了解的信息，你觉得下一步应该重点看什么？"——它能给出非常好的答案。

**所以能力是有的，只是架构不给它这个机会。**

## 当前范式的天花板

| 什么 | LLM 能做到吗 |
|------|------------|
| 理解上下文 | ✅ |
| 规划步骤 | ✅ |
| 执行单步 | ✅ |
| 执行过程中发现"不对劲" | ✅（如果你问它的话） |
| 主动暂停并重新评估 | ❌（没人问它） |
| 认识到自己走了弯路 | ❌（没人问它） |
| 从失败中提炼教训 | ❌（没人问它） |
| 承认"这个我做不到" | ❌（没人问它） |

后面四个不是 LLM 不行，是**调用架构不允许它这么做**。

## 本质区别

```
当前架构：
  LLM 是一个工具 —— 你问它，它回答。
  
Metacognition 架构：
  LLM 是一个心智 —— 它有一个持续运行的自我监控循环。
```

区别不在于 LLM 本身，在于**有没有一个独立的、持续的、不参与执行的观察者**。这个观察者也可以用 LLM 实现——但它是一个不同的 instance，带着不同的 context，问不同的问题。

就像你写代码时，过一会站起来走一走，脑子自动开始想"刚才那个方向对吗？"——不是你刻意切换的，是你大脑里一直有一个后台进程在跑。

**Metacognition 就是那个后台进程。当前 Agent 没有后台进程，只有一个前台线程。**

---

一句话总结：**不是 LLM 做不了复杂任务，是我们只给了它一个"做事"的接口，没给它一个"想事"的空间。**
