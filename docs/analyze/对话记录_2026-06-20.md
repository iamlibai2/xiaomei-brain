不是不可能，只是需要不同的抽象层。

LLM provider 能统一是因为"文本对话"这个语义本身高度标准化：

```
输入: messages[]  →  输出: choices[].message.content
```

所有厂商都能映射到这个模型上，所以 `Transport.chat_completions()` 的签名是通用的。

图片生成的语义同样可以抽象：

```
输入: prompt + config  →  输出: image files
```

只是粒度不同。比如可以定义：

```python
class ImageProvider(ABC):
    def generate(self, prompt: str, size: str, n: int, **kwargs) -> list[bytes]: ...
```

然后：

| 厂商 | 适配方式 |
|------|---------|
| 豆包 | `self.client.images.generate(prompt=prompt, size=size, n=n)` → 从 response.data 取 b64_json |
| MiniMax | `POST /v1/image_generation` 传 `aspect_ratio` 映射到 `size` → 从 response 下载 image_urls |

真正的区别在于：**LLM provider 的核心接口是 `/v1/chat/completions`，行业已在协议层面统一。图片生成协议层面没统一，但语义层面是统一的**，所以 provider 模式仍然适用——只是需要在适配器里做更多转换，而不是像 text provider 那样基本透传。

要不要做？可以做，但等价于等图片生成领域自己出一版"Chat Completions 式的协议标准"——现在各家还在各自探索，或者像豆包这样直接跟 OpenAI 的 `images.generate()` 接口靠拢。

OpenClaw 的做法很清晰：

## 一个工具，多个 Provider

```
image_generate（唯一工具）
    │
    ├── google/gemini-3-pro-image-preview
    ├── openai/gpt-image-1  
    ├── fal/fal-ai/flux/dev
    └── minimax/image-01
```

**核心机制：**

1. **Provider 是对象字面量**，不是类。只要满足接口即可注册：

```typescript
type ImageGenerationProvider = {
  id: string;                    // "google", "minimax" ...
  label?: string;
  defaultModel?: string;
  models?: string[];             // 这个 provider 支持哪些模型
  capabilities: {                // 这个 provider 的能力
    generate: { maxCount, maxPromptLength, sizes, ... }
    edit: { ... }
    geometry: { aspectRatios, resolutions }
  };
  generateImage: (req) => Promise<ImageGenerationResult>;
};
```

2. **一个 `image_generate` 工具**，LLM 调用时传 `model` 参数指定 `"provider/model"`：
   ```
   image_generate(prompt="...", model="google/gemini-3-pro-image-preview")
   ```

3. **配置层控制默认值**，有 primary + fallback 链：
   ```yaml
   imageGenerationModel:
     primary: "google/gemini-3-pro-image-preview"
     fallbacks:
       - "minimax/image-01"
   ```

4. **每个 provider 自己管自己的 API 调用**，运行时不关心内部实现

跟你当前 `web_search` 的 provider 派发模式思路类似，但更灵活——不是靠 priority 自动选，而是让 LLM 显式指定 `provider/model`，也可以配 fallback 链。

核心好处就一个：**对 LLM 屏蔽 provider 选择**。

```python
# 不用派发 — LLM 需要知道"用哪个"
```

派发层的好处体现在三个场景：

**1. 用户换 provider 时 LLM 无感知。** 用户把配置从 MiniMax 切到豆包，LLM 看到的工具签名不变，不需要重新理解"这个工具参数怎么变了"。
