基于深入对比，以下是 Gateway 与 OpenClaw 的差距分析：

---

## 一、架构质量问题

### 1.1 Pydantic Schema 死代码
`gateway/schemas.py` 定义了 `ReqFrame/ResFrame/EventFrame/ConnectParams` 等模型，但 **从未被使用**。`server.py` 仍用手动 dict 访问 (`raw.get("type")`)，缺少类型校验和 FastAPI 自动文档。

### 1.2 全局单例污染
`gateway/server.py`、`admin/routes/*.py` 都通过模块级全局变量传递 `living`/`agent_manager`，导致无法单进程测试、无法多实例运行。

### 1.3 Peer Rule 泄漏
WS 每连接一次就 `register_peer()`，但 `Router` 没有 `remove_rule()` 方法。断连后死规则永久堆积。

### 1.4 Session ID 断裂
`server_methods.py` 用 `ws-{conn_id[:8]}` 派生 session_id，同一客户端重连得到不同的 session_id，对话连续性丢失。

---

## 二、Gateway RPC 方法缺失

当前 Gateway WS 只有 **3 个方法**（connect / chat.send / chat.abort），对照 OpenClaw 缺少：

| 优先级 | 缺失方法 | 说明 |
|--------|----------|------|
| P1 | `chat.history` | 获取对话历史 |
| P1 | `sessions.list` / `sessions.get` | 会话管理 |
| P2 | `config.get` / `config.patch` | WS 上也能读写配置（现在只在 Admin REST） |
| P2 | `agents.list` / `agents.get` | Agent 信息查询 |
| P2 | `tools.list` | 可用工具列表 |
| P2 | `models.list` | 模型列表 |
| P3 | `sessions.subscribe` | 实时推送会话变更事件 |
| P3 | `channels.status` | 渠道状态 |
| P3 | `skills.status` | 技能状态 |

---

## 三、事件推送缺失

当前只推 `chat.chunk` 和 `session.message`，OpenClaw 还有：

- `sessions.changed` — 会话创建/关闭通知
- `presence` — agent 在线状态变更
- `tick` / `heartbeat` — 服务端主动心跳
- `health` — 健康状态广播
- `shutdown` — 优雅关闭通知

---

## 四、Admin API 不足

- `GET /api/config` 不支持 dot-notation 路径查询 (`config.get("drive.desire.belonging")`)
- 无 config 变更实时推送（SSE）
- Agent 管理缺 `DELETE`、`PATCH`、`start`/`stop`/`restart`
- Session 信息过于简陋（无创建时间、消息数、最后活跃时间）
- 缺 CORS 中间件（浏览器 dashboard 无法直连）
- Admin token 每次请求都重读 YAML 文件，应缓存

---

## 五、安全与可靠性

- Comms Server 无认证、无 TLS，裸 HTTP
- Gateway WS 无连接速率限制、无消息大小限制
- `WSAdapter.send()` 消息发送失败静默丢弃，无背压

---

**长期**：Comms Server 认证/TLS、限流、Admin agent 生命周期管理

**[00:44:26] 🤖助手**

问题就出在 `server.py` 的 WS 连接处理里。

每次 WebSocket 连接进来（第 49-58 行），都生成一个 `conn_id`，然后调用：

```python
_global_router.register_peer(peer_id=conn_id, ...)
```

这个调用在 `Router` 内部做了这件事：

```
self._rules = []                          # 规则列表，装在里面

def register_peer(peer_id):                # 每次连接就 append 一条
    self._rules.append(PeerRule(...))      # _rules 只增不减
```

连接断开时（第 113 行）：
```python
cm.unregister(conn_id)     # ← 清理了 ConnectionManager
# 但 _global_router 里的规则 —— 没人删
```

**Router 类压根没有 `remove_rule()` 方法**。所以每次客户端连一次再断开，就在 `_rules` 列表里留下一条永远匹配不到的垃圾规则。

堆 100 次连接 → 100 条死规则。堆 10000 次 → 10000 条死规则。每次 `route()` 调用都要扫描所有这些死规则做 `_match_score()` —— 纯浪费。

修复很简单：给 Router 加个 `remove_peer(peer_id)`，断连时调用。要我修吗？

**[04:52:29] 🤖助手**

不影响。因为**现在本来就没有会话连续性**。

原因是 session_id 的生成方式：

```python
conn_id = str(uuid.uuid4())                      # 每次连接随机生成
session_id = f"ws-{conn_id[:8]}"                 # session 跟着 conn 走
```

断开再连 → 新的 `conn_id` → 新的 `session_id` → 全新的对话。旧规则匹配的是旧 `conn_id`（UUID），永远不会再出现，所以它是纯垃圾。

**会话连续性是另一个问题**，需要客户端重连时带上之前的 session_id：

```
connect {token, session_id: "ws-abc12345"}  →  服务端恢复这个 session
```

这个问题在我们之前对比 OpenClaw 时就标记为"会话 ID 断裂"的缺陷。两个问题可以一起修：断连删旧规则 + connect 支持恢复 session。

## 3. Message Rendering

### 3.1 User Messages

`UserMessageComponent` extends `MarkdownMessageComponent` (which extends `Container`) and renders user text with a distinct background color (`theme.userBg`) and text color (`theme.userText`).

```typescript
// user-message.ts

  constructor(text: string) {
    super(text, 1, {
      bgColor: (line) => theme.userBg(line),
      color: (line) => theme.userText(line),
    });
  }
}
```

### 3.2 Assistant Messages (Streaming and Final)

`AssistantMessageComponent` uses `HyperlinkMarkdown` (which wraps `Markdown` from pi-tui with OSC 8 hyperlink support). Streaming is handled by the `ChatLog` which tracks `AssistantMessageComponent` instances by `runId`:

- **`startAssistant(text, runId)`**: Creates new streaming component or reuses existing
- **`updateAssistant(text, runId)`**: Updates text in-place (finds existing by runId, creates if missing)
- **`finalizeAssistant(text, runId)`**: Sets final text and removes from streaming map
- **`dropAssistant(runId)`**: Removes a component entirely (for "(no output)" suppression)

### 3.3 System Messages

System messages are simple styled `Text` components with `theme.system()` color:

```typescript
addSystem(text: string) {
  this.append(new Spacer(1));
  this.append(new Text(theme.system(text), 1, 0));
}
```

### 3.4 Streaming Architecture

The `TuiStreamAssembler` (`tui-stream-assembler.ts`) manages the transition from delta events to rendered text:

1. **Delta ingestion**: Each `"delta"` ChatEvent passes through `ingestDelta()` which accumulates thinking and content blocks separately
2. **Thinking/Content separation**: Extracts `text` blocks and `thinking` blocks from structured message content arrays
3. **Boundary drop detection**: When model providers send both streamed partial text AND final content, the assembler detects that the final content is a subset of what was streamed and preserves the streamed version to avoid visual jumps
4. **Finalization**: `finalize()` produces the final display text, falling back through: final composed text -> streamed text -> error message format -> "(no output)"

### 3.5 BTW (Behind-the-Words) Inline Results

BTW events are side-result operations shown inline:
- Displayed as `BTW: <question>` header + assistant-rendered response
- If error: rendered with `theme.error()`
- Dismissed via Enter or Escape
- A global input listener intercepts Enter when a BTW is visible and editor is empty

### Input Pre-processing (`tui.ts`)

A global input listener **deduplicates backspace** keys within an 8ms window to prevent double-delete issues on some terminals.

An additional global listener intercepts `Enter` when a BTW inline message is visible and the editor is empty, dismissing the BTW.

### Redundant Ctrl+C Handling

Both `editor.onCtrlC` and a `process.on("SIGINT")` handler call `handleCtrlC()`. The Ctrl+C logic implements a **three-stage exit pattern**:
1. If input has text -> clear input (first press)
2. Second press within 1 second -> warn "press ctrl+c again to exit"
3. Third press within 1 second -> exit

This works regardless of whether the signal is from the editor (Ctrl+C key) or SIGINT signal.

### Command Parsing (`tui-submit.ts`)

The submit handler has a three-way dispatch:
1. **`!` prefix** (non-empty): Execute local shell command (`runLocalShellLine()`)
2. **`/` prefix**: Parse as slash command (`handleCommand()`)
3. **Anything else**: Send as chat message (`sendMessage()`)

### Paste Burst Coalescing (`tui-submit.ts`)

`createSubmitBurstCoalescer` addresses a problem where some terminal emulators (Windows Git Bash, macOS iTerm2/Apple Terminal) emit multiline pastes as rapid single-line submits. It buffers submits within a 50ms window, joining them with `\n` before sending as a single message. This is only enabled for specific platforms/terminals.

### State Flow

1. **Gateway connection** -> `onConnected` callback -> `refreshAgents()` -> `loadHistory()` -> update header/footer
2. **User sends message** -> `sendMessage()` -> `activityStatus = "sending"` -> `activityStatus = "waiting"` -> render status loader
3. **Chat event arrives** -> `handleChatEvent()` -> updates streaming assistant -> `activityStatus = "streaming"` -> on final: `activityStatus = "idle"`, `refreshSessionInfo()`, maybe `loadHistory()`
4. **Agent event (tool/lifecycle)** -> `handleAgentEvent()` -> updates tool components, activity status transitions
5. **Session switch** -> `setSession()` -> clears all local run IDs -> `loadHistory()` -> refresh

### Run ID Tracking

The TUI distinguishes **local runs** (originating from this TUI client) from **external runs** (from other clients/channels) using `Set<string>` collections:
- `localRunIds`: Tracks chat runs started by this TUI (max 200 entries, oldest evicted)
- `localBtwRunIds`: Tracks BTW runs started by this TUI (same eviction policy)

This distinction affects history reload logic: external runs trigger immediate history reload, while local runs defer reload if a newer run is active.

### Event Handler State

The event handler (`tui-event-handlers.ts`) maintains its own maps:
- `finalizedRuns: Map<string, number>`: Completed runs with timestamps (pruned at 200 entries)
- `sessionRuns: Map<string, number>`: Active/streaming runs with timestamps
- `streamAssembler: TuiStreamAssembler`: Per-session, reset on session switch

These maps enable filtering duplicate/out-of-order events and detecting concurrent active runs.

## 9. Interesting Patterns and Design Decisions

### 9.1 Pure Synchronous UI Architecture

Despite the asynchronous nature of WebSocket communication, the TUI itself is **purely synchronous** in its rendering pipeline. The pi-tui framework uses a retained-mode scene graph with explicit `tui.requestRender()` calls. All state mutations update the scene graph components directly; rendering happens on-demand.

### 9.2 Overlay System

Overlays (selectors, settings) are managed through a minimal abstraction in `tui-overlays.ts`:

```typescript
export function createOverlayHandlers(host: OverlayHost, fallbackFocus: Component) {
  const openOverlay = (component: Component) => host.showOverlay(component);
  const closeOverlay = () => {
    if (host.hasOverlay()) host.hideOverlay();
    else host.setFocus(fallbackFocus);
  };
  return { openOverlay, closeOverlay };
}
```

The `closeOverlay` function has a dual purpose: if an overlay is active, it dismisses it; otherwise, it returns focus to the editor. This is used as the "onCancel" handler for selectors.

### 9.3 Factory Function Pattern

Rather than class hierarchies, most business logic uses factory functions that accept dependency objects and return closures:

```typescript
createCommandHandlers(context) -> { handleCommand, sendMessage, ... }
createEventHandlers(context)  -> { handleChatEvent, handleAgentEvent, handleBtwEvent }
createSessionActions(context) -> { refreshAgents, loadHistory, setSession, abortActive }
createOverlayHandlers(host, fallback) -> { openOverlay, closeOverlay }
createEditorSubmitHandler(params) -> submitFn
createLocalShellRunner(deps) -> { runLocalShellLine }
```

This gives explicit dependency wiring, easy testability (all dependencies can be mocked), and clear interfaces for each subsystem.

### 9.4 Serialized Session Info Refresh

`refreshSessionInfo()` uses a promise chaining pattern to serialize concurrent refresh requests:

```typescript
let refreshSessionInfoPromise: Promise<void> = Promise.resolve();
const refreshSessionInfo = async () => {
  refreshSessionInfoPromise = refreshSessionInfoPromise.then(
    runRefreshSessionInfo,
    runRefreshSessionInfo,
  );
  await refreshSessionInfoPromise;
};
```

Both the fulfillment and rejection handlers are `runRefreshSessionInfo`, so a failed refresh never breaks the chain (subsequent refreshes still work). If 10 rapid calls arrive, they will execute sequentially.

### 9.5 Timestamp-Based Staleness Detection

Session info updates are gated by `updatedAt` timestamps to prevent stale data from overwriting fresher state. The `applySessionInfo()` function checks that the incoming entry's `updatedAt` is newer than the current stored `updatedAt` (unless `force: true`). Session switches clear `updatedAt` to `null` to ensure the next refresh always applies.

### 9.6 Text Sanitization Pipeline

`sanitizeRenderableText()` (`tui-formatters.ts`) applies multiple transformations in sequence:
1. **ANSI stripping**: Removes existing ANSI escape sequences
2. **Control character stripping**: Removes ASCII control chars (0x00-0x1F, except tab/newline/CR) and C1 control chars (0x7F-0x9F)
3. **Binary line redaction**: Lines with >=12 replacement characters covering >=50% of the line are replaced with `[binary data omitted]`
4. **Long token splitting**: Non-URL, non-filepath tokens >=33 chars are chunked with spaces to prevent layout overflow, but URL/path/credential-like tokens are preserved
5. **RTL isolation**: Lines containing RTL script characters are wrapped in Unicode bidi isolation characters

### 9.7 OSC 8 Terminal Hyperlinks

The `HyperlinkMarkdown` component (`components/hyperlink-markdown.ts`) wraps pi-tui's `Markdown` and adds OSC 8 terminal escape sequences to rendered output. OSC 8 is a terminal protocol that makes arbitrary text behave as clickable hyperlinks. The implementation:
1. Extracts URLs from raw markdown using regex (both `[text](url)` markdown links and bare URLs)
2. After Markdown rendering (which strips the original URL syntax), finds URL-like substrings in the visible text
3. Matches fragments to known URLs (exact match > prefix > superstring)
4. Handles **cross-line URL splits** caused by pi-tui's word wrapping, using a `pending` state that bridges contiguous URL fragments across lines
5. Inserts OSC 8 open/close sequences at URL range boundaries, carefully passing through ANSI SGR sequences

### 9.8 Smart Session Selection with Fuzzy Search

The `SearchableSelectList` (`components/searchable-select-list.ts`) implements a 4-tier filtering system:
1. **Tier 0**: Exact substring match in label (sorted by position)
2. **Tier 1**: Word-boundary prefix match in label
3. **Tier 2**: Exact substring match in description
4. **Tier 3**: Fuzzy match (lower score = better)

Results are combined tier-by-tier with fuzzy matches scored and sorted. The component also supports:
- **Match highlighting**: Multi-token query highlighting in both label and description
- **Description column**: If terminal width allows (>=40 cols), description is shown in a right-aligned column
- **Vim navigation**: `j`/`k` for navigation when no search query is active
- **Ctrl+P/Ctrl+N**: Alternative navigation

### 9.9 Local Shell Execution with Explicit Consent

`tui-local-shell.ts` (`createLocalShellRunner`) requires explicit user consent before any local shell command runs. The flow:
1. First `!command` triggers a Yes/No overlay prompt
2. User must select "Yes" to enable for the session
3. Once enabled, all subsequent `!` commands run immediately
4. Commands run via `child_process.spawn` with `shell: true` in the current working directory
5. Output is capped at 40,000 characters (rolling window from end)
6. Exit code/signal is displayed

### 9.10 BTW (Behind-the-Words) Integration

BTW is a pattern for "side computation" results. In the TUI:
- BTW events arrive on `chat.side_result` channel
- Displayed inline in the chat log (not as a separate overlay)
- Can be triggered by `/btw:` prefix in messages
- Dismissed by Enter or Escape
- The BTW component reuses `AssistantMessageComponent` for result rendering, so it gets the same markdown/formatting treatment as regular assistant messages

### 9.11 Lifecycle State Machine

The activity status transitions through states:
- `idle` -> `sending` -> `waiting` -> `streaming` -> `idle`
- Any state -> `aborted` or `error` -> `idle`
- `idle` -> `running` (lifecycle start) -> `idle` (lifecycle end)

The status renderer (`renderStatus()`) maintains two separate timer systems:
- **Status timer** (1s): For non-waiting busy states (shows elapsed time)
- **Waiting timer** (120ms): For waiting state with animated shimmer text

These timers are started/stopped on state transitions to avoid unnecessary intervals.

### 9.12 Auto-Send Message Feature

When launched with `--message`, the TUI connects, loads history, then auto-sends the message once. This is tracked by `autoMessageSent` (set to true after first send) and only fires after history is loaded (to avoid sending before the session context is established).

### 9.13 Session Key Isolation for `/new`

When creating a new session, the TUI generates a unique `tui-{UUID}` key via `randomUUID()` to isolate the session from other connected TUI clients sharing the original session key. This prevents the new session from broadcasting to other clients.

## Summary of Key Design Themes

1. **Explicit dependency injection**: All subsystems receive their dependencies as constructor parameters or factory arguments, making the entire TUI highly testable and composable.

2. **Mutable state with getter/setter interface**: State is centralized in `runTui()` scope, exposed through a `TuiStateAccess` interface with get/set accessors. This provides clear ownership while allowing subsystems to read/write state.

3. **Retained-mode UI**: The component tree is built once and mutated in place. Components track their own state and re-render on demand via `tui.requestRender()`.

4. **Gateway-as-backend**: The TUI is a thin client; all LLM logic, session management, tool execution happens server-side. The TUI only handles display, input routing, and session/agent selection.

5. **Graceful degradation**: From "(no output)" placeholders to binary data redaction to tool output collapsing, the TUI handles edge cases without crashing or corrupting the display.

两者的核心区别：

**CLI (`python -m xiaomei_brain run xiaomei --cli`)**
- 基于 `readline` 的简单终端交互，没有 UI 框架
- 直接在同一进程中启动 `ConsciousLiving`（意识系统 + Drive + Purpose + 记忆全栈）

**TUI (`python -m xiaomei_brain tui`)**
- 基于 `prompt_toolkit` 的专业终端 UI 框架（全屏、布局系统、key binding）
- 通过 **WebSocket** 远程连接到已运行的 Gateway（端口 19766），架构上是客户端/服务端分离

  - **TUI 内部命令**（`/clear`、`/theme`、`/help` 等）→ 本地处理
  - **Gateway 透传命令**（`/intent`、`/drive`、`/flame` 等）→ 作为聊天消息发给 Gateway，由服务端 Agent 处理
- 用 prompt_toolkit 样式系统着色（`Style.from_dict()`），不依赖终端 ANSI

简单说：**CLI = 本地全栈，自包含运行；TUI = 瘦客户端，连远程 Gateway**。
