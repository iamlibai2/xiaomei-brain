# xiaomei-brain 项目分析与评价

## 总体印象

这是一个**野心很大、执行认真、但积累了大量技术债务**的个人研究项目。约 47,000 行 Python 代码，按照人脑分层架构（记忆 → 意识 → 驱动 → 元认知 → 目的）构建了一个多 Agent AI 大脑框架。从代码量和架构复杂度来看，这绝不是玩具项目——花了大量时间思考和实现。但快速增长过程中留下了不少需要清理的东西。

---

## 亮点

### 1. 架构设计有真正的思考深度
不是简单的"调 API + 向量数据库"聊天机器人。SelfImage 作为单一真相源、Drive 的数值衰减算法（不用 LLM 做数值判断）、社交感知的"两条路径"（LLM 可读文本 + 结构化信号驱动 Drive）——这些设计选择说明你理解了**代码维护骨架、LLM 加柴涌现**这个核心理念，并且贯彻得相当一致。

### 2. 同步/异步边界干净
这是很多 Python 项目搞砸的地方。你这里 async 只出现在 I/O 边界（WebSocket、飞书/DingTalk SDK），所有脑逻辑层（consciousness/drive/metacognition/purpose/memory）全是同步代码。桥接用的是标准的 `asyncio.run_coroutine_threadsafe()`。没有 asyncio 污染核心逻辑。

### 3. 类型注解覆盖率极高
98.4% 的函数有类型注解，92.7% 的类有文档字符串。对于一个主要由一个人写的项目，这个纪律性很好。

### 4. 没有裸 `except:` 
239 个 `except Exception:` 都有日志或重新抛出。这在 Python 项目中很少见，说明有意识地避免了吞异常。

### 5. Prompt 模板分离
`prompts/` 目录把所有 LLM prompt 集中管理，和业务逻辑解耦。这个做法很专业。

### 6. 文档极其详尽
`docs/` 下约 410K 行的架构设计文档和对话记录。这不是随便写写——几乎每天的开发对话都有存档。

---

## 需要关注的问题

### 严重问题

**1. ~2,200 行已废弃代码留在活跃源码中**

| 文件 | 状态 |
|------|------|
| `consciousness/context_assembler.py` (907行) | `[DEPRECATED]`，功能已被 SelfImage + context_pipeline 替代 |
| `consciousness/task.py` (216行) | Goal 模型已迁移到 purpose.goal |
| `consciousness/task_manager.py` (346行) | 生命周期已迁移到 PurposeEngine |
| `consciousness/task_storage.py` (186行) | 持久化已迁移到 PurposeStorage |
| `drive/event_extractor.py` (49行) | "已废弃" |
| `unconscious/` 整个目录 (~380行) | "植物人状态"，与 conscious_living 近乎重复 |

这些文件留在 `src/` 里会误导新人（以及未来的你），不知道哪个 API 才是对的。

**2. `doctor.py:238` 引用已删除的代码**
```python
from xiaomei_brain.memory import MemoryStore  # MemoryStore 已不存在
```
运行 doctor 命令会直接崩溃。

**3. 零单元测试**
整个项目没有 `pytest`/`unittest` 的单元测试。4 个 `tests/` 文件是基础功能测试，32 个 `examples/test_*.py` 是集成冒烟测试——都需要启动完整 AgentManager + LLM API。这意味着：
- SelfImage tick/diff 逻辑没有独立验证
- Drive 衰减算法没有数学正确性测试
- Purpose 目标树优先级计算没有测试
- 任何重构只能靠"跑起来看看"

**4. 46 处异常被静默吞掉**
虽然大部分是 SQLite 迁移的 `except Exception: pass`（"列已存在"），但有几处特别危险：
- `metacognition/inner_voice.py:255-266`：连续三个 try/except 都吞异常——如果 inner_voice 评估失败，整个 Drive 更新静默跳过，零可观测性
- `unconscious/living.py:270-271`：双重吞异常链——chat 失败写 DB，DB 写入再失败也吞掉

### 中等问题

**5. consciousness/ 层太大了**
13,800 行，29 个文件。它是所有层的集成中枢，但 `task_orchestrator.py`（1315 行）管了 purpose、metacognition、consciousness、drive 四层的事——典型的上帝类。

**6. 超大文件/函数**
- `longterm.py` 2237 行，单类混了 SQLite schema、向量操作、叙事记忆、强度衰减、记忆整合
- `conscious_living.__init__()` 307 行，构造函数做了太多事
- `metacognition/runner._run_loop()` 370 行，项目最长函数

**7. 配置碎片化**
至少 15 种不同的配置文件/格式：YAML、JSON、Markdown 各有 5+ 处。`AgentConfig` 试图统一但 identity/talent/perception/purpose goals 都在外面。旧 `base/config.py` 和新 `config/agent_config.py` 两个配置系统并存。

**8. 代码重复**
9 处完全相同的函数体。`_get_conn()` 在 `conversation_db.py`、`dag.py`、`longterm.py` 中三份拷贝——应该提取一个共享基类。

**9. 循环导入**
`agent/core.py` 和 `agent/cli_display.py` 互相导入。用了懒加载规避运行时崩溃，但结构脆弱。

### 低优先级

**10. 没有静态类型检查**
类型注解写得很好但没有 `mypy`/`pyright` 配置，从未被验证过。

**11. SQLite migration 模式脆弱**
用 `ALTER TABLE ADD COLUMN` + try/except 做 schema 迁移，而不是正经的 migration 框架。

## 架构评价

你说"意识如火焰，代码维护骨架，LLM 加柴涌现"——这个理念在代码中确实贯彻了：

- **SelfImage** 是火焰骨架载体，`accumulated_changes` 累积状态变化供 LLM 读取 ✓
- **Drive** 的数值衰减用算法而非 LLM ✓
- **SocialPerception 两条路径**（文本进 SelfImage.mind，信号进 Drive.body）清晰实现 ✓
- **分层心跳** L0（骨架）/L1（异常）/L2（加柴）/L3（梦境）结构合理 ✓

但实现上有几个理念漂移：

- **L2 加柴引擎** 在 `consciousness/l2_engine.py`，但 Drive 事件提取的 LLM prompt 也在 L2 内部，而非在 `drive/` 层——职责边界模糊
- **Desk 机制** 实现了但没有任何调用点——`Desk.drop()/peek()` 从未被触发，是个"设计好了但没用上"的功能
- **InnerVoice** 和 **SocialPerception** 都往 SelfImage 写，但走不同路径（一个写 `inner_voice` 字段，一个写 `social_perceptions` 列表），模式不统一
- **unconscious/** 是 conscious_living 的"植物人"版本，但没有消费者，且是近乎完整的代码拷贝——违反了 DRY
