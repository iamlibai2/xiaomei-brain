## 四、核心流程

### 4.1 主循环流程（ConsciousLiving.run()）

```
启动 → DORMANT → WAKING → _on_wake() → AWAKE
                                    ↓
                              _loop_awake()
                              │  _heartbeat() → Consciousness.tick()
                              │  _wait_message(timeout=1s)
                              │  收到消息 → _handle_message()
                              │  空闲>5min → IDLE
                              │  空闲>30min → SLEEPING
                              ↓
                         _loop_idle() / _loop_sleeping()
                              │  _heartbeat() → L2加柴 / L3梦境
                              │  收到消息 → 唤醒回AWAKE
                              ↓
                         _loop_dreaming()
                              │  tick_L3() → 深度意识报告
                              │  完成后 → 回SLEEPING
```

### 4.2 消息处理流程（_handle_message）

```
消息 → 清洗 → 命令检测(/cmd) → 是? → 执行命令
                          ↓ 否
              意图模式(intask)? → 是 → _analyze_intent() → _handle_task_intent()
                          ↓ 否
              "继续"检测 → 匹配活跃目标 → 延续任务
                          ↓ 否
              确认状态? → _handle_confirmation()
                          ↓ 否
              意图模式? → _analyze_intent() → _run_chat(msg, intent_context)
                          ↓ 否(聊天模式)
              _run_chat(msg, intent_context)
```

### 4.3 对话流程（_run_chat）

```
_run_chat(msg, intent_context)
  │
  ├─ build_context() → 决定模式 → 组装system+DAG+长期记忆
  ├─ agent.stream() → ReAct循环(工具调用)
  ├─ 解析PROGRESS标签 → 更新目标进度
  ├─ 子目标完成? → 自动推进到下一个
  ├─ 全部完成? → _complete_task() → 知识提取
  └─ 更新意识系统 + 更新最近对话
```

### 4.4 上下文组装流程（build_context）

```
build_context(agent, user_input, consciousness_state, intent_context)
  │
  ├─ 1. 记录用户消息到ConversationDB
  ├─ 2. 添加到agent.messages
  ├─ 3. determine_mode() → flow/daily/reflect
  ├─ 4. _auto_compact() → DAG自动压缩(超75% token)
  ├─ 5. ContextAssembler.assemble() → system + DAG摘要 + 长期记忆
  ├─ 6. 注入intent_context到system prompt
  ├─ 7. filter_compressed_messages() → 过滤已摘要消息
  ├─ 8. token裁剪(倒序保留最新)
  └─ 9. 返回 [system_msg] + agent.messages
```

## 六、关键设计决策

| 决策 | 原因 | 代码体现 |
|------|------|----------|
| 纯同步架构 | chat/embedding都是同步阻塞，asyncio会导致input()卡死 | `queue.Queue`替代`asyncio.Queue` |
| 火焰骨架 | 不假装涌现意识，代码只维护状态，真正意识来自LLM | `tick()`返回`FlameState`，不返回"假装的意识" |
| 1个system message | MiniMax API限制(error 2013) | DAG摘要+长期记忆+intent_context全合并到system |
| L2合并prompt | 一次LLM调用同时产出意识+事件，减少延迟成本 | `---EVENTS---`分隔符 |
| 第一人称记忆 | 长期记忆是小美视角 | "我..."="小美"，"用户..."=用户信息 |
| 任务独立认知实体 | Task不是Goal的附属，有自己的认知日志 | `Task.cognitive_log`，暂停/恢复 |
| 规则驱动动作 | 不用LLM判断"要不要做"，用规则+冷却 | `ActionDispatcher`遍历`RULES` |
| context_guard | 防止上下文超限导致API报错 | `ContextGuard`包装`LLMClient`，自动裁剪 |

## 四、核心流程

### 4.1 主循环流程（ConsciousLiving.run()）

```
启动 → DORMANT → WAKING → _on_wake() → AWAKE
                                    ↓
                              _loop_awake()
                              │  _heartbeat() → Consciousness.tick()
                              │  _wait_message(timeout=1s)
                              │  收到消息 → _handle_message()
                              │  空闲>5min → IDLE
                              │  空闲>30min → SLEEPING
                              ↓
                         _loop_idle() / _loop_sleeping()
                              │  _heartbeat() → L2加柴 / L3梦境
                              │  收到消息 → 唤醒回AWAKE
                              ↓
                         _loop_dreaming()
                              │  tick_L3() → 深度意识报告
                              │  完成后 → 回SLEEPING
```

### 4.2 消息处理流程（_handle_message）

```
消息 → 清洗 → 命令检测(/cmd) → 是? → 执行命令
                          ↓ 否
              意图模式(intask)? → 是 → _analyze_intent() → _handle_task_intent()
                          ↓ 否
              "继续"检测 → 匹配活跃目标 → 延续任务
                          ↓ 否
              确认状态? → _handle_confirmation()
                          ↓ 否
              意图模式? → _analyze_intent() → _run_chat(msg, intent_context)
                          ↓ 否(聊天模式)
              _run_chat(msg, intent_context)
```

### 4.3 对话流程（_run_chat）

```
_run_chat(msg, intent_context)
  │
  ├─ build_context() → 决定模式 → 组装system+DAG+长期记忆
  ├─ agent.stream() → ReAct循环(工具调用)
  ├─ 解析PROGRESS标签 → 更新目标进度
  ├─ 子目标完成? → 自动推进到下一个
  ├─ 全部完成? → _complete_task() → 知识提取
  └─ 更新意识系统 + 更新最近对话
```

### 4.4 上下文组装流程（build_context）

```
build_context(agent, user_input, consciousness_state, intent_context)
  │
  ├─ 1. 记录用户消息到ConversationDB
  ├─ 2. 添加到agent.messages
  ├─ 3. determine_mode() → flow/daily/reflect
  ├─ 4. _auto_compact() → DAG自动压缩(超75% token)
  ├─ 5. ContextAssembler.assemble() → system + DAG摘要 + 长期记忆
  ├─ 6. 注入intent_context到system prompt
  ├─ 7. filter_compressed_messages() → 过滤已摘要消息
  ├─ 8. token裁剪(倒序保留最新)
  └─ 9. 返回 [system_msg] + agent.messages
```

## 六、关键设计决策

| 决策 | 原因 | 代码体现 |
|------|------|----------|
| 纯同步架构 | chat/embedding都是同步阻塞，asyncio会导致input()卡死 | `queue.Queue`替代`asyncio.Queue` |
| 火焰骨架 | 不假装涌现意识，代码只维护状态，真正意识来自LLM | `tick()`返回`FlameState`，不返回"假装的意识" |
| 1个system message | MiniMax API限制(error 2013) | DAG摘要+长期记忆+intent_context全合并到system |
| L2合并prompt | 一次LLM调用同时产出意识+事件，减少延迟成本 | `---EVENTS---`分隔符 |
| 第一人称记忆 | 长期记忆是小美视角 | "我..."="小美"，"用户..."=用户信息 |
| 任务独立认知实体 | Task不是Goal的附属，有自己的认知日志 | `Task.cognitive_log`，暂停/恢复 |
| 规则驱动动作 | 不用LLM判断"要不要做"，用规则+冷却 | `ActionDispatcher`遍历`RULES` |
| context_guard | 防止上下文超限导致API报错 | `ContextGuard`包装`LLMClient`，自动裁剪 |

```
ActionDispatcher.tick(SelfImage) → 规则匹配 → ActionItem队列
    ↓
process_queue() → _do_proactive/_do_trigger_l3/_do_tool/_do_notify
    ↓
ContextAssembler.assemble() → build_context()
    ↓
Agent.stream() → ReAct循环 → LLM → tool_calls → 结果 → 累积
    ↓
MemoryExtractor.extract_memory_block() → LongTermMemory.store()
    ↓
MemoryExtractor.extract_think_block() → thoughts表
    ↓
Drive.tick_minute() / tick_hour() → 衰减/恢复
    ↓
（如在Sleeping）→ DreamProcessor → 梦境记忆强化
```
