换一种更可靠的方案 — 直接在 `input()` 调用后用 `repr()` 验证字节，处理后二次校验：



**助手回答（2026-04-21 00:40:36）**

搜索结果没有直接解决方案。分析了一下，这是 **WSL2 终端 + 输入法** 的编码异常问题。

DEBUG 输出 `'晚上第好啊'` 里多了一个"第"字，是因为 WSL 终端在输入过程中产生了 `\udce5` `\udcae` `\udce5` `\udca5` `\udce4` `\udcb8` 等孤立高位字节（Trail bytes）。这些字节落在 `\udc80~\udcff` 范围内，会被过滤掉，但"第"字的字节序列恰好避开了这个范围。

最终修复逻辑：

```python
# 先构建 buffer（退格处理）
buf: list[str] = []
for ch in raw_input:
    if ch == "\x08" or ch == "\x7f":
        if buf: buf.pop()
    elif ord(ch) < 0x20 and ch not in ("\t", "\n", "\r"):
        pass
    else:
        buf.append(ch)

# 后处理：移除 UTF-16LE 残留的孤立 Trail 字节（\udc80~\udcff）
user_input = "".join(c for c in "".join(buf) if not ("\udc80" <= c <= "\udcff")).strip()
```

这样即使 WSL 终端在输入过程中混入异常字节，也不会进入最终的 `user_input`。

1. `_on_wake()` 是在 `run()` 循环里同步调用的，加 LLM 调用会阻塞主循环（等 3-6 秒才出问候语）
2. CLI `--cli` 模式是单线程，处理完问候才能收下一条消息，不能等

要 LLM 生成的话需要改成**异步触发**：
- proactive 触发时，把上下文丢给后台线程
- 后台线程调 LLM 生成消息
- 生成完通过 `on_proactive` 推送到 CLI

这属于"主动输出架构"改造，要改的话需要较大的架构调整。先维持当前的简单拼接，后续可以加异步版本。

用户输入 → LLM生成回复（第一次）
       ↓
memory_extractor.extract_every_turn() → LLM再判断一次（第二次）
```

现在是单次调用：

```
用户输入 + <MEMORY> → LLM生成回复 + 记忆决策（一次完成）
```

具体改动：

- `MEMORY_DECISION_PROMPT` 加在 user message 末尾，让 LLM 回复末尾带上 `<MEMORY>ACTION|...</MEMORY>`
- 回复到达后，从内容里剥离 `<MEMORY>` 块，交给 `extractor._execute_actions()` 执行
- `agent_manager.py` 里删掉了单独的 `extract_every_turn()` 调用

所以流程上是一次 LLM 调用完成两件事：生成回复 + 判断记忆。

## 50. 之前extractor 是如何处理 add  update delte 等操作的

**🟥 用户提问（2026-04-21 10:55:24）**

> 之前extractor 是如何处理 add  update delte 等操作的

**助手回答（2026-04-21 10:56:03）**

`_execute_actions` 的处理逻辑：

**ADD**：直接 `ltm.store()` 存入

**UPDATE / MERGE / DELETE**：流程相同
1. 用 `ltm.recall(content)` 找到最相似的已有记忆
2. 没找到 → 降级为 ADD
3. 找到了：
   - **UPDATE**：调 `_llm_update_content()` 提炼核心事实，丢弃"确认/纠正"类元信息 → `ltm.update_content()`
   - **MERGE**：调 `_llm_merge()` 合并两条记忆 → `ltm.update_content()`
   - **DELETE**：`ltm.soft_delete()`

**关键点**：
- UPDATE/MERGE/DELETE 都需要先 recall 找到目标记忆
- `content_to_id` 字典记录新创建的 ADD 记忆 id，供 RELATES 第二阶段用
- 每步都 `save_history()` 保存操作历史

现在合并后传进来的 `memory_block` 格式和原来一样，所以这套逻辑没变，只是调用入口从 `extract_every_turn()` 里的 LLM 调用改成了直接解析 LLM 生成的 `<MEMORY>` 块。

## 66. 现在从启动living 到一轮对话结束，是怎么样的流程，具体解释一下

**🟥 用户提问（2026-04-21 12:11:38）**

> 现在从启动living 到一轮对话结束，是怎么样的流程，具体解释一下

**助手回答（2026-04-21 12:12:28）**

完整流程分两阶段：

---

## 启动阶段

```
__main__.cmd_run(agent_id, attach_cli=True)
  ↓
AgentManager.build_agent(agent_id)       # 构建 AgentInstance（SelfModel/Tools/LTM/Extractor等）
  ↓
AgentLiving(instance)                    # 创建 living 实例
  ↓
living._on_wake()                      # 手动触发（CLI 模式不走 living.run() 循环）
  ↓
proactive.check(WAKE) → LLM 生成问候语
  ↓
[阶段] WAKING → AWAKE，准备就绪
```

---

## 一轮对话阶段（用户输入 → 回复）

```
1. 用户输入 → input("You: ")
2. 清理 UTF-16 Trail 字节
3. living.put_message(user_input, source="cli")  # 放入队列
4. living._handle_message(msg)                   # 从队列取出处理
   ↓
   agent.chat(user_input)
   ↓
   Agent.stream(user_input)          # core.py
   ├─ conversation_db.log(user)      # 记录用户消息
   ├─ context_assembler.assemble()  # 组装上下文（system + DAG + 长期记忆 + fresh tail）
   ├─ system_prompt += MEMORY_DECISION_PROMPT  # 追加记忆决策格式指令
   ├─ _call_llm(messages)           # 发给 LLM（一次调用同时返回回复 + MEMORY block）
   │   ├─ 无工具调用 → response.content 包含 "回复文本\n<MEMORY>\nADD|...\n</MEMORY>"
   │   └─ 有工具调用 → ReAct 循环（工具调用 → 结果 → 再次 LLM → ... → 最终回复）
   ├─ extract_memory_block(content) → 剥离 MEMORY block，得到 clean_content
   ├─ memory_extractor.execute_block(block, user_id) → 解析 ADD/UPDATE/NOOP，执行
   ├─ conversation_db.log(assistant, clean_content)  # 存干净回复（无 MEMORY 标签）
   └─ yield from stream_chunks      # 流式输出
   ↓
   living._handle_message 输出到 CLI
```

---

## 关键点

1. **一次 LLM 调用**同时完成回复生成 + 记忆决策（`<MEMORY>` 块）
