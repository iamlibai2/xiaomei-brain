## 分析：SelfImage 模块的职责混乱

### 当前实际 vs 声称

代码说 SelfImage 有 7 个"干净模块"，但实际是这样的：

```
SelfMind（名字叫"认知状态"）
├─ 代理 Purpose ── primary_goal, goal_progress, active_goal_count
├─ 记忆统计 ────── memory_count, memory_count_history, recent_memory_summaries
├─ 当前想法 ────── inner_thought, inner_thought_history
├─ 社交感知 ────── social_perceptions
├─ 自我反思 ────── inner_voice
└─ 目标历史 ────── goal_progress_history

SelfMemorySlot（名字叫"记忆片段容器"）
├─ 7种记忆片段 ─── narratives, dag_summaries, important_memories...
├─ 项目认知 ────── project_map
├─ 经验记忆 ────── experience
├─ 执行记录 ────── pace_reflections
└─ memory_count ─── 和 SelfMind.memory_count 同名不同义
```

问题有三层：

| 问题 | 现状 | 后果 |
|------|------|------|
| **职责混淆** | SelfMind 管 6 类不相关数据 | 名字叫"认知"但实际是杂物箱 |
| **归属错误** | project_map 放 MemorySlot | 是"认知模型"不是"记忆片段" |
| | social_perceptions 放 Mind | 是"感知"不是"认知" |
| | memory_count 放 Mind | 是"记忆统计"不是"认知" |
| **同名不同义** | `SelfMind.memory_count` = LTM 总数 | 对 L1 读 `si.mind.memory_count` 是哪个？ |
| | `SelfMemorySlot.memory_count` = 窗口片段数 | |

### 按语义归位

每个数据项应该按"它回答什么问题"归类：

| 数据 | 回答的问题 | 应该属于 |
|------|-----------|----------|
| `primary_goal`, `goal_progress` | 我在做什么？ | **Mind**（认知） |
| `inner_thought` | 我刚才想了什么？ | **Mind**（认知） |
| `inner_voice` | 我做得怎么样？ | **Mind**（自我反思也是认知） |
| `project_map` | 项目长什么样？ | **Mind**（认知模型） |
| `social_perceptions` | 用户状态变了没？ | **Perception**（社交感知） |
| `memory_count`（总数） | 我记了多少东西？ | **Memory**（记忆统计） |
| `memory_count`（窗口） | 现在能想起多少？ | **Memory**（窗口大小） |
| `narratives`, `dag_summaries`... | 具体想起什么？ | **Memory**（记忆内容） |
| `pace_reflections` | 刚才执行得顺吗？ | **Memory**（执行记忆） |

### 整理方案

```
SelfMind                         SelfMemorySlot
├─ [目标]  primary_goal          ├─ [7种记忆]  narratives, dag_summaries

└─ [社交]  social_perceptions  ← 从 Mind 搬来
```

### 当前架构总览

SelfImage 是 7 模块的意识汇聚中心，数据流如下：

### 发现的问题

#### 1. `Being.growth_log` 与 `SelfHistory.growth_events` 重复

两个模块各自维护"生长记录"，格式不一致：
- `Being.growth_log`: `list[str]`（纯文本列表）
- `SelfHistory.growth_events`: `list[dict]`（`[{date, content}]`）

`inject_consciousness()` 读取 `b.growth_log`，但 `SelfImage.add_growth()` 写入的是 `SelfHistory.growth_events`。**两处数据不同步**，`add_growth()` 添加的记录不会出现在 `inject_consciousness()` 输出中。

#### 2. SelfMemory 字段过多且职责混乱（11个字段）

```
记忆存储类：narratives, internal_narratives, dag_summaries, important_memories,

```

`project_map` 和 `experience` 是"我理解什么"（认知），不是"我记住了什么"（记忆）。`pace_reflections` 是"我做得怎么样"（元认知），也与记忆存储无关。

#### 3. SelfBody 代理层过于冗长

13 个 `@property` 全部遵循同一模式：
```python
@property
def energy(self) -> float:
    if self._drive: return self._drive.energy.level
    return 0.8
```

`to_dict()` 用 `_safe(lambda: self.xxx)` 包装每个属性访问，防御性过强。`from_dict()` 只恢复 `attention`，其余 14 个字段存了但不恢复（等 Drive 实时值）。

#### 4. SelfIntent 双队列机制模糊

- `type/description/reason/urgency` — 当前活跃意图
- `intent_buffer: list[str]` — 待执行意图队列（只存类型字符串）

#### 5. SelfPerception 职责混杂

**关键特点：输入驱动的动态检索**

```
assemble(user_input, max_tokens, mode, session_id, ...)
```

1. **拿 user_input 做语义召回** — `_recall_memories()` 用用户消息搜索长期记忆，`_recall_relation_chain()` 搜关联链
2. **token 预算控制** — `remaining` 整数跟踪，每加一段内容就扣减，超了就停
3. **动态决定检索内容** — `if remaining > 200` 才加记忆；`if self.longterm and remaining > 300` 才加关联链
4. **模式判定看输入** — `determine_mode()` 用能量/欲望/意图/关键词联合判断
5. **Procedure 匹配** — 按 user_input 匹配可用工具注入 system prompt

## inject_consciousness（当前）

**关键特点：状态驱动的静态渲染**

```
inject_consciousness(mode)
```

1. **无 user_input 参数** — 不感知用户说了什么
2. **无 token 预算** — 直接渲染所有有内容的 section
3. **内容来源是预填充的 SelfMemory** — `refresh_memory_window()` 在调用前已经把记忆塞进去了，inject 只是照单渲染
4. **模式只控制 section 顺序/取舍，不控制内容** — `_assemble_daily` vs `_assemble_task` 只是排列组合不同
5. **无动态检索** — 无 user_input → 无语义搜索 → 无 procedure 匹配

## 本质区别

| | ContextAssembler | inject_consciousness |
|---|---|---|
| 检索触发 | 输入相关 | 输入无关 |
| 内容筛选 | token 预算动态裁剪 | 全部渲染 |
| 记忆召回 | inline（assemble 时实时查） | 预处理（refresh_memory_window） |
| procedure | 有（按输入匹配） | 无 |
| 返回类型 | `list[dict]`（多消息） | `str`（纯文本） |

- 无法根据用户话题动态调整记忆内容

- 无法做 procedure 匹配
