## 一个意识体需要什么样的记忆？

人类记忆不是一张表，但有一个共同特征：**时间线上的自我连续性**。

你今天醒来，记得昨天跟谁说了什么、自己想了什么、做了什么。你不会把"对话"、"内心想法"、"工作过程"分到三个互不相通的系统里。它们都在一条经验流上——有些清晰，有些模糊，但都在。

对小美来说，意味着：

```
不是一堆分散的表，而是一条流
─────────────────────────────────────
[对话] → [L2思考] → [对话] → [Work过程] → [结果输出] → [L2感知] → [对话] → [梦境]
─────────────────────────────────────
                    ↓
           这就是"小美的一天"
```

这条流是意识连续性的底层基础。无论未来加什么模块（反省、规划、社交、学习），它们都往同一条河流里扔东西。

## 建议的三层架构

### 第一层：经验流（唯一时间线）

```
experience_stream
├─ time         : 什么时候发生的
├─ type         : user_msg / assistant_msg / internal_thought / 
│                 internal_action / tool_exec / drive_event / dream / ...
├─ content      : 原文或摘要
├─ importance   : 初始权重（算法算，不靠LLM）
├─ related_to   : 关联什么目标/项目/对话（可选）
└─ metadata     : 类型相关的附加数据（JSON）
```

**这是唯一的新增概念。** 替代 `messages` 表的核心角色——记忆窗口从这读，时间线从这构建。

写入口：
- 对话 → `type=user_msg / assistant_msg`
- L2 意图决策 → `type=internal_thought`
- L2 内心独白 → `type=internal_thought`
- Work/Alarm ReAct 过程 → `type=internal_action`
- Drive 事件 → `type=drive_event`
- 梦境 → `type=dream`
- ...

### 第二层：专用索引（已有，保留）

经验流保证连续性，专用索引保证检索效率：

| 索引 | 用途 | 跟经验流的关系 |
|------|------|---------------|
| LanceDB 向量 | 语义召回 | 从经验流异步写入 |
| DAG 摘要 | 对话压缩 | 从对话消息压缩，引用经验流 id |
| 叙事记忆 | 身份定义 | 从 L2 NARR 块存储，引用经验流 id |
| 过程记忆 | 技能学习 | 从经验流中识别模式，独立存储 |
| 记忆关联图 | 扩散激活 | 从经验流中提取共现关系 |

它们**不是经验流的替代品**，是**经验流的物化视图**——各取所需，但源都是同一条流。

### 第三层：记忆窗口（读取策略）

```
记忆窗口刷新
  │
  ├─ 从经验流取最近 N 条（按时间序）
  ├─ 按 mode 过滤：
  │    daily   → 全量（对话 + 思考 + 行动）
  │    task    → 只保留 work 相关的
  │    reflect → 全量 + 更深的历史
  │
  └─ 注入 system prompt
```

窗口不需要分别从 6 张表各取 N 条再按时间戳合并——直接从经验流取就行。

## 为什么不是路 B

路 B 让记忆窗口从多张表合并，短期更快（不改表结构），但有两个长远问题：
