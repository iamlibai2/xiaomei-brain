# 上下文模式判断设计

## 概述

每次用户发消息，xiaomei-brain 需要决定用哪种"上下文模式"来组装 LLM 的 system prompt。不同模式注入的内容不同——闲聊需要完整的记忆和历史，工作中断只需要最小上下文。

这个判断曾经靠关键词匹配，现在由 InnerVoice（LLM）在每次对话后感知判断，下次对话时取用。

## 模式定义

| 模式 | 含义 | 注入内容 |
|------|------|---------|
| `flow` | 心流状态，用户在工作/思考，不打断 | header + being + essence + body + dag_summaries |
| `daily` | 日常闲聊、情感交流 | + longterm_memories + relation_chains + narratives + learn_queue + desk |
| `task` | 明确工作任务、技术问题 | + longterm_memories + learn_queue + desk（不含 narratives/relation_chains） |
| `reflect` | 深度反思（L3 显式触发） | 同 daily，完整上下文 |

## 架构演进

### V1：关键词匹配（已废弃）

```
用户输入 → 扫关键词列表 → 命中 reflect_keywords → "reflect"
                             命中 past_keywords → "daily"
                             命中 simple_patterns → "flow"
                             ...
```

问题：
- 关键词列表需要人工维护，永远不全
- "你觉得呢"可能是闲聊也可能是征求意见，边界模糊
- 中文表达的灵活性导致大量漏判和误判
- 增加配置复杂度（7 组关键词，60+ 个词）

### V2：InnerVoice LLM 感知（当前）

```
对话完成 → InnerVoice.pause(CHAT_TURN) → LLM 输出 ---MODE--- → 存在 _last_mode
                                                                      ↓
下次用户消息 → determine_mode() 读取 InnerVoice._last_mode → daily / task / flow
```

LLM 在 CHAT_TURN 反省时附带判断这段对话的上下文模式：

```
在 ---MODE--- 分隔符后，判断这段对话的上下文模式（只输出一个词）：
---MODE---
daily、task 或 flow
daily: 闲聊、问候、情感表达、征求意见、回忆往事——队友般的日常对话
task: 明确的工作任务、技术问题、开发需求、执行指令——需要工具和技能记忆的任务模式
flow: 你正在心流状态中持续工作，用户只是旁观或简单确认，不需要切换上下文
如果不确定选择哪种模式，默认输出 daily
```

`determine_mode()` 的判断顺序：

```
1. recent_has_tool_calls  → "daily"（工具调用流中不降级）
2. energy < threshold     → "flow"（能量极低，最小上下文）
3. pending REFLECT/DREAM  → "reflect"（L3 显式触发）
4. has_active_goal        → "task"（Purpose 有活跃目标）
5. InnerVoice MODE        → 对应模式（LLM 感知结果）
6. high desire tension    → "daily"（高欲望需要完整上下文）
7. default                → "daily"
```

优势：
- 不需要维护关键词列表
- LLM 理解语义，不受具体用词限制
- 模式判断和社交感知复用同一次 LLM 调用，不增加额外开销

### V3：多维度信号（未来方向）

当前 InnerVoice 输出单一模式标签（daily/task/flow），足够覆盖主要场景。但人类对话的模式不止这三种——亲密对话、社交场合、学习探索等场景对上下文的需求各不相同。

如果未来需要更细粒度的判断，可以让 LLM 输出多个维度值，由代码组合决策：

```
---MODE---
context_depth: high
talk_style: casual
goal_oriented: false
emotion_charge: high
```

| 维度 | 含义 | 影响什么 |
|------|------|---------|
| `context_depth` | 需要多深的历史（high/low） | 记忆召回量、DAG 摘要层数 |
| `talk_style` | 对话风格（casual/formal/intimate） | 是否渲染叙事记忆、关系信息 |
| `goal_oriented` | 是否有明确任务（true/false） | 是否加载学习队列、桌面、项目 |
| `emotion_charge` | 情感浓度（high/low） | 是否渲染关系链、增强身体状态 |

LLM 只负责感知（这段对话情感浓度高不高？），代码负责决策（情感高 → 走 reflect 模式）。

示例——同一人三段不同对话：

```
# 工作群讨论技术方案
context_depth: high, talk_style: formal, goal_oriented: true, emotion_charge: low
→ 代码路由: task

# 深夜跟小美吐露心事
context_depth: high, talk_style: casual, goal_oriented: false, emotion_charge: high
→ 代码路由: reflect

# 边工作边随口聊两句
context_depth: low, talk_style: casual, goal_oriented: false, emotion_charge: low
→ 代码路由: flow
```

映射规则：

```python
def _dimensions_to_mode(dims):
    if dims["goal_oriented"]:
        return "task"
    if dims["context_depth"] == "low":
        return "flow"
    if dims["emotion_charge"] == "high":
        return "reflect"
    return "daily"
```

这样的好处：
- LLM 做擅长的感知题，不做不擅长的工程决策
- 新增维度不影响已有判断逻辑
- 维度值可以在未来驱动更精细的上下文裁剪（比如 `context_depth: medium` 召回 8 条记忆而非全部）
