## 18. ● PYTHONPATH=src python3 -m xiaomei_brain run xiaomei --cli ...

**🟥 用户提问（2026-04-20 02:29:35）**

> ● PYTHONPATH=src python3 -m xiaomei_brain run xiaomei --cli  解释下这个命令原理

**助手回答（2026-04-20 02:29:50）**

**`python3 -m xiaomei_brain`** — 以模块方式运行，Python 查找 `xiaomei_brain/__main__.py` 并执行它的 `main()`

**`run xiaomei --cli`** — 三个参数传给 `argparse`：
- `run` — 子命令，对应 `cmd_run()`
- `xiaomei` — agent_id，指定启动哪个 agent
- `--cli` — 附加终端交互，读 stdin 输入塞进消息队列

**`PYTHONPATH=src`** — 告诉 Python 从 `src/` 目录找模块，这样 `import xiaomei_brain` 才能找到

执行流程：

```
python3 -m xiaomei_brain
    → src/xiaomei_brain/__main__.py::main()
    → argparse: command="run", agent_id="xiaomei", cli=True
    → cmd_run("xiaomei", attach_cli=True)
        → AgentManager.build_agent("xiaomei")  → AgentInstance
        → AgentLiving(instance)
        → thread: cli_reader()  → input() → living.put_message()
        → asyncio.run(living.run())
            → WAKING → AWAKE → ...主循环
```

不加 `--cli` 就是纯后台服务，消息只能从 channel（飞书/ws）进来。

在这套系统里，程序记忆是指 **agent 学会"怎么做事"的经验**——哪些工具能解决什么问题、什么参数组合效果好。

| tool_name | args | result |
|-----------|------|--------|

如果程序记忆接通了，下次用户问"搜一下天气"，agent 看到历史记录知道 `web_search` 能用、该怎么传参，而不是重新摸索。

类比人脑：
- **语义记忆**（长期记忆）：知道"用户喜欢吃川菜" — 关于世界的事实
- **程序记忆**（tool_history）：知道"搜天气用 web_search" — 关于怎么做的技能

## 51. 你认为tool_history应该如何使用

**🟥 用户提问（2026-04-20 14:45:20）**

> 你认为tool_history应该如何使用

**助手回答（2026-04-20 14:45:49）**

说实话，我认为当前阶段 `tool_history` 的价值不大。

**LLM 已经有的信息：**
- 工具定义（schema 告诉它有什么工具、参数怎么传）
- `_fresh_tail()` 从 DB 读到的最近对话（包含当前会话里的工具调用和结果）

**tool_history 理论上能额外提供的：**
- 跨会话的工具使用经验（"上次搜天气用 web_search 成功了"）

## 71. llm.py 具体用途，和core.py 区别

**🟥 用户提问（2026-04-20 19:47:08）**

> llm.py 具体用途，和core.py 区别

**助手回答（2026-04-20 19:47:47）**

## llm.py vs core.py 的区别

### llm.py — 底层 HTTP 客户端

**职责**：封装与 LLM API 的通信，只做一件事：**发请求，收回答**。

```
用户代码 → llm.chat() → HTTP POST → LLM API → 返回 ChatResponse
```

| 功能 | 说明 |
|------|------|
| `LLMClient` | HTTP 客户端，持有 api_key / base_url / model |
| `chat()` | 发送消息，获取回复（含 tool_calls），自动重试 |
| `_build_messages()` | **合并同角色连续消息**（解决 MiniMax 400 问题）|
| `_strip_thinking()` | 去除 `<think>...</think>` 标签 |
| 重试策略 | 429/5xx 指数退避（1s, 2s, 4s），最多 3 次 |
| 日志控制 | `log_level` 参数：普通调用 INFO，记忆提取用 DEBUG |

**不关心**：Agent 在做什么、上下文是什么、工具是什么。

---

### core.py — Agent 主循环

**职责**：驱动整个 ReAct 循环，调用 llm.py 执行对话。

```
用户输入 → core.stream() → 组装上下文 → 调用 llm → 工具调用 → 流式输出
```

| 功能 | 说明 |
|------|------|
| `Agent` | ReAct 主循环：think → tool → act → repeat |
| `stream()` | 流式输出，`run()` 内部调用的非流式封装 |
| `_clean_messages()` | **清洗 surrogate 字符**（防止 400 错误）|
| `_call_llm_streaming()` | 调用 llm 流式接口，追踪 `in_think` 过滤思考块 |
| 上下文组装 | `ContextAssembler` 合并 DAG 摘要 + 长期记忆 + recent tail |
| 记忆写入 | 每次 tool_call 和 assistant 回复都写入 `ConversationDB` |
| 流程控制 | `max_steps=10` 上限，工具执行，异常捕获 |

---

### 关系图

```
core.py (Agent)
    │
    ├── 持有 LLMClient 实例
    │
    ├── self.context_assembler.assemble()  →  组装上下文
    ├── self._clean_messages()            →  清洗 surrogate
    │
    ├── 调用 llm.chat() 或 _call_llm_streaming()
    │       │
    │       └── llm.py (LLMClient)
    │           ├── _build_messages()  →  合并同角色消息
    │           ├── HTTP POST /chat/completions
    │           └── 重试 / 超时 / 解析响应
    │
    └── 执行 tools.execute() → 写 ConversationDB
```

**简单说**：
- **llm.py** = 工具人，只管 "把消息发出去，把回答拿回来"
- **core.py** = 大脑中枢，理解用户 → 决策 → 工具调用 → 流式回复，全部流程

## 76. 这样合理吗

**🟥 用户提问（2026-04-20 20:04:27）**

> 这样合理吗

**助手回答（2026-04-20 20:04:54）**

从设计上值得商榷，取决于你怎么看。

## 当前设计：llm.py 完全自治

```
调用链：agent_manager → new LLMClient(model, api_key, base_url)
                           ↑
                    agent_manager 负责读 config.json 并传入
```

**优点**：
- llm.py 是纯基础设施，不绑定任何项目概念
- 可以独立测试：直接 `LLMClient("glm-5", "key", "https://...")` 就能跑
- 调用方有权衡余地：可以读 config，也可以从环境变量/命令行参数等传入

**缺点**：
- 所有调用方都要重复 config 读取逻辑
- 如果未来有多处创建 LLMClient（如 ws server / channel / 工具），每处都要自己处理配置

## 当前实际：只有 agent_manager.py 创建 LLMClient

所以实际上没有重复问题，agent_manager 自然承担了"从哪里取配置"的职责。

## 更好的做法建议

给 llm.py 加一个 `from_config()` 工厂方法，让配置读取内聚：

```python
# llm.py 新增
@classmethod
def from_config(cls, config: Config, provider: str | None = None) -> "LLMClient":
    """从项目 Config 对象创建 LLMClient。
    
    Args:
        config: 项目配置对象（包含 provider/model/api_key/base_url）。
        provider: 可选，覆盖 config 中的 provider。
    """
    p = provider or config.provider
    model = config.model or "glm-5"
    api_key = config.api_key
    base_url = config.base_url
    return cls(model=model, api_key=api_key, base_url=base_url, provider=p)
```

这样：
- **llm.py 仍然不直接 import config.py**（解耦）
- 但提供了 from_config 便捷方式
- 调用方仍然可以手动传参（灵活性）

**助手回答（2026-04-20 21:49:18）**

## config_provider.py — 配置热管理（读写+冲突检测+订阅通知）

---

### config.py vs config_provider.py 的区别

```
config.py         — 配置数据结构（只读：from_json / to_json）
config_provider.py — 配置读写中心（可读写 + 监听 + 冲突检测）
```

简单说：config.py 是"静态数据结构"，config_provider.py 是"动态配置管理器"。

---

### ConfigProvider 核心能力

**1. get(path)** — 读取配置（dot notation）
```python
provider.get("agent.model")        # → "glm-5"
provider.get("channels.feishu")     # → {...}
provider.get("")                    # → 完整配置
```

**2. patch(partial, base_hash)** — 部分更新（JSON Merge Patch）
```python
provider.patch({"tts_enabled": True})  # 只改一个字段，其他保留
```

**3. apply(new_config, base_hash)** — 全量替换

**4. hash** — 配置版本号，用于冲突检测

---

### 冲突检测（乐观锁）

```python
# 客户端 A 读取配置
hash_a = provider.hash

# 客户端 B 在这期间修改了配置
provider.patch({"key": "new"})

# 客户端 A 想提交修改，必须带 hash
provider.patch({"key": "changed"}, base_hash=hash_a)
# → ConflictError！说明中途被改过
```

防止多人同时修改配置导致覆盖。

---

### 订阅通知

```python
provider.subscribe(lambda new_config: print("配置变了！"))
# 任何 get/patch/apply 都会触发所有 handler
```

用途：CLI 修改配置 → WebSocket 服务自动收到通知 → 重载生效（不需要重启服务）。

---

### 全局单例

```python
get_provider()  # 返回全局唯一的 ConfigProvider
```

CLI / WS / Agent 都用同一个 provider，确保配置一致性。

---

### 在项目中的使用

```
CLI (cli.py)           → get_provider().get/set
WS Server (server.py)  → get_provider() 处理 config.get/patch/apply
```

两者都操作同一个配置文件，通过 hash 检测并发修改冲突。

```python

ImageProvider.generate_to_files(prompt, output_dir)
    → 调用 MiniMax API，返回图片路径

# tools/builtin/image.py — 工具层
@tool(name="generate_image", ...)

    paths = _image_provider.generate_to_files(...)  # 调用 image.py

```

- `image.py` 可以被其他模块直接用（不只 Agent 工具）
- `tools/builtin/image.py` 是 Agent 的工具接口，把 provider 变成 tool 格式

所以不是重复，是**基础设施 + 集成层**的分离。

## 121. 你认为怎么改

**助手回答（2026-04-20 23:39:45）**

2. **Service 级别的硬编码**（TTS/Music/Image 的 `base_url`）→ 这些是 MiniMax 专属的，不应该作为通用默认值

## 建议方案

参考 `PROVIDER_DEFAULTS` 的设计模式，为各 service 增加独立的默认配置表：

```python
# 顶部新增
SERVICE_DEFAULTS = {
    "tts": {
        "base_url": "https://api.bj.minimaxi.com",
        "env_key": "MINIMAX_TTS_API_KEY",
    },
    "music": {
        "base_url": "https://api.minimaxi.com",
        "env_key": "MINIMAX_MUSIC_API_KEY",
    },
    "image": {
        "base_url": "https://api.minimaxi.com",
        "env_key": "MINIMAX_IMAGE_API_KEY",
    },
}
```

然后在 `__post_init__` 中做统一解析：

```python
def _resolve_service_settings(self) -> None:
    for service, defaults in SERVICE_DEFAULTS.items():
        enabled = getattr(self, f"{service}_enabled", False)
        api_key = getattr(self, f"{service}_api_key", "")
        base_url = getattr(self, f"{service}_base_url", "")

        if not api_key:
            api_key = os.environ.get(defaults["env_key"], "")
            setattr(self, f"{service}_api_key", api_key)

        if not base_url and enabled:
            setattr(self, f"{service}_base_url", defaults["base_url"])
```

**效果**：
- TTS/Music/Image 的 `base_url` 只在对应 service `enabled=True` 且未配置时才回落到 MiniMax URL
- 用户不启用这些 service 时不会意外获得 MiniMax 端点
- 环境变量 `MINIMAX_TTS_API_KEY` 等可独立设置 service API key
