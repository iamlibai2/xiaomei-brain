sys.exit 和os.exit 什么区别

---

**[03:26:16] 🤖助手**

`sys.exit(0)` 走 Python 正常退出流程，`os._exit(0)` 直接交给 OS。

```
sys.exit(0)                          os._exit(0)
─────────────────────────            ─────────────────────────
1. raise SystemExit                  ❌ 不抛异常
2. except/finally 正常执行            ❌ 不走
3. atexit 回调                       ❌ 不走
4. __del__ 析构                      ❌ 不走
5. threading._shutdown()             ❌ 不走
   → 等待非 daemon 线程
6. flush stdout/stderr buffer        ❌ 不走（可能丢输出）
7. C 层 exit()                       ⬇
                                     ⬇
                             1. C 层 _exit()
                                → 内核立即回收进程
                                → OS 回收文件句柄/内存/锁
```

核心差异就一个：**`sys.exit` 让 Python 体面告别，`os._exit` 直接死。** 代价是 `os._exit` 可能丢缓冲区里没 flush 的输出，所以 `_do_exit` 在打印"再见"时用了 `flush=True`。

**[03:28:14] 🤖助手**

一般程序都用 `sys.exit()`，因为那些体面清理是有意义的：

- 日志系统需要 flush 缓冲区，不然最后几行日志丢了
- 数据库连接要 close，释放连接池
- 文件写入要 flush，避免数据截断
- 网络连接要挥手，让对端正常感知断开

**xiaomei-brain 用 `os._exit()` 是特殊场景下的取舍**，不是因为 `sys.exit()` 不好。

具体到这个项目：

| 场景 | 用哪个 | 原因 |
|------|--------|------|
| 正常 `exit` 命令退出 | `os._exit` | 已经在 `_do_exit` 里手动 save 了关键状态，剩下的不必等 |
| Ctrl+C 登录阶段 | 刚改成 `os._exit` | 音频流阻塞问题，`sys.exit` 等不起 |
| 启动失败 `sys.exit(1)` | `sys.exit` | 还没启动任何后台线程/设备，正常退出即可 |

说白了就是：当你确定自己已经干了该干的善后（save 状态、stop 设备），剩下的线程 join / 资源析构反而可能卡住，就用 `os._exit` 快刀斩乱麻。否则用 `sys.exit`。

能解决。问题不在 `sys.exit`，而在 `RealMicrophone.stop_stream()` 的实现有 PortAudio 死锁。

简化的调用链：

```
stop_stream()
  → _stream_running = False        # 设标志
  → _stream_thread.join(timeout=1)  # 等 reader 线程退出
  → ❌ reader 线程卡在 Pa_ReadStream() C 调用里，1秒后 join 超时
  → _stream.close()                # 尝试关流
  → Pa_CloseStream 要拿 PortAudio 内部锁
  → ❌ 锁被 reader 线程的 Pa_ReadStream 占着，死锁
```

根因是 Windows WASAPI/MME 后端下，`Pa_ReadStream` 不会因另一个线程调 `Pa_CloseStream` 而被安全中断。

修法：不要在 reader 线程还活着的时候调 `_stream.close()`。直接把 `_stream` 置 None，让 `Pa_Terminate` 统一收拾。

**[05:19:46] 🤖助手**

三个是 OpenCV 在 Windows 上打开摄像头的不同 API 后端：

- **`CAP_ANY`（默认）** — 让 OpenCV 自己选，内部按优先级尝试。最简单，兼容性最好
- **`CAP_DSHOW`（DirectShow）** — Windows 老牌多媒体框架（2000 年代至今），可以指定分辨率/帧率/编码格式，但需要摄像头驱动正确实现 DirectShow 接口。有些 USB 摄像头和虚拟摄像头实现有 bug，预热就失败
- **`CAP_MSMF`（Media Foundation）** — Vista 后替代 DirectShow 的新框架，接口更现代，但很多摄像头根本没实现 MSMF 支持

技术上三者都调用 Windows 内核的 `ks.sys`（Kernel Streaming）驱动栈，差异在上层 API。你机器上 MSMF 不可用、DSHOW 不稳定，默认后端工作正常——所以调成默认优先。

**[07:35:39] 🤖助手**

登录时的人脸识别走的是 `Eyes.recognize_faces()`，和 ExpressionMonitor 是**两套独立流程**：

**登录（按空格）：**
```
摄像头 capture() → jpg bytes → 临时文件 → FaceID.detect_all(文件路径) → match()
```
- 拍照一张，存临时文件，用 `face_recognition.load_image_file()` 加载（天然连续内存）
- 这就是为什么登录能识别成功，ExpressionMonitor 却报错 — 同样的 `FaceID.match()`，但上游的编码提取方式不同

**ExpressionMonitor 实时监控：**
```
摄像头 subscribe_frames() → BGR ndarray → frame[:,:,::-1] → face_locations() → face_encodings() → match()
```
- `frame[:, :, ::-1]` 产生**非连续内存视图** → dlib `compute_face_descriptor()` 拒绝
- 刚才的 `np.ascontiguousarray()` 修复就是解决这个

两者最终都调同一个 `FaceID.match()`，tolerance 都是 0.5。修完后 ExpressionMonitor 的身份识别应该也能正常工作了。

为什么没用拍照的方式监控？

---

**[07:36:53] 🤖助手**

因为拍照方式每 3 秒要走一次 **jpg 编码 → 写文件 → 读文件 → jpg 解码** 的完整 IO 路径，而实时帧流直接在内存中传递 BGR ndarray，零 IO 开销。

两者的取舍：

| | 拍照 (`capture()`) | 帧流 (`subscribe_frames()`) |
|---|---|---|
| IO | jpg编解码 + 磁盘读写 | 无 |
| 延迟 | ~50-100ms 额外 | 即时 |
| 实现 | 简单，走文件 | 需要处理内存连续性 |

刚才的 bug 不是架构问题，只是漏了 `np.ascontiguousarray()` — 修好后帧流方式更高效。

## 退出链路

```
Ctrl+C (主线程)
  → KeyboardInterrupt (run.py:869)
    → _do_exit("再见")
      1. voice_listener.stop()
      2. em.stop()
      3. body.eyes._device.close()  ← 直接释放摄像头硬件
      4. drive.save() + purpose.save()
      5. living.stop()  → _running=False + queue.put(_STOP_SENTINEL)
      6. thread.join(timeout=10)  ← 等待 living 线程退出
         ├─ 正常退出 → _on_stop() 做完整清理 → sys.exit(0)
         └─ 超时 → os._exit(0) 强制终止
```

**设计意图**：`_do_exit()` 先抢跑关键清理（摄像头、状态保存），再通知 living 线程优雅退出。10 秒超时后强制 `os._exit(0)` 兜底。

## 发现的问题

### 1. `ConsciousLiving._loop_idle()` 没检查 `_STOP_SENTINEL`

`Living._loop_idle()` 基类有 `if msg is _STOP_SENTINEL: return`，但 `ConsciousLiving` 覆盖了这个方法，**只检查 `if msg is not None`**。`_STOP_SENTINEL` 不是 None，所以会被当作普通消息传给 `_handle_message()`，触发 AttributeError，靠 `finally` 里的 `_on_stop()` 兜底清理。功能上没坏，但是靠异常驱动退出，不干净。

```python
# conscious_living.py:1177 — 缺少 _STOP_SENTINEL 检查
msg = self._wait_message(self.tick_interval)
if msg is not None:          # ← _STOP_SENTINEL 也走这里
    self._on_wake_up()       # 不必要的唤醒
    self._transition(LivingState.AWAKE)  # 不必要的状态切换
    self._handle_message(msg)  # ← AttributeError
```

### 2. `ConsciousLiving._loop_dormant()` 同样问题

覆盖了基类，也没检查 `_STOP_SENTINEL`。

### 3. 重复清理（功能正确但不优雅）

- `em.stop()` 在 `_do_exit()` 和 `_on_stop()` 各调一次
- `drive.save()` / `purpose.save()` 也是
- `body.eyes._device.close()` 先被 `_do_exit()` 直接调用，然后 `_on_stop()` → `body.close()` → `device.close()` 再调一次

好在我们改过的 `close()` 是幂等的（`if cap is None: return`），不会崩溃。但职责边界模糊。

## Windows 摄像头架构

Windows 上应用程序不直接操作摄像头硬件，而是通过操作系统提供的**多媒体框架**来访问。主要有两套框架：

### 1. DirectShow（DShow）— 老牌框架，2000年代

```
应用 → DirectShow → 摄像头驱动 → 硬件
```

- 诞生于 Windows 98/XP 时代，成熟稳定
- App 直接连接驱动，**用完后驱动立即释放硬件**
- 缺点：不支持多 App 同时使用同一个摄像头（独占模式）

### 2. Media Foundation（MSMF）— 微软推荐的新框架，2010年代

```
应用 → Media Foundation → Windows Camera Frame Server（系统服务）→ 摄像头驱动 → 硬件
```

- Windows 8+ 引入，UWP 应用的标配
- 中间多了一层 **Camera Frame Server**——一个常驻系统服务
- 优点：**多个 App 可以同时访问摄像头**（Frame Server 负责分发画面）
- 缺点：App 关闭后，Frame Server 不一定释放硬件，因为它是独立的系统进程，不知道"所有 App 都用完了"

### 3. 问题根源

```
我们的 Python 程序
  → cv2.VideoCapture(0, cv2.CAP_ANY)
    → OpenCV 默认选 MSMF
      → cap.release() 只是告诉 Frame Server "我不读了"
        → Frame Server 说 "但我是系统服务，我不能随便退出"
          → 摄像头指示灯持续亮，进程残留
```

`cap.release()` 只释放了 OpenCV 和 Frame Server 之间的连接，但 Frame Server 本身是系统级进程，它不会因为一个客户端断开就退出。只有你**注销 Windows 或者杀掉 Frame Server 服务**它才会停。

这是 OpenCV 的已知 bug：`cv2.CAP_ANY` 在 Windows 上会优先选 MSMF，而 MSMF 的 release 不彻底。

### 4. 解决方案

```
我们的 Python 程序
  → cv2.VideoCapture(0, cv2.CAP_DSHOW)
    → 强制走 DirectShow，绕过 Frame Server
      → cap.release() → 驱动直接释放硬件
        → 摄像头指示灯熄灭
```

**为什么不是默认**：微软已经不推荐 DShow 了，新 API、UWP 应用都强制走 Media Foundation。OpenCV 优先 MSMF 是顺应微软方向。但对于 Python 脚本这种"用完就关"的场景，DShow 更适合。

### 5. 会不会有副作用

DShow 的缺点是**不支持多人共享摄像头**——如果你的程序在用摄像头，其他 App（比如系统相机、Teams、浏览器）就不能同时用。但对于本地 Agent 来说，这正是我们想要的行为——独占硬件，用完释放。

### 6. 我们代码里的双重保险

```python
_BACKENDS = [
    (cv2.CAP_DSHOW, "CAP_DSHOW"),  # 首先尝试 DShow，干净释放
    (cv2.CAP_ANY,   "默认"),        # 如果 DShow 不可用，回退默认
]
```

1. **优先 DShow**：绝大多数 Windows 摄像头的驱动都支持 DirectShow，正常工作就完全没有 Frame Server 问题
2. **回退 CAP_ANY**：极少数新型摄像头可能只支持 Media Foundation，此时退化为 MSMF（Frame Server 残留是 opencv 的锅，无法避免）
3. **exit 兜底**：哪怕走了 MSMF，我们之前改的 `_do_exit()` → `os._exit(0)` 会直接终止整个 Python 进程，不留给 Frame Server 继续活跃的理由

### MSMF 本身设计正确

MSMF 的 `IMFSourceReader` 是 Microsoft Edge、Teams、Zoom 等应用都在用的同一套 API。如果框架本身有问题，这些应用的摄像头切换早就炸了。

搜索到的警告信息很有价值：

```
[WARN:0] cap_msmf.cpp SourceReaderCB::~SourceReaderCB terminating async callback
```

`SourceReaderCB` 是 OpenCV MSMF 后端的异步回调对象。**析构函数里打 warning，说明回调线程在被销毁时还没有正常退出**。

MSMF 的正规释放流程是：


如果 OpenCV 没有等 Flush 完成就释放了 IMFSourceReader，那异步回调仍然挂在 Frame Server 上，释放不干净。

### 我们没法修 OpenCV

因为 `cv2.VideoCapture` 把 MSMF 的释放逻辑封死在 C 扩展里，Python 层没办法补调用。

## 根因：OpenCV DShow 后端的清理 bug

三个独立来源一致指向同一个问题：

### 1. EmguCV Issue #555
> "VideoCapture does not release all the resources when VideoCapture.API.DShow is used. Resources (possibly threads) continue to run after Stop and Release."

### 2. OpenCV 论坛
> "DirectShow filter graph management code... graph builder appears to **never be fully torn down or released**." — 内存和 GDI 句柄在 `open() → read() → release()` 循环中无限增长。

### 3. DirectShow 规范要求的清理顺序
```

    → Release(每个 filter)
      → Release(IGraphBuilder)
        → Release(ICaptureGraphBuilder2)
```

**OpenCV 的 `cap.release()` 跳过了 Stop 和 RemoveFilter 步骤**，直接释放了 graph builder。图形管道没有被停止，Frame Server 不知道客户端已经断开，摄像头硬件继续被占用。
