```python
# DAG 自己的方法
class DAGSummaryGraph:
    def get_active_messages(self, session_id: str, limit: int = 100) -> list[dict]:
        """返回未摘要的消息（core.py 直接用这个，不需要自己拼 ID）"""
        unsummarized = self.get_unsummarized_messages(session_id, limit=limit)
        unsummarized_ids = {m["id"] for m in unsummarized}
        return [m for m in self.db.get_recent(limit*2, session_id=session_id)
                if m.get("id") in unsummarized_ids]
```

然后 core.py 里直接：

```python
# 清理压缩消息（委托给 DAG）
active_messages = self.context_assembler.dag.get_active_messages(self.session_id)
```

这样 DAG 的压缩状态管理内聚在 DAG 层，core.py 不用知道 unsummarized_ids 怎么查。

## 92. LLM 输出清洗是做什么用

**🟥 用户提问（2026-04-25 03:53:09）**

>  LLM 输出清洗是做什么用

**助手回答**

防止 LLM 输出的文本里夹带了**孤立代理字符**（lone surrogates, U+D800~U+DFFF）。这些字符不是合法的 UTF-8，一旦后续写入文件/数据库/JSON 就会触发 `UnicodeEncodeError`。

这层防御和用户输入过滤是**两个场景**：

| | 用户输入 | LLM 输出 |
|---|---|---|
| 来源 | 终端/IME，容易产生孤立代理 | API 返回的 JSON，正常应是干净 UTF-8 |
| 风险 | 高（输入法组合字符） | 极低（API 服务端已规范化） |
| surrogatepass 副作用 | **破坏 CJK 字节边界** → 乱码 | 几乎不会触发，因为输出本身不含代理字符 |

LLM 输出这层基本是个"不出事就不执行"的保险。实际触发概率极低，所以 surrogatepass 在这里没有造成过实际问题。需要我也换成字符级过滤吗？

当前问题：

## 204. _assemble_reflect()，是什么用途

**🟥 用户提问（2026-04-25 23:03:34）**

> _assemble_reflect()，是什么用途

**助手回答**

`reflect` 模式用于反省场景，从对话日志中看：

- 用户说"答对了吗"、"做错了"、"反思"、"反省"时触发
- 或者意识系统有待处理的 `DREAM`/`REFLECT`/`RECALL` Intent 时触发
- 比 `daily` 模式给更多 token 预算给历史上下文（`remaining // 2` vs `remaining * 2 // 5`），记忆召回阈值也更低（0.4 vs 0.6），方便深度反省

三种模式对比：
- **flow**：最轻量，只给最近 4 条，简单问答用
- **daily**：常规对话，DAG 摘要 40% 预算 + 长期记忆
- **reflect**：反省模式，DAG 摘要 50% 预算 + 更多记忆召回
