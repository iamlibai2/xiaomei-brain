## PACE 作为默认执行模式

### 做了什么

在 `task_orchestrator.py` 的 `_run_react()` 中，每次普通聊天（chat 模式）执行完后，自动调用 `assess_chat()` 跑一遍规则检测。

具体改动：
- `runner.py`：新增 `assess_only()` 方法——不干预执行，只跑 7 条规则检测，记录异常信号
- `task_orchestrator.py`：新增 `assess_chat()` 公共 API，`_run_react()` 结束后自动调用

### 意义

**之前**：规则检测只在 PACE 模式（`/intask` 或 `!` 前缀）下生效。普通聊天时完全不做检测——即使 LLM 陷入 TOOL_LOOP、输出空响应、连续重复，系统也看不见。

**之后**：每次聊天完成都跑规则检测。检测到异常就记录到日志（`[PACE-assess] 检测到异常信号: ...`），积累到 `observations` 历史中，为后续的 lesson 复盘和能力校准提供数据。

核心思想：**认知控制层不应该是"开关"，而应该是持续运行的背景监控**。就像人不管是做任务还是闲聊，脑子里的自我监控一直在跑。

代价：零 LLM 调用，纯规则检测（正则匹配 + 计数器 + Jaccard 相似度），微秒级完成。

| **时机** | chat 结束后事后跑一次 | 在对话过程中持续感知 |
| **影响** | 记日志，无反馈 | 影响下一轮的 context / nudge |

| **对象** | 只看单轮输出 | 看多轮模式——重复、漂移、情绪变化 |

### 缺的几个维度

**1. 跨轮模式检测**

现在每条规则只看当前步 + 上一步。但真正的监控应该跟踪整个会话的趋势：
- 连续 3 次 `REPEATED_OUTPUT` 比单次更值得注意
- `EMPTY_RESPONSE` 之后出现 `TOOL_LOOP` 可能是"模型在乱撞"
- 话题从 A 漂移到 B 再漂移到 C——目标漂移检测

**2. 反馈闭环**

检测到了异常，然后呢？现在是记日志就没了。应该：
- 积累到一定阈值 → 自动注入 nudge 到下一轮 context（"你连续3次回答高度相似，注意一下"）
- 异常模式 → 降低 drive 能量（表示"执行不顺利"，让 Agent 更谨慎）
- 重复输出 → 上下文压缩触发（旧的没用的消息该清掉了）

**3. 对话健康度**

单个 chat session 应该有"健康度"概念，类似人的"聊天状态"：
- 流畅度：没有卡顿/重复 = 健康
- 聚焦度：话题没有漂移 = 健康
- 效率：工具调用不多不少 = 健康

这个健康度可以实时影响 context_assembler 的行为——不健康时就少给记忆、多给提示。

**4. 与 Drive 层联动**

这才是"背景监控"的核心——监控结果应该进入边缘系统：
- 检测到 GAVE_UP → 皮质醇升高，能量降低
- 对话流畅 → 多巴胺微涨
- 用户打断/纠正 → 血清素微降

这样认知控制就不只是一个"质检员"，而是 Agent 内在状态的一部分。

## 对话的物理时间线

```
User Msg → Context Assembly → LLM Stream → Output → [用户看到] → ... → Next User Msg
           |                  |            |        |
           ①                  ②            ③        ④
```

现在我们的规则检测只在 **③ 之后** 跑一次。持续感知意味着在 **①②③④** 每个节点都有感知能力：

| 节点 | 现在 | 持续感知应该做 |
|------|------|---------------|
| ① 上下文组装前 | 无 | 读完上一轮的"遗留状态"，注入提示 |
| ② LLM Streaming 中 | 无 | 观测 token 流速、工具调用堆积 |
| ③ 输出完成后 | 规则检测 | 规则检测 + 更新"会话状态" |
| ④ 等待用户输入 | 无 | 状态自然衰减（类似激素半衰期） |

## 关键洞察：感知 ≠ 实时插话

**持续感知不等于在 streaming 过程中打断 LLM。** LLM 生成是一个黑盒——token 逐个蹦出来，你很难在中途判断"这条思路对不对"。强行中途打断反而更糟糕。

真正有价值的是 **在 LLM 每一轮执行的前后插入感知点**，然后用一个跨轮累积的 **会话健康度状态** 把各轮的感知结果串联起来。

## 具体实现方案

### 1. SessionHealthState — 跨轮累积状态

```python
@dataclass
class SessionHealthState:
    """单次对话会话的健康度状态，跨轮累积"""
    
    # 流畅度
    consecutive_empty: int = 0        # 连续空响应次数
    consecutive_repeated: int = 0     # 连续重复输出次数
    consecutive_tool_loops: int = 0   # 连续工具循环次数
    
    # 聚焦度
    topic_drift_score: float = 0.0    # 话题漂移累计
    last_intent_type: str = ""        # 上轮意图（检测意图跳变）
    
    # 效率
    total_tool_calls: int = 0         # 本次会话总工具调用
    avg_response_length: float = 0.0  # 平均回复长度
    
    # 衍生状态
    health_score: float = 1.0         # 0-1，整体健康度
    needs_nudge: bool = False         # 下一轮是否需要注入提示
    nudge_message: str = ""           # 提示内容
```

关键：这个对象存活在整个 session 生命周期内，每轮 chat 结束更新，下一轮 chat 开始前读取。

### 2. 三个感知时机

**时机 A — 上下文组装前（`_run_react` / `_run_pace` 入口）**

检查 `SessionHealthState`，决定是否注入 nudge：

```python
def _pre_turn_check(self, health: SessionHealthState) -> str:
    """每轮执行前，根据健康度决定是否注入提示"""
    if health.consecutive_repeated >= 2:
        return "[元认知] 你连续多次回答高度相似，可能陷入了重复模式。请检查是否真的推进了对话。"
    if health.consecutive_empty >= 2:
        return "[元认知] 前两轮没有有效输出，请确保本轮产出实质内容。"
    if health.topic_drift_score > 0.7:
        return "[元认知] 对话话题相比最初已经漂移较远，请确认是否仍聚焦于用户意图。"
    return ""  # 健康，不注入
```

**时机 B — LLM Streaming 中（轻量观测）**

不打断，只观测。用 chunk callback 收集轻量信号：

```python
class StreamObserver:
    """streaming 过程中的轻量观测器"""
    
    def on_chunk(self, chunk: str) -> None:
        self.chunk_count += 1
        self.total_chars += len(chunk)
        # 检测工具调用出现（不计数，只标记）
    
    def on_tool_call(self, name: str) -> None:
        self.tool_calls.append(name)
        if len(self.tool_calls) > 8:
            self.flags.append("TOOL_STORM_WARNING")  # 提前预警
    
    def summarize(self) -> dict:
        """output → assess 阶段的输入"""
        return {
            "chunk_count": self.chunk_count,
            "tool_call_warning": "TOOL_STORM_WARNING" in self.flags,
            "response_too_long": self.total_chars > 8000,
        }
```

这个 observer 不做决策，只收集信号，交给 output 后的 assess 阶段统一判断。

**时机 C — 输出完成后（增强版 Assess）**

现有的 7 条规则检测 + 更新 SessionHealthState：

```python
def assess_and_update_health(self, obs, health):
    # 1. 现有规则检测
    surprises = detect_surprises(obs, ...)
    
    # 2. 更新 health state
    if EMPTY_RESPONSE in surprises:
        health.consecutive_empty += 1
    else:
        health.consecutive_empty = 0
    
    if REPEATED_OUTPUT in surprises:
        health.consecutive_repeated += 1
    else:
        health.consecutive_repeated = 0
    
    # 3. 重新计算健康度
    health.health_score = self._compute_health(health)
    
    # 4. 判断是否需要 nudge
    health.needs_nudge = health.health_score < 0.5
    health.nudge_message = self._generate_nudge(health) if health.needs_nudge else ""
```

## 把 PACE 检测接入 Drive 的实际用处

核心逻辑就一条：**执行不顺利 → 消耗更多能量 → 更早进入"不做主动行为"状态**。

```
当前: GAVE_UP → 记日志 → 下一轮照样继续 → 可能继续撞墙
之后: GAVE_UP → 皮质醇↑ 能量↓ → 主动行为被抑制 → Agent 变"安静"，不乱动
```

具体三个场景：

**1. GAVE_UP → 皮质醇升高 + 额外能量消耗**

不是让 Agent "感到沮丧"——而是让它 **消耗更多能量**。连续拒绝执行说明 Agent 在无效消耗资源，此时额外扣能量，让它更快进入低能量保护状态，不再瞎折腾。

**2. 对话流畅 → 多巴胺微涨 → 维持能量**

不是让 Agent "开心"——而是让 **成功的执行不消耗额外能量**。如果这一轮顺滑（无异常信号），能量消耗就是基础的 0.05；如果有异常，能量消耗加倍。这样能量曲线就反映了执行质量。

**3. 用户打断/纠正 → 血清素微降 → 能量消耗 + 认知欲抑制**

用户纠正说明 Agent 方向不对。此时降低认知欲（不要自己乱学），专注听用户的。

## 汇成一句话

> **Drive 是能量预算系统。PACE 检测结果决定这一轮花了多少能量。执行质量差 = 更多能量消耗 = Agent 更快进入"安静"状态 = 不会在错误方向上持续消耗用户 token。**

这是真正的闭环——不是让 Agent "有情绪"，而是让它在做不好时自动收敛。

## PACE 异常检测：7 条规则

```
每次 Agent step 执行完（一轮 LLM 调用 + 工具执行）后立即检测
输入：当前 StepObservation + 历史 StepObservation 列表
输出：surprises 列表写入 obs
```

| # | 规则 | 检测方式 | 阈值 |
|---|------|----------|------|
| **TOOL_LOOP** | 连续 ≥3 步调用同一工具 | 取最近 3 步的 `tool_calls[0]`，看是否全部相同 | 3 步 |
| **TOOL_STORM** | 单步工具调用暴增 | `obs.tool_call_count > 10` | 10 次 |
| **EMPTY_RESPONSE** | 去掉 PROGRESS 标签后输出为空 | `obs.llm_output.strip() == ""` | 空 |
| **REPEATED_OUTPUT** | 连续 2 步输出高度相似 | `content_similarity(prev, curr) > 0.9`（CJK bigram + Jaccard） | 0.9 |
| **SLOW_STEP** | 单步耗时远超历史均值 | `obs.elapsed > avg(history) * 3`，需 ≥2 步历史 | 3x |
| **NO_PROGRESS** | 调了工具但没标记进度 | `tool_call_count > 0 && !has_progress_tag` | — |
| **GAVE_UP** | Agent 明确拒绝/放弃 | 关键词匹配（"不重复"、"死循环"、"已经答"等）+ 上一步也有 REPEATED_OUTPUT 或 GAVE_UP | — |

## 问题

1. **`_observations` 是 PACERunner 实例变量** — 跨 chat 保留，但**没有被消费**。积累到 50 条后截断，白白丢掉
2. **检测结果只打了日志** — 没有流入 Drive，没有流入 Consciousness，没有变成下一次 L2 的输入
3. **`tool_calls=[]` 永远是空** — `_run_react` 只捕获了 `tc_count`（次数），没传工具名称列表，所以 **TOOL_LOOP 永远检测不到**

## 两条路径对比

```
日常 Chat                             /intask (PACE 模式)
────────────────────────────────────  ────────────────────────────────────
_run_react()                          _run_pace() → _run_loop()
  │                                     │
  ├─ ReAct 循环                          ├─ Pre-check: 目标是否模糊
  │   (while True, 多轮)                 │   → escalate / clarify
  │                                     │
  ├─ 不拆分步骤                          ├─ 注入历史 lesson
  │                                     ├─ 注入上一步 nudge
  │                                     │
  ├─ 不限重试                            ├─ 子目标重试上限 (5次)
  │                                     ├─ 连续3步空响应 → ERROR
  │                                     │
  ├─ 不检测异常                          ├─ detect_surprises() → 7条规则
  │                                     │
  ├─ 不决策                              ├─ _step_check()
  │                                     │   ├─ 硬规则: TOOL_STORM → RETRY
  │                                     │   ├─ 硬规则: EMPTY → RETRY  
  │                                     │   ├─ 硬规则: GAVE_UP×2 → ESCALATE
  │                                     │   └─ 其他 → LLM step_check (预算控制)
  │                                     │        ├─ 输出 Nudge 注入下一轮
  │                                     │        └─ 输出 CONTINUE / CLARIFY / ESCALATE ...
  │                                     │
  │                                     ├─ 能力埋点 (CapabilityTracker)
  │                                     ├─ 可观测性埋点 (PACEMetrics)
  │                                     │
  └─ 结束后:                              │
    assess_chat() → assess_only()        └─ 结束后:
      ├─ 仅规则检测                          _do_post_review()
      ├─ 只打日志                              ├─ LLM 复盘
      ├─ 不做任何事                            ├─ persist_lesson()
      └─ observations 白白丢掉                  └─ metrics 持久化
```

## 一句话总结

| | 日常 Chat | /intask PACE |
|---|---|---|
| **检测** | 规则（残缺，无工具名） | 规则 + LLM step_check |
| **决策** | 不决策 | 硬规则 + LLM → nudge 注入下一轮 |
| **反馈** | 无 | nudge → context，escalate → 暂停 |
| **学习** | 无 | lesson 持久化，下次注入 |
| **Observations** | 累积到 50 条后丢弃 | 累积到 50 条后丢弃 |
| **Drive 反馈** | 无 | consume_energy(0.05) |

日常 Chat 的 PACE 基本是**空壳**——检测残缺、无反馈、无学习。`/intask` 模式的 PACE 才是真正在工作的认知控制层。
