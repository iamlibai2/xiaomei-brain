---

**[00:35:15] 🤖助手**

好问题。实际上区别没那么大，但对实际工程有几个关键影响：

### 理论上的区别

| | System Prompt | User/Assistant Messages |
|---|---|---|
| **定位** | 设定"你是谁、你要遵守什么" | 对话本身——发生的事 |
| **优先权** | 通常更高，user 指令不应覆盖 system 约束 | 可以被后续消息修正 |
| **持久性** | 整场对话的背景布 | 来去流动 |

### 实际上，对现代 LLM 来说

区别不大。训练数据里 system/user/assistant 边界本就模糊。更重要的三个现实因素：

1. **MiniMax 硬限制**：只允许 1 条 system message（error 2013），所以我们把所有东西 merge 成一个大 system prompt。这倒逼我们不管三七二十一全塞 system。

2. **Attention 分布**：system prompt 在最前面，模型处理每条 user 消息时都会"穿过"它，相当于强制关注。放在 user 消息里的内容，可能被后续轮次稀释。

3. **对话结构**：`agent.messages` 是 user/assistant 交替的对话线程。如果把主动消息塞进去，结构会变成 `… → assistant(主动) → user(新消息) → assistant(回复)`，这其实不破坏结构，反而让 LLM 知道"我刚才说了 X，现在用户回复 Y"。

### 回到本来的问题

`fresh_tail` 第五遍过滤主动消息，隐含的假设是"孤立的 assistant 消息会破坏对话结构"。但这个假设值得商榷——人不会忘记自己说过的话，agent 主动说了什么，接着用户来聊天，这不是"孤立"，这就是**自然的时间线**。

所以我的建议不变：去掉第五遍过滤，让主动消息进入 `agent.messages`。与此同时 `recent_dialog` 在 daily 模式也不需要加了——主动消息已经在 messages 里了，再加就重复了。这样 agent 的对话上下文就是完整的、按时间顺序的，和人类的记忆体验一致。

你给我讲一下llm 对于消息的规则要求

---

**[00:46:10] 🤖助手**

以 OpenAI 兼容协议为基础，讲核心规则：

### 1. 角色类型

Messages 是一个数组，每条消息有 `role`：

| Role | 含义 | 约束 |
|------|------|------|
| `system` | 设定 AI 行为、身份、规则 | 通常 0-1 条，放最前面 |
| `user` | 用户输入 | 无限制，但对话必须以 user 开头（有些 API 强制） |
| `assistant` | AI 回复 | 可以带 `tool_calls`，也可以纯文本 |
| `tool` | 工具执行结果 | 必须有 `tool_call_id`，紧跟在对应的 assistant(tool_calls) 后面 |

### 2. 各平台的差异

```
Anthropic:
  system: [{type: "text", text: "..."}, ...]     ← 顶层字段，支持数组
  messages: [user, assistant, user, assistant]     ← system 不在 messages 里
  
OpenAI / 兼容协议（DeepSeek、智谱、MiniMax）:
  messages: [
    {role: "system", content: "..."},              ← 0-1条，在数组首位
    {role: "user", content: "..."},
    {role: "assistant", content: "..."},
    ...
  ]
  
Gemini:
  system_instruction: "..."                        ← 单独参数，只能1个
  contents: [user, model, user, model]             ← role 叫 "model" 不是 "assistant"
```

### 3. 结构约束（几乎所有 API 都要求）

- **对话必须 user 开头，user 结尾**（最后一条消息必须是可以回复的）
- **alternating**：user → assistant → user → assistant（ReAct 循环中的 tool 视为 assistant 的附属）
- **tool 必须配 assistant(tool_calls)**：孤立的 tool 消息会报错
- **空 content**：assistant 可以 `content: ""`（如果只是 tool_calls），但大多不允许 `content: null`

### 4. DeepSeek 特有约束

- **单 system**：V4 严格 1 条，多余的静默忽略
- **V4 不让 prefill assistant**：assistant 消息末尾不能不带 tool_calls 就停住（R1 时代可以靠 prefill `{"role": "assistant", "content": "..."}` 诱导格式，V4 禁了）

### 5. 这对我们的影响

回到主动消息问题。当前架构：

```
agent.messages = [user₁, assistant₁, user₂, assistant₂, ...]
                                       ↑ 对话线程
system_prompt = identity + DAG + narratives + ...  (merge成1条)
```

`fresh_tail` 从 DB 加载时，主动消息是孤立的 `assistant` 夹在对话中：

```
assistant(主动)  →  无前置 user，破坏了 alternating 结构
user₃           →  用户来聊天了
assistant₃      →  回复
```

**这才是第五遍过滤的真正原因**——不是"agent 不该知道自己发了什么"，而是怕破坏 `user → assistant` 交替结构，导致 API 报错。

但如果主动消息后面刚好接了一条 user 消息，结构就自然成立了：

```
assistant(主动)  →  OK（上一条对话的延续，或者新对话的...）
user₃           →  OK
assistant₃      →  OK
```

问题是：如果主动消息是 DB 里**第一条**（之前没有任何 user），那它就没有前置 user，就成了异常。

**[00:49:43] 🤖助手**

LLM 不返回 messages，返回的是**单条完成结果**：

### 非流式（chat）

```json
// 请求
POST /v1/chat/completions
{
  "messages": [
    {"role": "system", "content": "你是小美"},
    {"role": "user", "content": "你好"}
  ]
}

// 返回
{
  "choices": [{
    "index": 0,
    "message": {
      "role": "assistant",
      "content": "你好呀，今天过得怎么样？"
    },
    "finish_reason": "stop"
  }],
  "usage": {"prompt_tokens": 12, "completion_tokens": 8}
}
```

### 流式（chat_stream）

```
// SSE 流
data: {"choices":[{"delta":{"role":"assistant","content":"你"}}]}
data: {"choices":[{"delta":{"content":"好"}}]}
data: {"choices":[{"delta":{"content":"呀"}}]}
data: {"choices":[{"delta":{"content":""},"finish_reason":"stop"}]}
```

### 核心点

**LLM 只输出一条 assistant 消息。** 它不会返回新的 messages 数组，不会重写历史。那条 `{"role": "assistant", "content": "..."}` 是调用方（我们）自己写进 `agent.messages` 的：

```python
# core.py 的 ReAct 循环里
agent.messages.append({"role": "assistant", "content": content})
```

这就是为什么消息结构完全由我们控制——LLM 只负责填空，不负责维护对话结构。结构合法性（user/assistant 交替、system 单条、tool 配 tool_calls）全部是调用方的责任。

Anthropic   的格式 是什么样的

**[00:54:28] 🤖助手**

和 OpenAI 格式差异很大，三个关键点：

### 1. 请求结构

```
OpenAI:
  messages: [{role: "system", ...}, {role: "user", ...}, ...]
  扁平数组，system 也在 messages 里

Anthropic:
  system: "你是..."           ← 顶层字段，不在 messages 里
  messages: [
    {role: "user", content: "你好"},
    {role: "assistant", content: "你好呀"}
  ]
```

可以传多个 system 片段：

```json
{
  "system": [
    {"type": "text", "text": "你是小美"},
    {"type": "text", "text": "你不能说谎", "cache_control": {"type": "ephemeral"}}
  ],
  "messages": [...]
}
```

### 2. Content 是 block 数组

OpenAI 的 content 是纯字符串，Anthropic 每条消息的 content 是数组：

```json
// 纯文本
{"role": "user", "content": [{"type": "text", "text": "你好"}]}

// 工具调用嵌在 content 里
{"role": "assistant", "content": [
  {"type": "tool_use", "id": "tc1", "name": "read_file", "input": {...}}
]}

// 工具结果
{"role": "user", "content": [
  {"type": "tool_result", "tool_use_id": "tc1", "content": "文件内容..."}
]}
```

**工具调用和目标不在顶层**，而是在 content 数组里和文本交错。

### 3. 流式响应用 named event

```
OpenAI SSE:
  data: {"choices": [{"delta": {"content": "你"}}]}

Anthropic SSE:
  event: content_block_delta
  data: {"delta": {"type": "text_delta", "text": "你"}}
  
  event: message_delta
  data: {"delta": {"stop_reason": "end_turn"}}
```

有明确的 event 类型：`message_start` → `content_block_start` → `content_block_delta` → `message_delta` → `message_stop`，不像 OpenAI 全混在 `choices[0].delta` 里。

### 项目里怎么处理的

`AnthropicMessagesTransport.convert_messages()` 负责把内部格式（OpenAI 兼容的 messages）转成 Anthropic 格式，`normalize_response()` 再转回来。这样上层代码不感知差异，都拿到统一的 `NormalizedResponse`。

本项目的做法：一个主循环 + 多个通道 queue。

```
                    ┌──────────────┐
  CLI (terminal)  → │              │
                    │  queue.Queue  │ → AgentLiving 主循环 → 处理 → 产生回复
  WS (network)    → │              │
                    └──────────────┘
```

- **CLI 通道**：主线程 `input()` 阻塞等待用户打字 → `put_message()` 入队
- **WS 通道**：`asyncio` 事件循环在另一个线程跑 WebSocket 服务器 → 收到消息 → `put_message()` 入队

两个通道各自在独立的线程里收消息，统一往 `queue.Queue`（线程安全）扔。AgentLiving 主循环 `get_message()` 阻塞等待，收到就处理。

回复也是反过来：`Router.deliver()` 根据 session_id 路由到对应通道——CLI 的走 `print()`，WS 的走 `websocket.send()`。

核心就是：**queue 做解耦，Router 做分发**，两个通道互相不知道对方存在。
