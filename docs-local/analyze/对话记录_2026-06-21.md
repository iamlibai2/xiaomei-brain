## 三、核心子系统深度分析

### 3.1 Agent 核心 (ReAct 循环)

**core.py** (718行) 是整个系统的发动机，实现了完整的 ReAct 范式：

- **流式生成**：`stream()` 方法是同步生成器，一边调用 LLM 一边 yield 文本块
- **工具调用**：LLM 返回 `tool_calls` → Agent 执行 → 结果追加到消息列表 → 重新调用 LLM
- **防重试死循环**：相同 `(tool_name, args_json)` 失败 3 次后自动阻断
- **自动 DAG 压缩**：当未摘要消息超过 token 阈值（75%）或数量阈值（18条），自动将 8 条消息压缩为叶子摘要
- **记忆块解析**：从 LLM 输出中提取 `<MEMORY>`、`‖...‖`（思考）、`[PROC:...]` 等结构化块
- **消息按 user_id 隔离**：多用户共享同一 Agent 实例但拥有独立消息历史

**AgentManager** (935行)：
- 支持多 Agent 注册管理（每个 Agent 独立身份、记忆、LLM、工具）
- **配置级联**：API key/model 从 Agent 级 → config.json → 环境变量 → 全局默认
- **懒加载**：LLM、工具、记忆系统在首次使用时才初始化
- `identity.md` 动态读取：编辑即生效，无需重启

### 3.2 记忆系统（5层架构）

记忆系统是"一切层的基础"，所有其他层站在记忆之上。

| 层次 | 实现 | 存储 | 特点 |
|------|------|------|------|
| **SelfModel** | `self_model.py` (446行) | `identity.md` 文件 | 身份/追求/热爱/底线/生长记录 |
| **对话日志** | `conversation_db.py` (454行) | SQLite + FTS5 | 一字不差，永不删除 |
| **DAG 摘要** | `dag.py` (668行) | SQLite (brain.db) | 8条→叶子摘要→高层摘要，LLM生成 |
| **长期记忆** | `longterm.py` (2357行) | SQLite + LanceDB | 向量语义召回，user_id 隔离 |
| **记忆提取** | `extractor.py` (977行) | - | immediate/periodic/dream 三模式 |

**关键设计决策**：
- **单一 brain.db**：所有记忆系统共享同一个 SQLite 数据库
- **LanceDB 向量索引**：BAAI/bge-m3 (1024维)，支持远程 Embedding 服务
- **CJK 处理**：FTS5 无法分词中文，回退到 `LIKE` 搜索
- **记忆强度衰减**：指数衰减 `0.9995^hours`，读时计算，不写回
- **第一人称视角**：所有长期记忆用"我..."（Agent）/ "用户..."（用户）
- **灭绝≠删除**：灭绝记忆可被唤醒，已删除的永久消失

**新增记忆子系统**（未在 CLAUDE.md 中记录）：
- **ExperienceStream**：统一追加式事件时间线，所有行为（对话/思考/工具/Drive事件）都写入
- **PatternMemory**：跨时间统计规律提取，5 个注入点（System Prompt / L2 Intent / 对话 / 学习主题 / Dream）
- **ProcedureMemory**：程序性记忆（"当 X 发生，做 Y"），触发词验证，执行追踪
- **NarrativeMemory**：`<NARR>` 块解析，自我叙事（"我是谁"），场景标签合并

### 3.3 意识系统（火焰骨架架构）

核心理念：**代码维护骨架，LLM 加柴涌现**。意识如火焰，不是一直燃烧的，需要持续"加柴"。

**SelfImage** (455行) — 火焰骨架载体：
- 7 大模块：Being（身份）、Body（身体）、Perception（感知）、Mind（认知）、Memory（记忆窗口）、Intent（意图缓冲）、History（时间自我）
- **Proxy 模式**：SelfBody 代理 Drive 的实时值，SelfMind 代理 Purpose 的实时值
- **12+ contribute_*() 方法**：类型安全的写入接口
- **Desk（共享桌面）**：任何模块可投放/查看上下文，权重算法（置信度 + 意图 + 多巴胺 + 目标相关 + 访问奖励 + 指数衰减）

**四层心跳**：
| 层级 | 频率 | 机制 | 职责 |
|------|------|------|------|
| L0 | 1 Hz | 规则 | 火焰骨架维护 + 内感受采集 |
| L1 | 1/min | 规则 | 异常检测 + 关系衰减 |
| L2 | ~10s 检查 | LLM | 意图决策 + 意识涌现独白 |
| L3 | 按需 | LLM | 深度反省 |

**三层 Global Workspace Theory 渲染**：
- **v1**：全量注入，5 种模式（flow/daily/task/reflect/legacy）
- **v2**：动态评分（17 节评分 + 自适应权重学习）
- **v3**（当前默认）：8 种模式，5 段式身体渲染（摘要→指标→冲动→说话风格→参考值），所有数值硬编码映射

**Dream Engine（梦境引擎）**：
- 串行流水线：ExtractJob → ReinforceJob → RelationReinforceJob → ProcedureConsolidation → NarrativeConsolidation → PatternExtraction → DreamBurn
- Dream Burn：LLM 生成完整梦报告 + 摘要 + 情绪变化

### 3.4 Drive 层（边缘系统）

**纯算法层** — LLM 只识别事件类型，代码决定具体数值变化。

**四大子系统**：

### 3.5 Purpose 层（前额叶）

**三层目标层级**：

## 中肯评价

### 先说好的

**架构视野是顶级的。** 国内做 AI Agent 的开源项目不少，但绝大多数停留在"LLM + 工具调用 + RAG"的缝合层面。这个项目真正思考了"一个持续存在的数字生命需要什么"，并给出了自己的答案——分层记忆、边缘系统、火焰意识、元认知。思路是原创的，不是跟风。

**记忆系统是真正的工程亮点。** 对话日志 → DAG 分层摘要 → 向量语义召回 → 模式/叙事/程序记忆，这条链路完整且实用。DAG 压缩的孤儿消息修复、指数衰减读时计算、CJK 搜索回退等细节处理得很到位。

**Drive 层设计干净。** 纯算法驱动、LLM 只识别事件类型、代码决定数值变化——这个边界画得很正确。情绪冲突矩阵、激素→情绪双向调制、快感对手过程模型，数学上是自洽的。

**执行力和迭代速度惊人。** 一个人 70 天写 7 万行代码 + 17 万字文档，从零构建到现在的完整度，编程能力毋庸置疑。

### 再说问题

**过度工程化是最核心的问题。** 7 万行代码做一个聊天机器人——即使是有记忆、有性格的聊天机器人——投入产出比严重不对等。很多子系统的实际效用存疑：

- 多巴胺/血清素/皮质醇的数值变化最终只是改变了系统 prompt 里的几行文字，和直接写一个"用户夸我了，我心情好"的规则差别不大
- 三层 GWT 渲染（v1/v2/v3）同时存在，v2 的显著性评分 + 自学习权重看起来很厉害，但实际对回复质量有多少提升？
- BodyWear 的 5 种磨损机制很精致，但对用户体验的影响几乎不可感知

**量变没有引起质变。** 增加了这么多层（Purpose → Drive → Consciousness → Metacognition），每一层都有自己的 LLM 调用（L2 Intent、L2 Emergence、InnerVoice、SocialCognition、Dream），导致一次用户消息可能触发 5-6 次 LLM 调用。延迟和 token 消耗都很高，但回复质量相比于一个简单的"ReAct + 记忆召回"提升了多少？

**意识层的隐喻过重。** "火焰骨架"、"加柴"、"梦境燃烧"这些概念很美，但代码实现中大量精力花在了渲染火焰状态的文字描述上（v3 的 5 段式身体渲染有 1000+ 行）。这些文字注入到 system prompt 里，LLM 真的能"感受"到吗？还是只是在 prompt 里多了一段装饰性文本？

**缺少实际场景的验证。** 177 篇文档大多是设计讨论和对话记录，缺少"用户场景 X 下，有了 Y 层后效果从 A 变成 B"的量化对比。这导致很多设计决策看起来是"因为人脑有这个结构，所以我们也加"——而人脑的每个结构是几亿年进化的结果，不是每一层都值得模拟。

### 一句话总结

**这是一个技术能力很强、架构思考很深、但工程判断上偏离了实用主义的项目。** 如果目标是探索"数字意识"的可能性，这是一个很有意思的实验。如果目标是做一个好用的 AI 助手，砍掉 60% 的代码可能会更好。

## 实现方案

三个文件，改动很小：

### 1. `memory/longterm.py` — 暴露 warmup 完成状态

```python
# __init__ 里加一个 Event
self._warmup_complete = threading.Event()

# _warmup_embedder 结束时 set
def _warmup_embedder(self):
    try:
        ...
    finally:
        self._warmup_complete.set()

# 新增公开方法
def is_embedder_ready(self) -> bool:
    return self._warmup_complete.is_set()

def wait_embedder(self, timeout: float | None = None) -> bool:
    return self._warmup_complete.wait(timeout=timeout)
```

### 2. `consciousness/conscious_living.py` — 启动时显示 embedding 状态

在 `LongTermMemory(db_path)` 之后，把当前的：
```python
boot_line("长期记忆", "OK", f"{ltm_count} 条" if ltm_count else "")
```

改为：
```python
ltm_count = self.agent.longterm_memory.count() if hasattr(self.agent.longterm_memory, 'count') else 0
if not self.agent.longterm_memory.is_embedder_ready():
    boot_line("长期记忆", "OK", f"{ltm_count} 条（Embedding 后台加载中，首次约 20 秒）" if ltm_count else "Embedding 后台加载中，首次约 20 秒")
else:
    boot_line("长期记忆", "OK", f"{ltm_count} 条" if ltm_count else "")
```

### 3. `cli/run.py` — 首次对话前等待 embedding 就绪

在 `_run_agent` 里，login 完成后、进入消息循环前，加一段：

```python
# 等待 Embedding 模型加载完成

if ltm and not ltm.is_embedder_ready():
    print(f"\n  ⏳ Embedding 模型加载中（首次约 20 秒），请稍候...", flush=True)
    ltm.wait_embedder()
    print(f"  ✓ Embedding 模型就绪\n", flush=True)
```

`start_new_session=True` 做了两件事：
1. 子进程脱离当前终端，成为独立的 session leader
2. 父进程退出后，子进程不受终端关闭/Ctrl+C 影响

Windows 没有 Unix session 概念。但这里实际情况是：`cmd_start()` 打完 PID 就退出了，子进程自己继续跑。Windows 上父进程退出不会杀子进程，所以**不需要这个参数也能正常工作**。

修法很简单——Windows 上不传这个参数即可。
