1. **`intent_context` 是走后门** — 所有 consciousness 数据走 `SelfImage → render_* → inject_consciousness()`，目标数据却绕过它直接拼字符串。两条管道灌同一个 system prompt。

**1. GoalManager 是上帝对象**

800 行，什么都干——意图分析、目标创建、路由、PACE 初始化、ReAct 分派、完成处理、知识提取、继续处理、检查点管理。它既是门面也是实现，没有清晰的边界。

每张便利贴（能力）是独立加上去的，有自己的触发条件，贴在不同的地方。厨师做菜时不是按照统一流程走，而是在不同地方翻不同的便利贴。

**这就是你现在的 PACE 循环**——surprise detection 在 309 行，step_check 在 321 行，InnerVoice 在 364 行，视角切换在 531 行，价值重估在 557 行。它们彼此不知道对方的存在，靠 if/elif 串联。

## 统一认知循环：厨师有一套固定的工作台

好的厨师怎么做？**每一步做完，自动走四个工位**：

```
做完一步 →
  工位1【看到了什么】→ 工位2【这什么意思】→ 工位3【下一步做什么】→ 工位4【动手】
                                                                    │
                                                                    ▼
                                                              回到工位1
```

**每个工位只看上一个工位的产出，不关心其他工位内部发生了什么。**

---

### 工位1：PERCEIVE —— "我看到了什么"

**只收集事实，不做判断。**

```
输入：LLM 刚吐出来的原始文本
输出：一张"观察卡片"

观察卡片上写着：
  - LLM 说了什么：<完整文本>
  - 有没有 PROGRESS 标签：{"status": "completed"}
  - 调了什么工具：["shell", "file_read"]
  - 有没有异常：空响应 / 工具风暴 / 放弃信号
  - 耗时多久：3.2s
```

对应现在的代码：`detect_surprises()` + `parse_progress_tag()` + `_extract_tool_names()`。

**关键约束**：这一站不判断好坏。工具调了 20 次？只是记下来，不评论。LLM 给了空响应？只是标记 `EMPTY_RESPONSE`，不说"这不行"。

---

### 工位2：ASSESS —— "这意味着什么"

**拿观察卡片，产生理解。这一站是"大脑皮层"——做判断但不做决定。**

```
输入：观察卡片 + 历史观察 + 当前目标
输出：一张"理解卡片"

理解卡片上写着：
  - step_check 结果：CONTINUE（上一步正常，继续）
    或者：RETRY_DIFFERENT（上一步工具报错，换个方法）
    或者：CLARIFY（LLM 在提问，可能在等用户）
  
  - InnerVoice 信号：
    "方向不对" → 可能需要换视角
    "做不了" → 可能需要升级
    "用户在等" → 当前子目标应该阻塞
    ---INSERT--- → 有新子目标要插入
    
  - 价值重估：目标还值得做吗？是/否
```

对应现在的代码：`_step_check()` + `_invoke_inner_voice_task_step()` + `_value_reassess()`。

**关键约束**：这一站只产生"信号"，不决定下一步。InnerVoice 说"方向不对"，但 ASSESS 不下结论"所以要换视角"——那是 DECIDE 的事。

现在的 PACE 为什么乱？**因为 ASSESS 和 DECIDE 混在一起**。视角切换的触发条件（`should_breakthrough = iv_retry or retries >= 1`）写在 DECIDE 分支里（531行），价值重估的触发条件（`retries == 3`）也写在 DECIDE 分支里（557行）。信号产生和信号消费耦合了。

---

### 工位3：DECIDE —— "我该做什么"

**唯一做决策的地方。综合所有信号，输出一个 Action。**

```python
def decide(assessment):
    signals = {
        "step_check": assessment.step_check,    # CONTINUE / RETRY / ESCALATE / ...
        "iv_retry": assessment.iv_retry,        # InnerVoice: "方向不对"
        "iv_block": assessment.iv_block,        # InnerVoice: "等待用户"
        "iv_insert": assessment.iv_insert,      # InnerVoice: "插入子目标"
        "iv_escalate": assessment.iv_escalate,  # InnerVoice: "做不了"
        "retries": assessment.retry_count,
        "value_ok": assessment.value_ok,        # 价值重估结果
        "perspective_tried": assessment.perspective_tried,
    }
    return route(signals)  # 路由器
```

**路由规则（优先级从高到低）**：

```
1. iv_escalate + retries >= 2        → ESCALATE（升级求助）
2. step_check == ESCALATE            → ESCALATE（升级求助）
3. iv_block                          → EXIT_WAITING（退出等用户）
4. iv_insert                         → INSERT_SUB_GOALS（插入子目标）
5. iv_retry + not perspective_tried  → SWITCH_PERSPECTIVE（换视角）
6. retries == 3 + not value_ok       → SKIP_CURRENT（放弃当前子目标）
7. step_check == RETRY_DIFFERENT     → RETRY（重试）
8. step_check == CLARIFY             → EXIT_WAITING（LLM在问问题）
9. step_check == CONTINUE            → CONTINUE（正常继续）
10. step_check == REPORT_PARTIAL     → COMPLETE_CURRENT_AND_ADVANCE（完成并推进）
```

这就是**统一路由器**——所有信号平等进入，按优先级仲裁，输出一个 Action。不再有分散的 if/elif，不再有能力藏在别的能力的条件里。

---

### 工位4：ACT —— "动手"

**拿 DECIDE 输出的 Action，执行。**

```
输入：Action + 上下文
输出：LLM 的新一轮文本

Action 到执行的映射：
  CONTINUE              → 正常 LLM stream（保留当前上下文）
  RETRY                 → LLM stream + 注入重试提示
  SWITCH_PERSPECTIVE    → 调 PerspectiveEngine 生成方向感 → 注入上下文 → LLM stream
  INSERT_SUB_GOALS      → 调 PurposeEngine 插入新子目标 → 刷新上下文 → LLM stream
  SKIP_CURRENT          → 调 task_executor.apply_skip() → 推进到下一个子目标
  ESCALATE / EXIT       → 退出循环
  COMPLETE_CURRENT      → 标记完成 → 推进下一个 → 继续循环
```

## 聊天 vs 任务：同一套循环，不同的深度

关键洞见：**聊天就是"轻量版任务执行"**。一个聊天回复也经过 PERCEIVE → ASSESS → DECIDE → ACT，只是深度不同。

```
聊天（depth=LIGHT）：
  PERCEIVE: 收集 surprise + tool 信息
  ASSESS:   跳过 step_check（聊天不需要检查"这步做完了没"）
            InnerVoice 只做轻量反思（"我说话合适吗"）
  DECIDE:   总是返回 COMPLETE（聊天一轮就结束）
  ACT:      LLM stream

任务（depth=FULL）：
  PERCEIVE: 完整收集（surprise + progress + tool）
  ASSESS:   完整判断（step_check + InnerVoice + value reassess）
  DECIDE:   完整路由（可能继续/重试/换视角/升级/插入子目标）
  ACT:      LLM stream
```

**不需要两套循环**。`conversation_driver._run_react()` 和 `runner._run_loop()` 合并为一个 `CognitiveLoop.run(depth=LIGHT|FULL)`。

## 总结：为什么这个模型更好

| 现在的问题 | 统一模型的做法 |
|-----------|--------------|
| 能力触发条件分散在各处，不知道彼此存在 | 所有信号在 DECIDE 汇合，路由器统一仲裁 |
| 视角切换藏在 RETRY_DIFFERENT 分支里 | SWITCH_PERSPECTIVE 是一个独立的 Action，和其他 Action 平等 |
| 加新能力要到处改 if/elif | 新能力只改 PERCEIVE/ASSESS 产生信号 + DECIDE 路由表加一行 |
| ReAct 和 PACE 是两套独立代码 | 同一套 CognitiveLoop，depth 参数控制深度 |
| step_check 和 InnerVoice 独立调用，可能冲突 | 都在 ASSESS 阶段产生信号，DECIDE 按优先级处理冲突 |

一句话：**不是"把能力都塞进 PACE"，而是"设计一个管道，每个能力在管道中有固定位置，信号沿管道流动，集中路由"**。

```
       ▼                                         注入 system prompt
  _operations[]                                "项目当前状态" 段落
```

```
操作记录:  ○ ○ ○ ○ ○ ○ ○ ○  │  ○ ○ ○ ○ ○ ○ ○ ○  │  ...
```
