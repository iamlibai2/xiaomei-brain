# 过程记忆设计文档

## 什么是过程记忆

过程记忆存的是"怎么做"——技能、流程、习惯。跟情景记忆（"发生了什么"）和语义记忆（"世界是什么样的"）不同，过程记忆是可执行的：它告诉我在什么条件下该做什么事，怎么做，做完了怎么记下来。

当前情景/语义记忆已经能存能查，但过程记忆完全缺失。我没有"学会"的能力，只有"回想"的能力。这份文档补上另外半张图。

## 数据结构

### 存储格式

每条过程记忆用 YAML frontmatter + 步骤描述体 的方式存储。如下：

```yaml
- id: PROC-001              # 唯一标识
  name: 处理用户工作请求       # 简短名称
  description: 用户提出工作相关需求时的处理流程  # 一句话描述，用于LLM判断是否匹配
  version: 2                 # 当前版本号
  version_history:
    - version: 1
      updated_at: '2026-05-05T19:49:00'
      changes: 初始版本
    - version: 2
      updated_at: '2026-05-05T20:01:08'
      changes: '步骤 s1.description: 先确认需求，不要猜'
  created_at: '2026-05-05T19:49:00'
  updated_at: '2026-05-05T20:01:08'
  last_executed: '2026-05-05T20:01:08'
  execution_count: 1          # 执行次数，用于权重排序
  execution_success_rate: 0.0 # 成功率
  weight: 0.5                 # 权重，影响排序和衰减
  status: active              # active | archived | deprecated
  trigger_conditions:         # 关键词粗筛条件（见触发机制章节）
    type: any                 # all | any
    conditions:
      - field: user_message
        operator: contains
        value: 帮我
      - field: user_message
        operator: contains
        value: 做个
  steps:                      # 步骤序列
    - id: s1
      name: 确认需求
      description: 先确认需求，不要猜
      next: s2
      fallback: s1            # 失败时重试还是跳转
    - id: s2
      name: 评估可行性
      description: 判断能不能做、需要什么资源
      next: s3
      fallback: s_ask_user
    - id: s3
      name: 执行交付
      description: 动手做，做完说明结果
      next: null
      fallback: s_escalate
  execution_log:              # 执行日志（保留最近20条）
    recent:
      - timestamp: '2026-05-05T20:01:08'
        result: success
        user_feedback: 用户满意
        notes: ''
    max_entries: 20
```

### 关键字段说明

| 字段 | 说明 |
|------|------|
| `trigger_conditions` | 关键词粗筛条件，降低后续LLM判断的候选量 |
| `description` | 一段自然语言描述，用于LLM语义匹配时判断是否该用这条 |
| `steps` | 有序步骤序列，支持条件分支（conditions）和失败跳转（fallback） |
| `weight` | 动态权重。执行成功+成功率增加，长期不用衰减。影响排序 |
| `execution_log` | 保留最近执行记录，用于复盘和衰减计算 |

## 触发机制

这是过程记忆的核心设计问题：**什么时候该唤醒哪条过程记忆？**

### 设计原则

1. **LLM自己就是匹配引擎** — 不设计复杂的外部匹配逻辑。把过程记忆的信息注入到LLM能感知的位置，让LLM自行判断是否匹配
2. **记忆量少时走轻量路径** — 条数少时（< 50条），直接把所有过程记忆的描述注入system prompt，LLM在推理时自主匹配
3. **记忆量大了再加粗筛层** — 条数多了（> 50条），先用关键词粗筛缩小候选集，再让LLM从候选集中做语义精排

### 触发流程

```
用户输入
    │
    ▼
[关键词粗筛] ──── 候选集为空? ───→ 不触发，继续正常对话
    │
    │ 有候选
    ▼
[注入候选集到推理上下文]
    │
    ▼
[LLM自行判断] ───→ 不匹配? ───→ 不触发
    │
    │ 匹配
    ▼
[执行步骤序列]
```

### 关键词粗筛层

关键词粗筛使用 `trigger_conditions` 字段，支持：

- `type: all` — 所有条件都满足才算命中
- `type: any` — 满足任一条件就算命中
- 支持的 operator：`contains`（包含）、`startswith`（以...开头）、`endswith`（以...结尾）、`regex`（正则匹配）

粗筛的目的是**快速排除**，不是精确定位。允许一定的误召回（把不相关的也筛进来），但不允许漏掉。因为后续LLM会做二次判断。

### LLM语义精排（核心触发逻辑）

通过关键词粗筛的候选集，注入到LLM的推理上下文中。LLM在推理时自行判断：

1. **当前用户输入是否匹配这条过程记忆的场景？**
2. **匹配度够不够？要不要执行？**

判断依据包括：
- 过程记忆的 `description`（自然语言描述，比关键词更精准地表征场景）
- 过程记忆的 `name` 和 `steps` 概览
- 当前对话的上下文（聊的是什么、用户状态如何）
- 对用户意图的理解（用户是真的想让我做这件事，还是随口一说）

### 多条命中的裁决

当关键词粗筛+LLM判断后，仍有多个候选过程记忆被认为"可能匹配"时（这种情况会在记忆量大了之后出现）：

**第一层：排序**
- 按匹配分数降序排列（匹配分数 = LLM判断的匹配程度）
- 同分数时按 weight 降序

**第二层：裁决**
- 取 top-3 候选
- 如果第一名和第二名的分数差 > 0.1（或一个显著阈值）：直接取第一名
- 如果分数差很小（< 0.1）：查看辅助维度（weight、最近执行时间、用户当前情绪等），综合重排
- 如果仍然难以裁决：我来决策，不把决策权推回给用户

**兜底规则：**
- 如果所有候选的匹配分数都低于一个下限阈值（如 0.3）：不触发任何过程记忆
- 如果触发了一条但执行到一半发现不合适：走 `fallback` 路径，可以优雅退出或询问用户

### 触发时机

过程记忆的触发是**实时的**——对话进行中，用户输入到来时，立刻判断是否需要触发。不是在梦境里批量处理。

但触发后的**复盘和优化**可以在梦境里做（见更新机制章节）。

## 更新机制

过程记忆有三个更新来源：

### 来源一：执行后复盘（梦境阶段）

每次执行过程记忆后，记录执行结果到 `execution_log`。在梦境阶段批量处理：

- 分析执行日志，发现常见失败模式
- 根据成功率调整 `weight`
- 优化步骤描述或顺序
- 合并重复或相似的过程记忆

### 来源二：用户当场纠正

用户说"不对，不是这样做的"——这是最高优先级的更新。

- 直接修改对应的步骤或描述
- 版本号 +1，记录变更历史
- 不打断当前对话，记下来下次用

### 来源三：频率驱动衰减

长期不用的过程记忆自动降权：

- 超过 N 天未执行：weight 减半
- weight 低于阈值：标记为 `archived`
- 归档后如果又被触发：解归档，重新激活
- 完全用不上的：标记为 `deprecated`，保留但不注入

### 版本管理

- 每次修改版本号 +1
- 保留最近3个版本的变更记录
- 不保留旧版本的完整内容（只保留变更摘要）

---

## 与梦境模块的关系

梦境模块负责：
1. **复盘分析** — 分析执行日志，发现模式
2. **衰减计算** — 遍历所有过程记忆，更新 weight
3. **归档决策** — 将低权重的标记为 archived
4. **合并去重** — 发现两条相似的过程记忆，提示是否需要合并

梦境不负责：
- 触发判断（实时触发）
- 执行过程记忆（对话中执行）
- 用户纠正（对话中处理）

## 边界情况

| 场景 | 处理方式 |
|------|----------|
| 没有过程记忆 | 正常对话，不触发任何流程 |
| 只命中关键词但没有语义匹配 | 不触发，流程静默退出 |
| 执行到一半用户打断/纠正 | 停止当前步骤，走 fallback |
| 步骤执行失败 | 走该步骤的 fallback，如果 fallback 也失败则询问用户 |
| 两条过程记忆高度相似 | 触发匹配度高的那条，复盘时提示是否需要合并 |
| 记忆量爆炸（> 200条） | 关键词粗筛 + 冷热分离（高频常驻，低频按需加载） |

## 实施顺序

1. **先验证数据结构** — 在 procedural_memory.yaml 里手动写 2~3 条示例，确认结构合理
2. **关键词粗筛** — 实现 trigger_conditions 的匹配逻辑
3. **LLM语义精排** — 把候选过程记忆注入推理上下文，让LLM自行判断
4. **裁决逻辑** — 实现 top-3 排序和决策
5. **执行步骤序列** — 按 steps 顺序执行，支持条件分支和 fallback
6. **复盘&衰减** — 接入梦境模块
