# Task Session：让 Agent 从"决策器"变成"工作者"

## 问题

当前 Agent 的能力出口是 ReAct：每步一个工具调用，等结果，再想下一步。LLM 是一个水库，ReAct 是一个 1cm 的出水管。

更深层的问题是：已有的"工作管理"系统（PACE / Goal / 子目标分解 / CognitiveLoop）把工作当成可预定义的流程来管理——分解目标 → 按步骤执行 → 打勾 → 下一个。但真正的智力工作不是流水线。你不知道具体要几步，不知道中间会发现什么，不知道最终产出会长什么样。**工作是一次探索，不是一次执行。**

## 底层原理

### 人类最根本的两个能力

从原始社会到现在，人类核心就两件事：

- **学**（获取能力）：观察长辈怎么追踪猎物、识别可食用植物、制作石器。
- **做**（运用能力）：实际去狩猎、采集、搭建庇护所。

二者不是先后关系，是**交织的闭环**：

```
做（实践）→ 遇到障碍 → 学（获取新能力）→ 再做（运用新能力）→ 遇到新障碍 → ...
```

第一次狩猎失败 → 回来反思学到了什么 → 下次改进。这个循环的副产品就是**成长**。

### Agent 同理

一个有���的 agent 无非就是：能学、能做、学做互通。当前两条能力线（Learn + Work）各有雏形但断开——学了不一定用到工作中，工作中遇到障碍不会触发学习。

---

## 硬约束：权重冻结

LLM 权重是冻结的。微调/LoRA/持续学习在当前条件下不现实——灾难性遗忘、成本不对等、基础科学未突破。

但这不是妥协。这是一个清醒的边界：**动不了权重，就动运行时。**

人脑本身也是两个层的混合：
- 权重层（突触强度）——缓慢变化，需要睡眠巩固
- 运行时层（神经调节、注意、工作记忆）——秒级变化，实时调控行为

我们做不了权重层，但运行时层可以做得很好。

---

## 核心概念：Task Session = 容器 + 操作系统

Task Session 有两层：

- **容器层**：给 LLM 一个不被中断的专注工作空间
- **操作系统层**：三个运行时机制，控制信息如何流入 LLM

### 容器层

```
┌─────────────────────────────────────────┐
│            Task Session                  │
│                                          │
│  1. 目标感                               │
│     知道要做什么，不预设怎么做           │
│                                          │
│  2. 深度工作时间                         │
│     不被心跳打断，不被 L2 中断           │
│     沉浸式，自由切换做和学              │
│                                          │
│  3. 自我评判                             │
│     做完了自己看一眼                     │
│     满意 → 交付，不满意 → 继续改        │
│                                          │
└─────────────────────────────────────────┘
```

### 操作系统层

三个运行时机制，都不碰 LLM 权重：

#### 机制一：选择性注意

**问题**：Attention 是散光灯，每个 token 跟每个其他 token 算相关性。上下文越长，目标信号越被稀释。

**原理**：人类的前注意过滤——不是所有信息平等地进入深层加工。在完整 Attention 之前，极轻的过滤层根据目标状态快速弱化无关信息。

**实现**：不修改 Attention 机制本身。在上下文组装阶段，对注入的长期记忆/DAG摘要/技能内容做目标相关性评分（embedding 余弦 + 领域标签匹配），低于阈值的降权或裁剪。这一步不经过 LLM，是纯粹的预处理。

#### 机制二：持续目标表征

**问题**：目标是 system prompt 里的一条文本，跟其他十万 tokens 一起被 Attention 处理。聊到一半，目标信号已经淹没了。

**原理**：人类前额叶持续维持目标表征——像电路里的偏置电压，恒定存在，持续影响每条通路。

**实现**：目标不只在 system prompt 里出现一次。在 Session 的每一轮 react_nodb 调用中，目标作为独立的、恒定注入的上下文前缀，放在最前面、不被裁剪。不是让 LLM "记住目标"，是让它无法忘记。

具体做法：
- Session 开始时的目标声明 → 写入 SelfImage，作为 Session 期间的固定字段
- 每轮给 LLM 的消息列表中，第一条 user message 始终是「你当前的目标：{goal}」
- 上下文压缩时，目标不参与裁剪

#### 机制三：内在评判信号

**问题**：LLM 可以连续 50 步毫无进展而不自知。没有困惑、满足、烦躁来告诉它"该换方向了"或"该停了"。

**原理**：人类的情绪和身体信号不是干扰，是无认知层——困惑告诉你策略错了，满意告诉你路径对了，烦躁告诉你你在绕圈。

**实现**：不在 LLM 内部，在 Session 层面。每步工具调用后，一个**极轻量、不带 LLM 的检查**：

```python
def _session_step_check(self, step_result: str) -> str:
    """返回 "continue" / "warn" / "block" """

    # 规则层（零成本）
    if tool_call_count > 50:       return "block"   # 步数耗尽
    if same_tool_repeated > 8:     return "block"   # 工具风暴
    if len(step_result) < 10:      return "warn"    # 空响应
    if step_result == prev_result: return "warn"    # 重复输出

    # 微模型层（可选，几百 KB ONNX）
    # 输入：目标 embedding + 当前步骤结果 embedding
    # 输出：进度评分 (0-1)
    # 连续 5 步评分下降 → "block"
```

三个级别的行动：
- **continue**：正常，继续
- **warn**：给 LLM 插入一条提示「注意：你上一步没有实质进展，考虑换个方法」
- **block**：强制暂停，要求 LLM 反思当前策略（不是停止 Session，是踩刹车）

这个机制不需要理解"任务怎么做"——它只需要判断"有没有在前进"。规则层零成本，微模型层几百 KB。

## Session 三种类型

| | WORK | LEARN | MIXED |
|---|---|---|---|
| 驱动 | 要做出东西来 | 要搞懂一个东西 | 干活中发现不懂 |
| 终点 | 东西做完了，能用 | 能解释清楚，能教给别人 | 兼有 |
| 自评标准 | 这个产出好不好 | 我真的理解了吗 | 兼有 |
| 典型场景 | 写代码、做PPT、重构 | 研究新概念、读文档 | 实际工作（最常见） |

**三种类型共享同一个容器+操作系统**，区别只在 goal 和目标函数。

## 数据结构

```python
@dataclass
class TaskSession:
    """一次专注工作/学习会话"""

    id: str                          # 唯一标识
    goal: str                        # 目标描述
    session_type: str                # "work" | "learn" | "mixed"
    state: str                       # WORKING / SELF_EVAL / DONE / ABANDONED

    # 产出
    result_summary: str              # 做了什么（agent 自己总结）
    artifacts: list[str]             # 产出文件路径
    knowledge_gained: list[str]      # 学到的知识点

    # 自评
    iteration: int                   # 自评轮次（最多 3 轮）
    self_eval: str | None            # PASS 或 FAIL: <原因>

    # 运行时（操作系统层）
    active_goal_bias: str            # 当前目标，恒定注入每轮上下文
    attention_domains: list[str]     # 选择性注意的领域标签
    step_metrics: dict               # 内在评判记录（工具调用次数/重复/进度评分）

    created_at: float
    finished_at: float | None
```

不需要 TaskStep、不需要 current_step。LLM 自己在上下文里维护。

## 学→做→学闭环

这是 Task Session 最重要的设计目标。当前系统里的断层：

```
┌──────────┐         ┌──────────┐
│  Learn 系统 │   ✗    │  Work 系统  │
│  (独立运作)  │←─────→│  (独立运作)  │
└──────────┘         └──────────┘
```

Task Session 把它们装进同一个容器，让"学"和"做"自然流动：

```
Task Session
  │
  ├─ 容器层：目标 + 深度时间 + 自评
  ├─ 操作系统层：选择性注意 + 目标偏置 + 内在评判
  │
  ├─ "我要做一个PPT"（WORK）
  │     ├─ 写内容、调排版...
  │     ├─ "图表这块我不太会"
  │     │     → 切到学习模式：搜索图表设计
  │     │     → 学会了，回去干
  │     ├─ 内在评判：连续3步无进展 → warn → LLM 调整策略
  │     └─ 完成 → 自评
  │
  └─ Session 结束后：
        有价值的知识 → LTM
        有价值的经验 → 经验池（供下次选择性注意使用）
        能力的信号 → SelfImage 更新
```

## 与现有系统的关系

### 保留并复用的

- **react_nodb()** — 底层执行，不变。每轮自动注入目标偏置
- **Learn 系统（LearningEngine/Queue/Storage）** — Session 内的"学"阶段复用
- **自评机制** — PASS/FAIL + 衰减
- **InnerVoice** — 自评 prompt 复用其 LLM 调用能力

### 新增的

- **TaskSession 数据结构**
- **选择性注意** — 上下文预过滤，不经过 LLM
- **持续目标表征** — goal 作为恒定注入，不参与裁剪
- **内在评判信号** — 规则层 + 可选微模型，每步检查进度

### 简化的

- **GoalManager / PACE / CognitiveLoop** — Session 模式下不再需要
- **子目标分解** — 不再强制

### 不修改的

- Drive SLEEP / L2 意图决策 / L0 火焰骨架 / 心跳 / 对话驱动

---

## 两条路径并存

| | 结构化路径（现有） | 自由路径（Task Session） |
|---|---|---|
| 适用 | 用户指定的明确任务 | 自主意图、探索性工作、学习 |
| 管理 | 系统管（子目标 + 状态机） | Agent 自己管 |
| 运行时 | 无 | 选择性注意 + 目标偏置 + 内在评判 |

---

## 实现路径

### Phase 1：最简容器 + 目标偏置

- `TaskSession` 数据结构
- `_do_work()` / `_do_learn_topic()` 创建 Session
- 深度工作 system prompt
- 目标作为每轮恒定前缀（机制二的最简实现）
- 结束后 PASS/FAIL 自评
- max_steps 放大

### Phase 2：内在评判

- 规则层检查（机制三的规则部分）
- 工具风暴 / 空响应 / 重复输出检测
- warn → 注入提示，block → 强制反思
- FAIL 后迭代（最多 3 轮）

### Phase 3：选择性注意

- 上下文预过滤：注入内容做目标相关性评分
- 领域标签自动匹配（PPT/code/research/chat）
- Session 结束后经验回写经验池
- 可选：微模型进度评分（机制三的微模型部分）

### Phase 4：观察效果，再决定

- 用户触发的任务是否也走 Session？
- PACE/GoalManager 是否逐渐淡出？

## 不做的事

- 不替代 Goal/PACE 体系（先并存）
- 不预设正确的工作方式
- 不在系统层管理学做切换
- 不碰 LLM 权重
- 不跨 Session 暂停/恢复
