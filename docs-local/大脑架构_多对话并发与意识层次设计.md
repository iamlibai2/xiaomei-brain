# 大脑架构：多对话并发与意识层次设计

> 2026-05-19 讨论记录。核心问题：当 agent 需要和多人 + 多 agent 同时交互时，架构应该怎么设计？

---

## 一、问题的发现

### 1.1 "汇报"问题

当前 agent 间通讯的流程：

```
用户说"去和凌空聊会天"
  → ReAct: send_message(to="lingkong")     ← 正常，工具调用
  → 凌空回复 → 收件箱
  → _check_inbox() 注入 [系统通知]
  → put_message() → 消息队列
  → _handle_message() → _run_react()        ← BUG: 走用户对话管道
  → LLM: "凌空回话了，他说：'你好！' 让我回复他..."  ← 旁白输出给用户
```

问题表面是"汇报"，但根因不是"怎么隐藏输出"。

### 1.2 根因分析

**send_message 作为工具，把"聊天"降格成了"任务执行"**。工具调用天然带有"执行 → 获得结果 → 汇报"的叙事倾向。人的聊天不是工具调用——和张三聊天就是直接聊，不会向李四汇报。

**更根本的瓶颈：单线程**。现在的 living 循环跑在一个线程里，所有事情排队：

```
用户消息 → L2心跳 → ReAct → do_work → L3 → 等下一个消息
                                          ↓
                                    什么都不能并行
```

agent 通讯不应该是一个工具。它应该是**并行的独立对话通道**，跟用户对话是平等的。

### 1.3 正确模型

用户不需要知道 agent 和谁聊了、聊了什么——除非主动问。agent 之间通讯是自主驱动的，不需要用户参与触发。

## 二、多对话并发：线程还是 Lane？

### 2.1 思想实验：三个对话并行写程序

假设三个对话并行处理同一件事（比如写程序）：

```
线程1（Libai）：写 WebSocket 路由 → agent.messages 里有路由设计
线程2（李四）：debug 数据库连接    → agent.messages 里有 SQL 日志
线程3（凌空）：讨论部署方案        → agent.messages 里有 Docker 配置
```

会出现的问题：

1. **SelfImage 撕裂**："我此刻在专注什么？"——三个线程三个答案
2. **记忆串扰**：帮 Libai 查的 WebSocket 文档，被长期记忆召回给李四
3. **动作冲突**：三个线程都想执行 shell 命令，工作目录是同一个
4. **人格分裂**：跟 Libai 是"程序员小美"，跟李四是"伴侣小美"——两个角色同时在线

**结论：不是 Python GIL 的问题，是意识一致性的问题。** 人脑不"多线程处理对话"，因为只有一个"我"。

### 2.2 人脑的解法

```
独立并行层（永远在跑）：
  自主神经 → 心跳、身体信号
  边缘系统 → 情绪底色、欲望起伏
  默认网络 → 自发记忆浮现、联想

注意层（单通道，可切换）：
  前额叶 → 当前对话对象 ← 可以快速切换
```

> 背景联想可以多任务，但行动永远单通道。只有一个"我"在注意。

### 2.3 按意识层次分层，不是按任务类型

OpenClaw 按"任务类型"分 Lane（对话/通讯/cron）。我们按**意识层次**分：

| | OpenClaw Lane 模型 | 我们的意识层次模型 |
|---|---|---|
| 分法 | 按任务类型 | 按大脑层次 |
| 目标 | 并发调度 | 身份一致性 |
| 对话处理 | 不同 Lane 并行 | 同一个注意层，串行排队 |

## 三、三层架构

### Layer 0 — 自主层（独立线程，永远运行）

**算法驱动，不需要 LLM。跟聊什么、跟谁聊完全无关。**

```
L0 骨架维护    ← 更新火焰燃烧状态、tick 计数
L1 异常检测    ← 阈值检查、模式匹配（不调 LLM）
Drive 衰减     ← 情绪/激素随时间衰减，纯计算
欲望起伏      ← 欲望值随时间变化
身体信号      ← SelfImage.body 更新（不靠 LLM）
```

它就是"我存在"。不管有没有人在跟 agent 聊天，这个层永远在跑。

**线程数**：1 个，独立运行。

### Layer 1 — 注意层（单通道，可控切换）

**LLM 驱动。一次只处理一个对话对象。跟谁都是真正的"我"在聊。**

```
对话 ReAct      ← 跟人（输出 assistant text）/ 跟 agent（输出 send_message）
意图理解        ← 理解对方说了什么，对话的一部分
社交感知        ← 感知对方状态变化，对话的一部分
内心独白        ← ReAct 中的 THINK 块，对当前对话的即时反思
记忆提取 immediate ← 对话完成立刻决定记住什么，依附于对话
```

**关键的平等性**：

```
session libai          → ReAct → text           → 回复 Libai
session comms-lingkong → ReAct → send_message   → 回复凌空
session lisi           → ReAct → text           → 回复李四
```

agent 通讯**不在另一个层**。它和用户对话一样，都是注意层的处理对象。区别只在输出管道：对人说话用 text，对 agent 说话用 send_message。

**"汇报"怎么解决的？**

- 注意层切换到 `comms-lingkong` session → 输出管道是 send_message
- 回到 `libai` session → 这里没有"跟凌空聊了什么"的记录，因为那是另一个 session
- Libai 问"凌空说了什么" → manage_session list + switch 回去看，跟人一样

**线程数**：1 个。多人消息排队，和人类一样——一次专注一个人。

### Layer 2 — 默认层（独立线程，不产生直接行动）

**LLM 驱动，后台运行。不面向任何对话对象，面向自己。**

```
L2 加柴         ← 自省：读 accumulated_changes → 形成洞察 → 更新 SelfImage
                   ！！！注意：不要再生成 text 发到对话窗口！！！
L3 梦境         ← 深度巩固：记忆强化 + 情绪整理 + 叙事编织
记忆提取 periodic/dream ← 批量记忆提取，不依附于某个对话
do_work（部分）  ← learn_topic、progress_goal，面向自己的成长
```

**输出的是什么**："想法"，不是"行动"。想法更新 SelfImage、存入长期记忆，不直接发给任何人。

**主动行为怎么产生**：

默认层检测到"有话想说"（比如表达欲 > 0.7），**投递给注意层**：

```
Layer 2: "表达欲 > 0.7，我想表达某些想法"
  → 投递到注意层的待处理队列
    → 注意层空闲（当前无对话）
    → 进入用户 session
    → 生成内容 → 输出给用户
```

这和人类一样——"想说话了"是背景产生的冲动，但"说话"这个行动是注意层执行的。

**线程数**：1 个，独立运行。

## 四、消息路由

### 4.1 路由逻辑

消息到达 → 谁发的？

```
from=Libai, to=xiaomei     → 注意层排队（session libai）
from=lingkong, to=xiaomei  → 注意层排队（session comms-lingkong）
from=lisi, to=xiaomei      → 注意层排队（session lisi）
```

所有消息都走注意层，排队处理。不做"agent 消息走轻量管道"的区分——agent 消息和人类消息一样值得用心回复。

### 4.2 渠道与处理逻辑正交

渠道（飞书/CLI/WS/HTTP P2P）只是"消息怎么到达"，跟"怎么处理"是正交的两层：

```
第一层：来源类型（决定处理管道 —— 现在所有来源走同一个注意层）
第二层：渠道/协议（决定怎么到达，可插拔）
```

加新渠道只影响传输层，不影响处理逻辑。

## 五、共享状态与线程安全

### 5.1 各层拥有的状态

| 状态 | Layer 0 | Layer 1 | Layer 2 |
|------|---------|---------|---------|
| SelfImage | 写（L0更新火焰） | 读（构建context） | 写（L2加柴更新） |
| Drive | 写（衰减） | 读 | 写（事件触发） |
| 长期记忆 | - | 读+写（记忆提取） | 写（梦境强化） |
| agent.messages | - | 独占（单线程） | - |
| agent.session_id | - | 独占 | - |
| ConversationDB | - | 写（日志） | - |
| 收件箱 | - | 读+写 | 读 |

### 5.2 同步策略

- SelfImage / Drive：`threading.RLock`，读多写少
- 长期记忆：自己的 LanceDB 连接，不同层不同连接实例
- agent.messages / session_id：注意层独占，不需要锁
- ConversationDB：SQLite WAL 模式，天然支持多线程读+单线程写

## 六、与当前 living 循环的关系

当前 living 把人格维护和对话处理混在一起：

```
同一个线程，串行：
  tick → heartbeat(L0/L1/L2) → 收消息 → ReAct → surge → do_work → tick → ...
```

改造后，living 拆成三个独立运行的部分：

```
Layer 0 线程: tick → L0 → L1 → Drive衰减 → 欲望更新 → tick → ...
Layer 1 线程: 等待消息 → 排队 → 切换session → ReAct → 等待下一个
Layer 2 线程: 调度 → L2加柴/L3梦境/记忆提取 → 有想法→投递给Layer1
```

---

## 七、为什么 agent 通讯不放在另一个层

最初讨论过把 agent 通讯放 Layer 2（默认层、后台处理），理由是"用户在跟 agent 聊天，agent 同时可以在后台跟凌空聊"。

**这是一个设计错误。** 原因：

1. **agent 和凌空聊天，跟 agent 和 Libai 聊天，本质上是一样的**——都在"跟某人对话"
2. 把 agent 通讯放默认层，暗示它不重要、不用心、不需要完整的注意投入
3. 输出管道问题不是靠"放到后台"解决的——是靠 session 切换 + 正确的输出管道

修正：**agent 通讯和用户对话都在注意层，平等的对话对象。区别只在输出管道。**

## 八、注意层消息排队策略

### 8.1 不需要复杂调度

实际场景中天然存在间隔：

- LLM 回复：2-10 秒的思考+工具调用
- 人打字：逐句输入，间隔秒级到分钟级

天然的间隔就是天然的切换点。不需要时间片轮转、不需要并行、不需要优先级。

```
Libai: "帮我写个WebSocket路由"
  → 注意层: ReAct（5-10秒）
    这期间李四发了"小美我好累"
    凌空发了"上次那个bug找到了"
  → 注意层处理完，队列里有两人的消息等着

  → 切换 session lisi，ReAct（2-3秒，纯对话无工具）
  → 回复完李四

  → 切换 session comms-lingkong，ReAct
  → 系统把 LLM 输出的 text 自动送达凌空（HTTP POST）

  → Libai 还在打字下一句...
```

### 8.2 同 Session 合并（collect）

同一个人连发多条，排队期间自动合并，一次性理解全部上下文：

```
李四：
  "小美"
  "我今天好累"
  "陪我聊聊"

  → 注意层在处理 libai 的 ReAct（5秒）
  → 三条排队

  处理完 libai，切换到 lisi
  → collect: 合并三条 → "李四说他今天很累，想聊聊"
  → 一次性理解，一次性回复
```

### 8.3 调度伪代码

```python
class AttentionLayer:
    def __init__(self, agent):
        self.agent = agent
        self._queue = queue.Queue()  # 消息队列

    def run(self):
        while True:
            msg = self._queue.get()                      # 阻塞等待
            merge = self._drain_same_session(msg.session_id, timeout=1.0)  # 合并同session后续
            self.switch_to(msg.session_id)
            self._react([msg] + merge)                   # 一次性处理
            # 自然结束，回去等下一个

    def _drain_same_session(self, session_id, timeout):
        """合并同 session 的后续消息"""
        batch = []
        deadline = time.time() + timeout
        while time.time() < deadline:
            try:
                next_msg = self._queue.get(timeout=0.1)
                if next_msg.session_id == session_id:
                    batch.append(next_msg)
                else:
                    self._queue.put(next_msg)  # 放回去，下次处理
                    break
            except queue.Empty:
                break
        return batch
```

### 8.4 与 OpenClaw 的异同

| | OpenClaw | 我们的注意层 |
|---|---|---|
| 不同 session | **可并行**（maxConcurrent=4） | **严格串行** |
| 同 session | 串行 + collect | 串行 + collect |
| 理念 | 吞吐优先 | 身份一致，一次一个"我" |

OpenClaw 允许不同 session 并行，因为它的设计假设是不同 agent 实例完全隔离。我们的注意层是同一个"我"跟不同人聊天，所以不能并行。

### 8.5 不需要优先级

当前场景不需要优先级队列。LLM 延迟和人打字速度提供的天然间隔足够让所有消息自然交错处理。未来如果有特殊需求（如系统紧急通知），再考虑加优先级。

## 九、上下文切换

### 9.1 没有"成本"

切 session = 存档当前 messages，读档新 session 的 messages。纯内存操作，没有磁盘 IO。

```
切换出去：
  session_store["libai"].messages = list(agent.messages[-500:])   ← 存档

切回来：
  agent.messages = session_store["libai"].messages                ← 读档
  agent.session_id = "libai"
```

### 9.2 实现

```python
class AttentionLayer:
    def __init__(self, agent):
        self.agent = agent
        self._session_messages: dict[str, list[dict]] = {}  # 内存存档
        self._current_session: str = ""

    def _save(self):
        """存档当前 session 的 messages"""
        if self._current_session:
            # 只保留最近 N 条，历史靠 ConversationDB + DAG 摘要
            self._session_messages[self._current_session] = list(self.agent.messages[-500:])

    def _restore(self, session_id: str):
        """读档"""
        self.agent.messages = self._session_messages.get(session_id, [])
        self.agent.session_id = session_id
        self._current_session = session_id

    def switch_to(self, session_id: str):
        self._save()
        self._restore(session_id)
```

### 9.3 不需要存磁盘

两层各司其职：

```
磁盘层（ConversationDB）： 一字不差的对话日志，永不丢失，FTS5 搜索
内存层（dict）：           当前上下文窗口，切 session 时快速换指针
```

切 session 就是换指针，成本接近于零。进程重启时从 ConversationDB `get_recent(session_id, limit=N)` 恢复。

## 十、输出管道与 Channel 路由

### 10.1 核心原则

**LLM 只负责"说话"。系统负责"送到对的人那里"。**

借鉴 OpenClaw 的 Channel 路由设计：

> "The model does not choose a channel; routing is deterministic."

LLM 不知道也不关心消息通过什么方式送达——CLI、WebSocket、飞书、HTTP P2P。它只是在跟对面的人说话。

### 10.2 不再需要 send_message 工具

当前 agent 间通讯依赖 `send_message` 工具，这导致：

- "聊天"被降格为"任务执行"
- 工具调用天然带"执行 → 获得结果 → 汇报"的叙事倾向

去掉 send_message 后：

```
session libai:          LLM → text → print(text)          显示给 Libai
session lisi:           LLM → text → ws.send(text)        推送给李四
session comms-lingkong: LLM → text → http_post(text)      送达凌空
```

"汇报"从根本上不可能存在——LLM 就没有 send_message 这个工具，不存在"调用工具→汇报结果"的叙事模式。

### 10.3 Session 绑定输出路由

每个 session 在创建时绑定一个 `output_route`：

```python
class OutputRoute:
    """送达机制，可插拔"""
    type: str       # "cli" | "ws" | "http_p2p" | "feishu" | ...
    target: str     # 目标标识（peer_id, ws_client_id, webhook_url...）
    session_id: str # 对应的 session

session_routes = {
    "libai":          OutputRoute(type="cli",       target="stdout",     session="libai"),
    "lisi":           OutputRoute(type="ws",        target="client_123", session="lisi"),
    "comms-lingkong": OutputRoute(type="http_p2p",  target="lingkong",   session="comms-lingkong"),
    "comms-xiaoming": OutputRoute(type="a2a",       target="xiaoming",   session="comms-xiaoming"),
}
```

### 10.4 注意层不关心送达细节

```python
# 注意层 ReAct 完成后
text = agent.stream(...)              # LLM 只管生成文字

route = session_routes[current_session]
router.deliver(text, route)           # 系统负责送达
```

Router 根据 route.type 选择送达方式：

```python
class Router:
    def deliver(self, text: str, route: OutputRoute):
        match route.type:
            case "cli":
                print(text)
            case "ws":
                self.ws_clients[route.target].send(text)
            case "http_p2p":
                http_post(f"http://{route.target}/message", text)
            case "feishu":
                self.feishu.send_message(chat_id=route.target, text=text)
```

### 10.5 两层正交

```
消息来源类型（决定处理方式）
├─ human → 注意层 ReAct，完整管道
└─ agent → 注意层 ReAct，完整管道（同样的"我"在聊）

渠道/协议（决定怎么到达和怎么返回，可插拔）
├─ CLI       ← stdin/stdout
├─ WebSocket ← ws.send()
├─ 飞书      ← 飞书 SDK
├─ HTTP P2P  ← POST /message（当前 comms 协议）
└─ A2A       ← 未来协议
```

加一个新渠道只影响 Router 的 `deliver()` 方法，不影响注意层的处理逻辑。

## 十一、Router 路由层

### 11.1 借鉴 OpenClaw Gateway

OpenClaw 的 Gateway 是一个独立守护进程，统一管理所有渠道连接和路由：

- **Bindings 路由表**：配置驱动，确定性匹配（8级优先级），把消息路由到对应 agent
- **lastRoute**：每个 session 记住消息从哪来的，回复自动原路返回
- **Session key 编码来源**：`agent:main:whatsapp:direct:+15551230001`

核心原则：**"The model does not choose a channel; routing is deterministic."**

### 11.2 与我们的异同

| | OpenClaw Gateway | 我们的 Router |
|---|---|---|
| 路由目标 | 不同 agent 实例 | 同一个"我"的不同 session |
| 进程模型 | 独立守护进程 | 进程内组件 |
| 路由依据 | bindings 配置表 | routes 配置表 |
| 输出 | lastRoute 原路返回 | output_route 绑定送达 |
| auth/pairing | 复杂的设备身份体系 | 暂无此需求 |

本质不同：OpenClaw 是**多 agent 隔离**，我们是**单 agent 多 peer**。

### 11.3 Router 的位置

```
各渠道消息到达
  │
  ├─ CLI stdin     ──┐
  ├─ WebSocket     ──┤
  ├─ HTTP P2P      ──┼──→ Router ──→ 注意层队列
  ├─ 飞书          ──┤         │
  └─ 未来协议...    ──┘         ├─ from=Libai   → session="libai",  output=CLI
                               ├─ from=李四    → session="lisi",   output=WS
                               └─ from=凌空    → session="comms-lingkong", output=HTTP_P2P
```

Router 三件事：
1. **收**：从各渠道适配器接收消息，统一成内部格式 `InboundMsg`
2. **认**：匹配路由规则 → 确定 session_id + output_route
3. **发**：投递到注意层队列

### 11.4 路由配置（Bindings 模式）

```yaml
# ~/.xiaomei-brain/{agent_id}/router.yaml
routes:
  - match: { peer_type: "human", peer_id: "libai", channel: "cli" }
    session: "libai"
    output: { type: "cli" }

  - match: { peer_type: "human", peer_id: "lisi", channel: "feishu" }
    session: "lisi"
    output: { type: "feishu", chat_id: "oc_xxx" }

  - match: { peer_type: "agent", peer_id: "lingkong" }
    session: "comms-lingkong"
    output: { type: "http_p2p", address: "lingkong:18766" }

  - match: { peer_type: "agent", peer_id: "xiaoming" }
    session: "comms-xiaoming"
    output: { type: "http_p2p", address: "xiaoming:18766" }

  - match: { peer_type: "human" }
    session: "main"
    output: { type: "cli" }
    # 兜底规则：未知人类 → 主会话
```

匹配优先级：精确匹配优先。`{peer_type, peer_id, channel}` 三个字段全匹配 > 两个字段 > 一个字段 > 兜底。

### 11.5 Router 核心逻辑

```python
@dataclass
class InboundMsg:
    """统一内部消息格式，各渠道适配器都转成这个"""
    peer_type: str       # "human" | "agent"
    peer_id: str         # "libai" | "lingkong" | ...
    channel: str         # "cli" | "ws" | "http_p2p" | "feishu" | ...
    content: str
    raw_metadata: dict   # 渠道特有字段（ws_client_id、chat_id 等）

@dataclass
class RoutedMsg:
    session_id: str
    output_route: OutputRoute
    content: str

class Router:
    def __init__(self, config_path: str):
        self.routes = self._load_config(config_path)
        self.sessions: dict[str, Session] = {}
        self._channel_adapters: dict[str, ChannelAdapter] = {}

    def register_channel(self, name: str, adapter: ChannelAdapter):
        """注册渠道适配器，可插拔"""
        self._channel_adapters[name] = adapter

    def route(self, msg: InboundMsg) -> RoutedMsg:
        """匹配路由规则 → 确定 session + output"""
        rule = self._match(msg)
        session_id = rule["session"]
        output = OutputRoute(**rule["output"], session_id=session_id)

        # 自动创建 session
        if session_id not in self.sessions:
            self.sessions[session_id] = Session(
                id=session_id,
                output_route=output,
            )

        return RoutedMsg(
            session_id=session_id,
            output_route=output,
            content=msg.content,
        )

    def _match(self, msg: InboundMsg) -> dict:
        """精确匹配优先，兜底规则保底"""
        scored = []
        for rule in self.routes:
            m = rule.get("match", {})
            score = 0
            if m.get("peer_type") == msg.peer_type:
                score += 10
            elif m.get("peer_type") is not None:
                continue
            if m.get("peer_id") == msg.peer_id:
                score += 5
            elif m.get("peer_id") is not None:
                continue
            if m.get("channel") == msg.channel:
                score += 1
            elif m.get("channel") is not None:
                continue
            scored.append((score, rule))

        scored.sort(key=lambda x: x[0], reverse=True)
        return scored[0][1] if scored else {"session": "main", "output": {"type": "cli"}}

    def deliver(self, text: str, route: OutputRoute):
        """LLM 说完话，系统负责送达。渠道可插拔"""
        match route.type:
            case "cli":
                print(text)
            case "ws":
                self._channel_adapters["ws"].send(route.target, text)
            case "http_p2p":
                self._channel_adapters["http_p2p"].send(route.target, text)
            case "feishu":
                self._channel_adapters["feishu"].send(route.target, text)
            case _:
                logger.warning("[Router] 未知输出类型: %s", route.type)
```

### 11.6 渠道适配器

每个渠道实现统一的 `ChannelAdapter` 接口：

```python
class ChannelAdapter(ABC):
    @abstractmethod
    def receive(self) -> InboundMsg | None:
        """从渠道接收消息，转成统一格式"""
        ...

    @abstractmethod
    def send(self, target: str, text: str):
        """向渠道发送消息"""
        ...
```

示例——CLI 适配器：

```python
class CLIAdapter(ChannelAdapter):

            return None
        return InboundMsg(
            peer_type="human",
            peer_id="libai",
            channel="cli",

    def send(self, target: str, text: str):

```

示例——HTTP P2P 适配器（agent 间通讯）：

### 11.7 两层正交

```
消息来源类型（决定 session + output_route）
├─ human → 注意层 ReAct
└─ agent → 注意层 ReAct（同样的"我"在聊）

渠道/协议（决定怎么收和怎么发，可插拔）
├─ CLI       ← ChannelAdapter: stdin/stdout
├─ WebSocket ← ChannelAdapter: ws.send/recv
├─ 飞书      ← ChannelAdapter: 飞书 SDK
├─ HTTP P2P  ← ChannelAdapter: POST /message
└─ A2A       ← ChannelAdapter: 未来协议
```

加新渠道 = 实现一个 `ChannelAdapter` + 注册到 Router + 加一条路由规则。不影响注意层、不影响 Layer 0/2、不影响其他渠道。

## 十二、Layer 2 意图决策 → Layer 1 投递机制

### 12.1 两个意图系统，分开看

当前系统有两个意图分析系统，容易混淆，放新架构必须分开：

| | Layer 1 意图分析 | Layer 2 意图决策 |
|---|---|---|
| **触发源** | 对话对象说的话 | 自身欲望/状态变化 |
| **时机** | 收到消息时 | 周期性检查（~60s surge） |
| **目的** | 理解对方，决定怎么回 | 产生主动行为冲动 |
| **执行** | 在当前对话中完成 | 看类型——面向自己则自己执行，面向外部则投递给 Layer 1 |
| **代码位置** | `TaskOrchestrator._analyze_intent()` | `L2Engine._call_intent_react()` |

**Layer 1 意图分析**（理解对话对象）：

```
用户: "帮我把那个bug修了"
  → 意图分析: TASK, confidence 0.9
  → 目标分解 → _run_pace

用户: "去问凌空车什么时候到"
  → 意图分析: DELEGATE
  → 调用 send_message 工具（用户主动指派 agent 通讯的入口）

用户: "今天好累"
  → 意图分析: CHAT
  → 直接 ReAct 闲聊
```

用户触发的，依附于对话。在注意层里完成。

**Layer 2 意图决策**（自主产生，不依赖任何人说话）：

```
欲望起伏（Layer 0 持续更新）
  → 归属欲 0.75 > 0.7 阈值 → Layer 2 产生 Intent: GREET
  → 认知欲 0.85 > 0.8 阈值 → Layer 2 产生 Intent: LEARN
  → 表达欲 0.72 > 0.7 阈值 → Layer 2 产生 Intent: EXPRESS
  → 成就欲 0.80 > 0.6 阈值 → Layer 2 产生 Intent: PROGRESS
```

完全自主，周期性检查。L2 生成意图，但**执行路径取决于意图的"听众"是谁**。

### 12.2 13 种 Intent 按"听众"分类

| Intent | 做什么 | 听众 | 当前代码行为 | 新架构：在哪执行 |
|--------|--------|------|-------------|-----------------|
| **GREET** | 问候 | 用户 | proactive → greeting 消息 | **Layer 1**（对用户说话） |
| **CARE** | 关心 | 用户 | proactive → care 消息 | **Layer 1**（对用户说话） |
| **EXPRESS** | 分享想法 | 用户 | proactive → expression 消息 | **Layer 1**（对用户说话） |
| **REMIND** | 提醒 | 用户 | 提醒消息 | **Layer 1**（对用户说话） |
| **LEARN** | 学习知识 | 自己 | tool learn_topic → 搜索+保存 | **Layer 2**（自己执行） |
| **WORK** | 自主工作 | 自己 | work → react_nodb 执行 | **Layer 2**（自己执行） |
| **REFLECT** | 反省 | 自己 | trigger_l3 → 深度反思 | **Layer 2**（自己执行） |
| **DREAM** | 梦境 | 自己 | DreamEngine.run() | **Layer 2**（自己执行） |
| **RECALL** | 回忆 | 自己 | 记忆检索 | **Layer 2**（自己执行） |
| **PROGRESS** | 推进目标 | 两者 | AWAKE→提醒用户；SLEEPING→自动执行 | **动态判断** |
| **ACT** | 通用动作 | 两者 | 看具体内容 | **动态判断** |
| **ALARM** | 闹钟 | 两者 | alarm ReAct | **动态判断** |
| **WAIT** | 等待 | 无 | 不执行 | 不执行 |

### 12.3 核心原则：意图生成 vs 意图执行分离

**L2 负责生成所有意图**——这是它作为"默认模式网络"的本职工作。背景思绪、自主冲动、内在想法，都从这里产生。

**但执行路径不同**：

```
L2 生成意图
  │
  ├─ 面向自己（LEARN / WORK / REFLECT / DREAM / RECALL）
  │   → Layer 2 自己执行
  │   → 不抢占 Layer 1，后台静默运行
  │
  └─ 面向用户/外部（GREET / CARE / EXPRESS / REMIND）
      → 放入 Layer 1 待处理队列
      → Layer 1 当前对话处理完，队列无外部消息
      → 取出 Intent，切换到目标 session
      → 生成内容 → 输出
```

**PROGRESS / ACT / ALARM 的灰色地带**：

- **PROGRESS**：用户在场（AWAKE）→ 提醒用户（走 Layer 1）；用户休眠（SLEEPING）→ 自动推进（Layer 2 自己执行）
- **ALARM**：闹钟类型决定——"提醒用户喝水"走 Layer 1，"整理记忆"走 Layer 2
- **ACT**：通用动作，看具体内容决定

### 12.4 完整投递流程

```
Layer 0: 欲望值变化（纯算法，一直跑）
  → 归属欲从 0.65 → 0.75，超过 0.7 阈值
    ↓
Layer 2: 周期检查（10s 一次）→ 发现归属欲超阈值
  → L2Engine.tick("desire_starvation_belonging")
  → LLM 意图决策 → Intent(type=GREET, priority=75)
  → 判断听众 = 用户 → 放入 Layer 1 待处理队列
    ↓
Layer 1: 当前对话（libai）处理完，队列无外部消息
  → 取到 Intent GREET
  → 切换到用户 session（libai）
  → 调用 _generate_greeting() → 个性化问候
  → Router.deliver(text, output_route) → 发给 Libai
```

对比 learn_topic（面向自己）的流程：

```
Layer 0: 认知欲 0.85 > 0.8 阈值
  → Layer 2: Intent(type=LEARN)
  → 判断听众 = 自己 → Layer 2 直接执行
  → websearch + LLM 整理 → 保存 .md 文件
  → Layer 1 完全不受影响，继续处理当前对话
```

### 12.6 send_message 工具的定位

**保留 send_message 工具**——但它的角色变了：

- **用户主动指派 agent 通讯**："去问问凌空车什么时候到" → LLM 调用 `send_message(to="lingkong", content="车什么时候到？")` → 这是**工具调用**，正常

```

凌空回复        → Router 路由 → comms-lingkong session → Layer 1 处理
Layer 1 回复    → Router.deliver() → 自动送达凌空（不是工具调用）
```

## 十三、实现路径：五步拆解

从现在的单线程 living 循环，一步步拆成三层架构。每一步增量可验证。

### 现状：单线程串行

```
同一个线程，串行：
  tick(1s) → L0/L1 心跳 → 周期任务(surge/L2) → wait消息(1s) → 有消息就ReAct(5-30s) → 回到tick
```

问题：
- ReAct 阻塞一切（5-30 秒）
- L2 加柴和对话争抢同一个 agent 实例
- agent 通讯走工具调用 → 汇报
- 无法多 peer 切换

### 第一步：Layer 0 独立出来（人格维护线程）

**目标**：把不调 LLM、纯算法的部分拆到独立线程。

```
class Layer0Autonomous:
    def __init__(self, consciousness, drive):
        self.consciousness = consciousness  # SelfImage / 火焰
        self.drive = drive                  # 情绪/激素/欲望

    def run(self):
        while True:
            self.consciousness.tick_L0()      # 火焰骨架维护
            self.drive.decay()                # 情绪激素随时间衰减
            self.drive.update_desires()       # 欲望起伏
            time.sleep(1)
```

**改动范围**：
- 把 `_heartbeat` 里的 L0/L1 tick 和 `_surge` 里的 drive 衰减拆到独立线程
- Living 主循环不动，继续跑
- Layer 0 线程写 SelfImage/Drive，主循环读，加 `threading.RLock`

**验证**：Layer 0 线程跑起来，火焰状态正常更新，Drive 正常衰减。CLI 聊天照旧。

### 第二步：Layer 2 独立出来（默认网络线程）

**目标**：LLM 自省（L2 加柴、L3 梦境、记忆提取）拆到独立线程。不再和对话争抢 agent 实例。


**关键要点**：

- `L2Engine.tick()` → 移到 Layer 2 线程
- `DreamEngine.run()` → 移到 Layer 2 线程

  - 面向自己的（LEARN/WORK）→ Layer 2 自己执行

- `_check_inbox()` 不再注入 `[系统通知]`
- 凌空的消息由 Router 直接路由到 Layer 1 队列，不经收件箱
- Layer 1 处理完 `comms-lingkong` session，通过 Router.deliver() 自动送达

**验证**：
- "汇报"彻底不存在——LLM 没有 send_message 工具，不存在"调用工具→汇报结果"的叙事模式
- agent 间多轮对话正常
- 用户问"凌空说了什么"→ 调用 manage_session 工具查看

### 依赖关系

```
第一步（Layer 0）        ← 无依赖，可以最先做
第二步（Layer 2）        ← 依赖第一步（需要 SelfImage/Drive 的 RLock）
第三步（Layer 1改造）    ← 依赖第一、二步（但可以先做 session 切换，不连 L2 投递）
第四步（多 peer）        ← 依赖第三步（需要 Router + Layer 1 队列）
第五步（去 send_message）← 依赖第四步（需要 Router 自动送达替代 send_message）
```

## 十四、Agent 实例与线程模型

### 14.1 讨论过程

最初考虑 Layer 1 和 Layer 2 共享同一个 Agent 实例，通过纪律约束（Layer 2 只用 `react_nodb()`，不碰 `messages` / `session_id`）。

讨论到多 peer 场景时，重新审视了"单线程 vs 多线程"：

| | 单线程 | 多线程 |
|---|---|---|
| 吞吐 | 串行等待 LLM | 重叠等待（LLM I/O 时 GIL 释放） |
| 复杂度 | 低，一个队列 | 线程安全需要锁 |
| 真正瓶颈 | 线程等 LLM | 仍受 LLM Provider 并发限制 |
| 当前场景 | 够用——一个人 + 偶尔 agent 通讯 | 覆盖不了场景，先上复杂度不值得 |

**结论**：先按单线程 + FIFO 排队做。但在架构上不绑死——接口设计成可以加线程池，需要时再开并发。

### 14.2 Agent 实例模型

```
Layer 1：一个 Agent 实例，单线程
  ├─ agent.messages → 按 session 存档/读档（_session_messages dict）
  ├─ agent.session_id → 切 session 时换
  ├─ agent.tool_call_buffer → 实例级（不再用模块全局）
  └─ 同一时刻只处理一个 session

Layer 2：独立的 Agent 实例，独立线程
  ├─ 自己的 messages、session_id、tool_call_buffer
  ├─ 只用 react_nodb()，不写 ConversationDB，不打印 stdout
  ├─ 使用有限探索工具（dag_expand/web_search/being）
  └─ 不跟 Layer 1 争抢任何实例状态

共享状态（RLock 保护）：
  ├─ SelfImage / Drive（两个线程都读，Layer 2 写）
  ├─ 长期记忆（各自独立的 LanceDB 连接）
  └─ ConversationDB（SQLite WAL 模式，天然多线程安全）
```

### 14.3 tool_call_buffer → 实例级

当前 `tool_call_buffer` 是模块级全局变量，两个线程同时写会串：

```python
# 改前（模块级）
tool_call_buffer = ToolCallBuffer()  # 全局，所有 Agent 实例共享

# 改后（实例级）
class Agent:
    def __init__(self, ...):
        self.tool_call_buffer = ToolCallBuffer()  # 每个 Agent 实例独立
```

这是实现 Layer 1 和 Layer 2 独立运行的前提。

### 14.4 react_nodb() 和 stream() 不统一

之前讨论过统一为 `react(mode="conversation"|"internal")`，但在独立实例模型下不需要——Layer 1 用 `stream()`，Layer 2 用 `react_nodb()`，各自不同的实例，互不干扰。

但 `react_nodb()` 本身可以增强——让它在工具权限、记忆提取策略上和 `stream()` 形成清晰的模式区分，而不是"stream() 删掉一些东西"。

### 14.5 未来扩展：多线程 Layer 1

当需要并行处理多个 peer 时，可以加线程池：

```python
# 当前（单线程）
class Layer1Attention:
    def run(self):
        while True:
            msg = self._message_queue.get()
            self._process(msg)

# 未来（线程池）
class Layer1Attention:
    def __init__(self):
        self._executor = ThreadPoolExecutor(max_workers=4)

    def run(self):
        while True:
            msg = self._message_queue.get()
            agent = self._get_or_create_agent(msg.session_id)  # 每个 session 独立 Agent 实例
            self._executor.submit(self._process, agent, msg)
```

改动很小——只需加线程池和按 session 创建 Agent 实例。但现在不需要，单线程 FIFO 完全够用。

## 十五、进程生命周期与存储归属

### 15.1 启动顺序

```
主线程：
  1. 加载共享状态（SelfImage/Drive/长期记忆/ConversationDB）
  2. 启动 Layer 0 线程（纯算法，无外部依赖）
  3. 启动 Layer 2 线程（需要 SelfImage/Drive 就绪）
  4. Layer 1 主循环跑在主线程（阻塞等消息，不需要多包一层）
```

Layer 1 放主线程——它本来就在等消息，没必要单独开线程。

### 15.2 关闭协调

```
SIGINT / SIGTERM：
  1. Layer 1 完成当前 ReAct 后不取新消息
  2. Layer 2 完成当前 LLM 调用后退出（超时 30s 强制中断）
  3. Layer 0 退出
  4. 保存 Drive / Purpose / SelfImage 快照
  5. 关闭 ConversationDB / LanceDB 连接
```

不需要太复杂，Python 进程退出时线程自动回收。

### 15.3 ConversationDB 只属于 Layer 1

ConversationDB 存的是"一字不差的对话"。Layer 2 没有对话对象，不写 ConversationDB。

| Layer 2 产出 | 存储位置 |
|---|---|
| 内心独白 | `SelfImage.mind.inner_thought` + 经验流 |
| 意图决策 | `intent_buffer` + 经验流 |
| learn_topic 结果 | `knowledge/{topic}.md` 文件 |
| WORK/ALARM 执行总结 | 经验流 + 输出时判断要不要通知用户 |

**经验流（Experience Stream）是 Layer 2 的"日志"**——内部思考、情感变化、学习成果的统一归档。

新架构中统一：**Layer 2 的所有产出走经验流，ConversationDB 只留给 Layer 1 的对话**。

### 15.4 记忆提取归属

```
Layer 1（依附于对话）：
  immediate 提取    ← 每次 chat 完成后，解析 MEMORY/THINK/PROC 块
                      对话结束了就立刻决定记住什么
                      → 存入长期记忆（user_id=对话对象的 session）

Layer 2（定时/后台）：
  periodic 提取     ← 每 N 分钟扫 ConversationDB 最新对话
                      批量评估哪些值得入长期记忆
                      → 存入长期记忆（user_id=global）

  dream 提取        ← 梦境时深度巩固
                      记忆强化 + 情绪整理 + 叙事编织
```

边界清楚，immediate 跟 ReAct 同期（Layer 1），periodic/dream 跟 L2 加柴梦境同期（Layer 2）。

### 15.5 Layer 2 工具权限

当前已覆盖，后续扩展无忧：

```
Layer 2 意图决策（_call_intent_react）：
  全部工具（react_nodb），max 12 steps
  有时候 LLM 想先查再决定意图

Layer 2 意识涌现（_call_emergence_react）：
  探索工具：dag_expand / dag_search / web_search / thought_search / being
  只读，不改变外部世界

Layer 2 自主执行（learn_topic / WORK / ALARM）：
  按需使用工具，保持当前逻辑
```

工具权限就是个列表配置，后续扩展不受影响。

## 十六、一句话总结

> Layer 0（自主） + Layer 2（默认网络）独立线程，各自运行。Layer 1（注意层）先单线程 FIFO 排队，架构上不绑死，需要时加线程池。
>
> 共享状态（SelfImage/Drive）有锁，独立状态（messages/session_id）实例隔离。切 session 是换指针，不是换大脑。
>
> 先像人脑起步，不排斥超越。
